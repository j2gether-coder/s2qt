# S2QT 환경설정 및 History 설계 초안

## 1. 개요

S2QT는 QT 준비 → Step1 → Step2 → Step3의 작업 흐름을 가지는 데스크톱 앱이다.  
현재 1차적으로 메시지 체계(window.alert 개선)가 정리되었고,  
이제 운영 기능으로서 **환경설정(Settings)** 및 **내역(History)** 기능을 설계한다.

본 문서는 다음 범위를 정리한다.

- 환경설정 저장 범위
- History 저장 범위
- SQLite 기반 저장 구조
- 화면 구성 방향
- 구현 시 우선순위

---

## 2. 기본 방향

### 2.1 환경설정(Settings)
환경설정은 사용자가 반복적으로 사용하는 공통 값을 저장하는 영역이다.

예:
- AI(LLM) API KEY
- SMTP 정보
- 교회명 / 로고 / 홈페이지 URL
- 라이선스 안내 / 사용법 안내

### 2.2 내역(History)
History는 완성본 보관소가 아니라,  
**이전 작업의 기본정보와 Step1 결과 JSON을 다시 불러와 Step2부터 이어서 작업하기 위한 재사용 저장소**로 정의한다.

즉, 저장 범위는 최소화한다.

저장 대상:
- 기본정보
- Step1 결과 JSON

저장 제외:
- 전사문(raw_text)
- source URL / source file path
- Step2 결과
- Step3 결과
- 산출물 파일 경로

---

## 3. History 설계

## 3.1 목적

History 기능의 목적은 다음과 같다.

1. 이전 설교 작업의 기본정보를 재사용
2. 이전 Step1 JSON 결과를 재사용
3. 불러온 후 Step2 → Step3 흐름으로 다시 작업

즉, History는 “재출발 지점”이다.

---

## 3.2 작업 흐름

### 저장 흐름
1. QT 준비에서 기본정보 입력
2. Step1에서 JSON 결과 생성/저장
3. Step1 저장 성공 시 DB에 History 저장

### 불러오기 흐름
1. History 목록에서 항목 선택
2. audience 선택
3. 기본정보 + Step1 JSON 불러오기
4. Step2로 이동
5. Step3로 진행

---

## 3.3 저장 범위

### 저장 대상
- title
- bible_text
- hymn
- preacher
- church_name
- sermon_date
- audience
- step1_result_json

### 저장 제외
- raw_text
- source_type
- source_url
- source_file_path
- step2_result_json
- step3_result
- output file path

---

## 3.4 SQLite 테이블 설계

### 3.4.1 history_master

설교 작업의 기본정보 저장

```sql
CREATE TABLE history_master (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    bible_text TEXT NOT NULL,
    hymn TEXT,
    preacher TEXT,
    church_name TEXT,
    sermon_date TEXT,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);
````

### 3.4.2 history_step1

설교 작업의 연령대별 Step1 결과 JSON 저장

```sql
CREATE TABLE history_step1 (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    history_id INTEGER NOT NULL,
    audience TEXT NOT NULL,
    step1_result_json TEXT NOT NULL,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    FOREIGN KEY (history_id) REFERENCES history_master(id)
);
```

### 3.4.3 인덱스

```sql
CREATE INDEX idx_history_master_created_at
ON history_master(created_at DESC);

CREATE INDEX idx_history_master_title
ON history_master(title);

CREATE INDEX idx_history_step1_history_audience
ON history_step1(history_id, audience);
```

## 3.5 화면 구성 방향

History는 Step형이 아니라 **목록형 화면**으로 구성한다.

### 좌측 목록

* 제목
* 본문 성구
* 설교일
* 저장일

### 우측 상세

* 찬송
* 설교자
* 교회명
* 사용 가능한 audience 목록

### 액션

* 장년 QT 이어서 작업
* 청년 QT 이어서 작업
* 중고등부 QT 이어서 작업
* 어린이 QT 이어서 작업

선택한 audience의 Step1 JSON을 불러와 Step2부터 시작한다.

## 4. 환경설정(Settings) 설계

## 4.1 목적

환경설정은 앱이 반복적으로 참조하는 공통 운영값을 저장한다.

예:

* API KEY
* SMTP 정보
* 교회 정보
* 안내 문구

환경설정은 작업 Step 흐름이 아니라 **운영 관리 화면**이므로
Step형보다 **탭형 화면**이 적합하다.

---

## 4.2 화면 구성 방향

환경설정은 다음 4개 탭으로 구성한다.

1. AI 설정
2. 이메일 설정
3. 교회/브랜드 설정
4. 안내

---

## 4.3 저장 범위

### 4.3.1 AI 설정

* LLM 공급자
* API KEY
* 모델명
* 자동/수동 모드
* 사용 여부

### 4.3.2 이메일/SMTP 설정

* 사용 여부
* 발신 이메일
* SMTP 서버
* 포트
* 사용자명
* 비밀번호
* 보안 방식

### 4.3.3 교회/브랜드 설정

* 교회명
* 로고 경로
* 교회 홈페이지 URL
* 기본 footer 문구

### 4.3.4 안내

* 라이선스 유형
* 라이선스 상태
* 라이선스 만료일
* 앱 버전
* 사용 안내 문구

---

## 4.4 SQLite 저장 방식

환경설정은 초기에는 개별 테이블보다
**key-value 방식의 단일 테이블**로 운영하는 것이 적절하다.

### app_settings

```sql
CREATE TABLE app_settings (
    setting_key TEXT PRIMARY KEY,
    setting_value TEXT,
    value_type TEXT NOT NULL DEFAULT 'text',
    is_secret INTEGER NOT NULL DEFAULT 0,
    updated_at TEXT NOT NULL
);
```

---

## 4.5 setting_key 설계

### AI 설정

* llm.provider
* llm.api_key
* llm.model
* llm.mode
* llm.enabled

### SMTP 설정

* smtp.enabled
* smtp.from_email
* smtp.host
* smtp.port
* smtp.username
* smtp.password
* smtp.security

### 교회/브랜드 설정

* church.name
* church.logo_path
* church.homepage_url
* church.default_footer_text

### 라이선스/안내

* license.type
* license.status
* license.expire_date
* app.version
* app.guide_text

---

## 4.6 암호화 저장 대상

민감정보는 평문 저장하지 않는다.

### 암호화 대상

* llm.api_key
* smtp.password

### 평문 저장 가능

* provider
* model
* from_email
* smtp.host
* smtp.port
* church.name
* logo_path
* homepage_url
* guide_text
* license 상태 정보

암복호화는 DB가 아니라 Go 서비스 레이어에서 담당한다.

---

## 4.7 value_type 예시

* text
* password
* number
* boolean
* url
* multiline

예:

* llm.api_key → password
* smtp.port → number
* smtp.enabled → boolean
* church.homepage_url → url
* app.guide_text → multiline

## 7. 최종 정리

### History

* 저장 범위: 기본정보 + Step1 JSON
* 저장 제외: 전사문, Step2/3 결과, 산출물 경로
* 목적: Step2부터 다시 작업하기 위한 재사용

### 환경설정

* 저장 방식: SQLite key-value 단일 테이블
* 민감정보: API KEY / SMTP 비밀번호 암호화 저장
* 화면 구조: Step형이 아니라 탭형

### 공통

* SQLite 사용
* JSON은 TEXT 저장
* 로고는 파일 경로만 저장
* 암복호화는 Go 서비스에서 담당
