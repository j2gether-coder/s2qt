좋습니다. 그 부분은 반드시 반영해야 합니다.

지금 초안은 설명 흐름상 **동영상 중심으로 읽힐 수 있게** 되어 있습니다.
하지만 실제 입력 원천은 이전에 정리한 대로 다음 3가지여야 합니다.

* **동영상**
* **오디오**
* **텍스트**

즉 공통 원문 준비 화면은 “영상 전사 화면”이 아니라,
**원문 입력 채널을 선택하고 공통 원문을 준비하는 화면**으로 다시 써야 맞습니다.

아래는 그 점을 반영해 다시 정리한 **수정본 화면 정의서 초안**입니다.

# QT 작성기 화면 정의서 초안

## 1. 문서 목적

본 문서는 QT 작성기의 화면 구성을 정의한다.
구조는 다음 원칙을 따른다.

* 설교 원문 준비는 1회 수행
* QT 생성/검토/산출물 생성은 audience별로 반복 수행
* 내부 기준 데이터는 JSON
* HTML은 미리보기
* PDF/DOCX/PPTX는 산출물
* 입력 원천은 **동영상, 오디오, 텍스트**를 모두 지원한다

---

## 2. 전체 화면 구조

전체 화면은 2개 영역으로 구성한다.

### 2.1 공통 원문 준비 영역

한 설교에 대해 한 번만 수행하는 영역

포함 내용:

* 입력 원천 선택
* 원문 입력 또는 전사 수행
* 공통 메타 입력
* 공통 원문 저장

### 2.2 audience별 작업 영역

장년/청년/중고등부/어린이별로 반복 수행하는 영역

포함 내용:

* Step1 생성 조건 확인
* Step2 AI QT 생성
* Step3 내용 검토/편집
* Step4 산출물 생성

### 4.2 화면 목적

설교의 공통 원본 데이터를 준비한다.
이 데이터는 이후 모든 audience 작업의 기준 원문으로 사용된다.

공통 원문 준비 화면은 아래 3가지 입력 방식을 지원한다.

* 동영상
* 오디오
* 텍스트

즉 공통 원문 준비의 핵심은 “전사” 자체가 아니라,

* 공통 메타 입력
* 원문 확인 및 정리
* 공통 원문 저장

### 4.5 주요 데이터

* source_id
* source_type
* source_url 또는 source_file
* title
* bible_text
* hymn
* preacher
* church_name
* sermon_date
* transcript.raw_text
* transcript.cleaned_text

## 9. 공통 원문 저장 결과

공통 원문 준비 화면의 저장 결과는 다음과 같은 공통 원문 모델이다.

```json id="source-model-v1"
{
  "source_id": "",
  "source_type": "",
  "source_ref": {
    "url": "",
    "file_path": ""
  },
  "metadata": {
    "title": "",
    "bible_text": "",
    "hymn": "",
    "preacher": "",
    "church_name": "",
    "sermon_date": ""
  },
  "transcript": {
    "raw_text": "",
    "cleaned_text": ""
  }
}
```

### 필드 설명

* `source_type`: `video`, `audio`, `text`
* `source_ref.url`: 원본 URL
* `source_ref.file_path`: 로컬 파일 경로
* `metadata`: 공통 메타
* `transcript`: 원문 텍스트와 정리본

## 11. 공통 원문 준비 상태값 정의

### 공통 상태

* NOT_READY
* SOURCE_SELECTED
* INPUT_LOADED
* TRANSCRIBED
* CLEANED
* SAVED

### 설명

* `NOT_READY`: 아직 입력 원천 미선택
* `SOURCE_SELECTED`: 입력 원천 선택 완료
* `INPUT_LOADED`: URL/파일/텍스트 확보 완료
* `TRANSCRIBED`: 전사 완료
* `CLEANED`: 정리본 생성 완료
* `SAVED`: 공통 원문 저장 완료

텍스트 입력의 경우에는 전사 단계를 건너뛰고 바로 `INPUT_LOADED → CLEANED → SAVED`로 갈 수 있다.

## 22. 결론

화면 구조는 다음 원칙으로 확정한다.

* 공통 원문 준비는 상단 고정 영역으로 분리
* 입력 원천은 동영상, 오디오, 텍스트를 모두 지원
* audience별 작업은 탭 구조로 분리
* 각 audience는 Step1~4를 독립 수행
* Step3는 JSON 기반 구조화 편집으로 전환
* Step4는 JSON 기준 산출물 생성으로 통일

이 정의서를 기준으로 이후에는

1. 공통 원문 준비 화면 상세 설계
2. audience별 작업 화면 상세 설계
3. 상태/버튼/저장 로직 정의
   순서로 구체화한다.
