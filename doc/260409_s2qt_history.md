# S2QT Step3 작업 히스토리 및 단계형 UI 제어 구상

## 1. 문서 목적
이 문서는 S2QT 프로젝트의 Step3(산출물 생성 단계) 정리 과정과 파일 열기/저장 기능 구현 히스토리를 기록하고,
향후 필요한 단계형 UI 제어 방향을 정리하기 위한 문서이다.

---

## 2. 프로젝트 기준 정리

- 프로젝트: S2QT
- 구조: Wails + Vite/JS + Go
- 기준 데이터: `temp.json`
- HTML은 기준 데이터가 아니라 산출물 중 하나
- DOCX / PPTX도 최종적으로는 JSON 기준 생성이 원칙

---

## 3. Step3 정리 배경

이전 검토에서 다음 사항이 확인되었다.

1. Python 단독 테스트로 DOCX / PPTX 생성 가능성은 확인함.
2. DOCX는 기본 생성 및 스타일 적용 방향까지 검토 완료.
3. PPTX는 템플릿 기반 치환 및 복제 구조까지 테스트 완료.
4. 그러나 연령대별 텍스트 길이 차이, 템플릿 보정, 분량 제어 문제가 남아 있어 즉시 운영 품질로 연결하기에는 이르다고 판단함.
5. 따라서 현재 Step3에서는 HTML / PDF / PNG 중심으로 먼저 안정화하기로 결정함.
6. DOCX / PPTX는 차후 별도 작업으로 재개하기로 함.

---

## 4. Step3 UI 정리 결정 사항

### 4.1 산출물 노출 정책
현재 Step3에서는 아래 산출물만 우선 제공한다.

- HTML
- PDF
- PNG

보류 대상:

- DOCX
- PPTX

### 4.2 DOCX / PPTX 처리 방식
초기에는 비활성화, 안내문구, 숨김 등의 방식도 검토했으나,
최종적으로는 **주석 처리하여 화면에서는 보이지 않게 하고, 코드에는 복구 흔적을 남기는 방식**으로 정리하였다.

이 방식의 목적은 다음과 같다.

- 사용자 화면에는 현재 지원 기능만 명확히 보이게 함
- 미완성 기능이 운영 화면에 노출되는 것을 방지함
- 향후 DOCX / PPTX 재개 시 복구 포인트를 빠르게 찾을 수 있게 함

## 5. Step3 파일별 반영 내용

## 5.1 `qtStep3.js`
역할:
- Step3 UI 렌더 담당

반영 내용:
- HTML / PDF / PNG 체크박스만 노출
- DOCX / PPTX 체크박스는 주석 처리
- HTML / PDF / PNG 결과 카드만 노출
- DOCX / PPTX 결과 카드는 주석 처리
- 저장 버튼 비활성 문제 해결을 위해 각 산출물 카드의 저장 버튼 영역에 wrapper id 추가

예시 wrapper id:
- `htmlSaveBtnWrap`
- `pdfSaveBtnWrap`
- `pngSaveBtnWrap`

의미:
- Step3 실행 후 링크뿐 아니라 저장 버튼도 동적으로 갱신할 수 있게 됨

## 5.2 `bindQTStep3.js`
역할:
- Step3 이벤트 연결 담당

반영 내용:
- `RunQTStep3` 호출 유지
- 기존 `SaveQTOutputAs` 대신 `OpenGeneratedFile`, `SaveGeneratedFile` 사용 구조로 정리
- 결과 적용 시 파일 링크뿐 아니라 저장 버튼도 다시 렌더하도록 수정
- 파일 링크 클릭 시 기본 앱으로 열기 연결
- 파일 저장 버튼 클릭 시 SaveDialog 기반 저장 연결
- DOCX / PPTX 관련 payload / result 반영 코드는 주석 처리하여 흔적 유지

## 5.3 `output_file_service.go`
역할:
- 생성된 산출물 파일 열기/저장 서비스 담당

반영 내용:
- `OpenGeneratedFile(filePath)`
  - 링크 클릭 시 해당 확장자의 기본 애플리케이션으로 파일 열기
- `SaveGeneratedFile(filePath, audienceID, formatKey)`
  - 저장 대화상자 표시
  - 기본 파일명 제안
  - 사용자 지정 경로로 복사 저장

기본 파일명 규칙:
- `QT_연령_YYMMDD.ext`

예:
- `QT_장년_260409.pdf`
- `QT_청년_260409.html`

참고:
- temp 경로의 원본 파일은 유지
- 사용자 저장은 복사 저장 방식으로 처리

## 5.4 `app.go`
역할:
- Wails 바인딩 진입점

반영 내용:
- `OpenGeneratedFile`
- `SaveGeneratedFile`

정리 사항:
- Step3에서는 위 두 메서드를 사용하도록 정리
- 과거 방식인 `OpenOutputFile`, `SaveQTOutputAs`는 현재 Step3 기준으로 미사용 상태이며, 필요 시 차후 정리 또는 제거 검토

## 6. Step3 저장 기능 테스트 결과

테스트 결과:
- 전체 Step 절차 수행 후 PDF 파일 저장 테스트 진행
- 사용자 지정 폴더 선택 가능
- 사용자 수정 파일명 저장 가능
- 기본 저장 동작이 의도대로 구현된 것을 확인함

현재 상태 평가:
- Step3의 HTML / PDF / PNG 생성 및 결과 확인 흐름은 실사용 가능한 수준으로 정리 중
- 특히 PDF 저장은 핵심 요구사항 기준으로 1차 구현 완료로 볼 수 있음

## 7. 현재 남은 과제

### 7.1 DOCX / PPTX 재개발
현재는 의도적으로 제외한 상태이며, 차후 아래 이슈를 해결한 뒤 재개한다.

- 연령대별 텍스트 길이 차이
- 템플릿 overflow 제어
- 페이지/슬라이드 분량 자동 보정
- 템플릿 품질 고도화
- JSON 기준 생성물 품질 검증

### 7.2 코드 정리
향후 아래 정리가 가능하다.

- Step3 미사용 legacy API 정리
- DOCX / PPTX 복구 포인트 주석 정리
- Step3 파일 액션 이벤트의 중복 바인딩 여부 추가 점검

## 8. 다음 과제: 절차형 UI 제어 필요성

현재 화면 흐름은 다음과 같은 절차를 의도한다.

### 장년 QT 예시
1. Step1. LLM 이용
2. Step2. 검토 및 편집
3. Step3. QT 문서 생성

그러나 현재는 사용자 조작이나 화면 상태에 따라,
**장년 QT에서 Step1/Step2를 거치지 않고 Step3로 이동하는 현상**이 발생할 수 있다.

이는 다음 문제를 만든다.

- 필수 데이터가 준비되지 않은 상태에서 Step3 진입 가능
- 사용자 입장에서 절차 이해가 흐려짐
- temp.json, temp.html, 산출물 파일 상태와 화면 상태가 불일치할 수 있음
- 미완료 단계에서 후속 단계로 이동하면서 오류 가능성이 증가함

따라서 Step 기반 진행 제어가 필요하다.

---

## 9. 절차형 UI 제어 구상안

## 9.1 기본 원칙
각 연령대별 QT 화면은 **선행 단계 완료 여부**에 따라 다음 단계 진입 가능 여부를 제어한다.

예시:
- Step1 완료 전 → Step2 진입 제한
- Step2 완료 전 → Step3 진입 제한

## 9.2 추천 제어 방식

### 방식 A. 하드 가드
사용자가 Step2 또는 Step3를 선택했을 때,
선행 단계가 완료되지 않았다면 해당 단계로 이동시키지 않고 알림을 표시한다.

예:
- Step3 클릭 시 Step2 미완료면
  - 이동 차단
  - 안내: `먼저 Step2. 검토 및 편집을 완료해 주세요.`

장점:
- 가장 명확하고 안전함
- 현재 구조에 적용하기 쉬움

### 방식 B. 단계 활성/비활성 UI
사이드 메뉴 또는 단계 탭에서,
선행 단계가 완료되지 않은 후속 단계는 클릭 불가 또는 흐리게 표시한다.

예:
- Step1 완료 전에는 Step2, Step3 회색 처리
- Step2 완료 후 Step3 활성화

장점:
- 사용자가 현재 가능한 단계와 불가능한 단계를 직관적으로 이해 가능

### 방식 C. 진입 시 자동 리다이렉트
사용자가 Step3를 요청하더라도,
실제 완료 상태를 확인하여 미완료 단계가 있으면 자동으로 그 단계로 보낸다.

예:
- Step3 클릭
- Step2 미완료 감지
- 자동으로 Step2 화면으로 이동 + 안내문구 표시

장점:
- UX가 부드러움
단점:
- 왜 Step3가 안 열렸는지 사용자가 순간적으로 헷갈릴 수 있음

## 10. 추천안

현재 S2QT 구조에서는 다음 조합이 가장 적절해 보인다.

### 추천 조합
- **1차:** 하드 가드
- **2차:** 단계 활성/비활성 표시 추가

즉,
1. 진입 자체를 제어하고
2. 화면에서도 갈 수 있는 단계만 보이거나 활성화되게 하여
3. 사용자가 절차를 자연스럽게 따라가게 만드는 방식이다.

## 11. 구현 아이디어

## 11.1 상태값 관리
연령대별로 단계 완료 상태를 명시적으로 관리한다.

예시 개념:

- `step1Done`
- `step2Done`
- `step3Done`

또는 현재 단계와 완료 단계를 함께 관리:

- `currentStep`
- `maxAvailableStep`

예:
- Step1 완료 후 `maxAvailableStep = 2`
- Step2 완료 후 `maxAvailableStep = 3`

## 11.2 이동 가드 함수
단계 이동 전 공통 검증 함수 예시:

- `canEnterStep(audienceId, targetStep)`
- `guardAudienceStepMove(audienceId, targetStep)`

판단 예:
- targetStep = step2 이고 step1 미완료 → false
- targetStep = step3 이고 step2 미완료 → false

## 11.3 표시 문구 예시
- `먼저 Step1. LLM 이용을 완료해 주세요.`
- `먼저 Step2. 검토 및 편집을 완료해 주세요.`
- `이전 단계가 완료되어야 다음 단계로 이동할 수 있습니다.`

## 12. 우선 적용 권장 순서

### 1단계
Step 진입 가드만 먼저 구현

목표:
- Step3 무단 진입 방지
- 오류 감소
- 데이터 상태 일관성 확보

### 2단계
단계 버튼/메뉴 비활성화 UI 추가

목표:
- 사용자에게 현재 가능한 절차를 시각적으로 안내

### 3단계
단계 완료 상태를 상단/사이드에 명확히 표시

예:
- 완료
- 진행 중
- 미진입

## 13. 결론

이번 Step3 작업을 통해 다음을 정리하였다.

- HTML / PDF / PNG 중심 Step3 안정화
- DOCX / PPTX는 주석 처리로 보류
- 링크 클릭 시 기본 앱으로 열기 구현
- 파일 저장 시 사용자 지정 경로 + 사용자 수정 파일명 저장 구현
- PDF 파일 저장 테스트 성공 확인

다음 과제는 단순 기능 추가보다,
**사용자가 Step1 → Step2 → Step3 절차를 자연스럽고 안전하게 따르도록 만드는 단계형 UI 제어**이다.

즉, 이제부터의 핵심은 기능 자체보다도,
**절차를 통제하는 UX와 상태 관리 구조를 정리하는 것**에 있다.
