# QT 작성기 구조 전환 및 프로젝트 추진 초안

## 1. 문서 목적

본 문서는 현재 진행 중인 QT 작성기 프로젝트를  
기존 **HTML 중심 구조**에서 **JSON 중심 구조**로 전환하기 위한 기준 문서이다.

본 문서의 목적은 다음과 같다.

1. 지금까지 수행한 작업 중 **살릴 수 있는 부분**과 **재구성해야 할 부분**을 구분한다.
2. QT 작성기의 향후 목표 구조를 정의한다.
3. 연령대별 QT 생성 구조를 반영한 Step 흐름을 재설계한다.
4. 이후 개발 진행 시 본 문서를 기준으로 우선순위와 작업 방향을 일관되게 유지한다.

---

## 2. 현재까지의 핵심 교훈

이번 작업을 통해 다음과 같은 점이 명확해졌다.

### 2.1 HTML은 원본(Source of Truth)으로 적합하지 않다
기존에는 LLM이 HTML을 생성하고, 이를 다시 해석하여 PDF/DOCX/PPTX로 확장하는 방식을 검토하였다.

그러나 이 방식은 다음과 같은 문제를 보였다.

- `오늘의 메시지`, `깊은 묵상과 적용`, `오늘의 기도` 등 QT의 의미 구조가 HTML 태그 형태에 따라 흔들림
- HTML 파싱 규칙이 복잡해짐
- 문서 구조를 다시 복원해야 하므로 산출물 간 일관성이 깨질 가능성이 높음
- 표현용 요소와 구조용 요소가 섞여 있어 유지보수가 어려움

따라서 HTML은 원본이 아니라 **미리보기 및 렌더링 결과물**로 사용하는 것이 맞다.

### 2.2 QT는 자유 문서가 아니라 형식 문서이다
QT는 대체로 아래와 같은 고정된 구조를 가진다.

- 제목
- 본문 성구 / 찬송
- 말씀의 창: 본문 요약
- 오늘의 메시지
- 깊은 묵상과 적용
- 오늘의 기도

즉 QT 작성기는 일반 리치 텍스트 문서 편집기가 아니라,  
**구조가 정해진 문서를 생성·편집·산출하는 전용 생성기**로 보는 것이 맞다.

### 2.3 연령별 차이는 구조가 아니라 표현의 차이이다
장년, 청년, 중고등부, 어린이의 차이는 다음과 같은 영역에서 발생한다.

- 문장 길이
- 어휘 수준
- 설명 깊이
- 적용 방식
- 기도문의 길이와 어조
- 사례와 표현의 친숙도

반면 QT 문서의 큰 구조 자체는 동일하게 유지할 수 있다.

즉 연령별 처리의 핵심은 다음과 같다.

- 구조는 공통 JSON 스키마로 통일
- 생성 프롬프트와 표현 규칙만 연령별로 분기

### 2.4 전사는 공통 1회, 생성은 연령별 반복이 맞다
한 설교를 대상으로 장년용/청년용/중고등부용/어린이용 QT를 만들 경우,  
전사 자체를 연령별로 다시 할 필요는 없다.

따라서 전체 흐름은 다음이 되어야 한다.

- 설교 전사 및 공통 메타 확보: 1회
- QT 생성/검토/산출물 생성: 연령대별 반복

## 3. 프로젝트 목표 구조

## 3.1 기본 원칙

향후 QT 작성기는 다음 원칙을 따른다.

1. **설교 전사문과 공통 메타데이터는 1회만 생성/입력한다.**
2. **QT 생성 결과는 공통 JSON 스키마로 받는다.**
3. **연령별 차이는 프롬프트/생성 규칙에서만 분기한다.**
4. **HTML/PDF/DOCX/PPTX는 모두 JSON을 기준으로 생성한다.**
5. **HTML은 원본이 아니라 미리보기 및 렌더링 결과물이다.**

---

## 3.2 목표 아키텍처

```text
[공통 원문 준비]
  영상/오디오/전사/공통 메타 입력
          ↓
  sermon_source.json 생성
          ↓
[연령대별 반복 작업]
  audience 선택
          ↓
  request JSON 생성
          ↓
  base prompt + audience prompt + request JSON
          ↓
  LLM 호출
          ↓
  output JSON 생성
          ↓
  검토/편집
          ↓
  HTML / PDF / DOCX / PPTX 생성
````

## 4. 살릴 수 있는 작업과 재구성해야 할 작업

## 4.1 그대로 살릴 수 있는 작업

### 1) 전사 파이프라인

다음 기능은 그대로 유지한다.

* 영상 다운로드
* 오디오 추출
* Whisper 전사
* 전사문 확보

이 부분은 연령대와 무관한 **공통 원문 생성 단계**이므로 그대로 살린다.

### 2) 공통 메타 입력 항목

다음 입력 항목은 계속 유지한다.

* 제목
* 본문 성구
* 찬송
* 설교자
* 교회명
* 설교일
* 원본 URL

다만 역할은 “즉시 문서 생성용”이 아니라
**공통 원문 메타 저장용**으로 재정의한다.

### 3) QT 구조에 대한 정리 결과

지금까지 확정된 QT 구조는 그대로 살린다.

* 말씀의 창: 본문 요약
* 오늘의 메시지
* 깊은 묵상과 적용
* 오늘의 기도

이 구조는 공통 JSON 스키마의 기준이 된다.

### 4) HTML/CSS 자산

현재 확보된 QT용 HTML 구조와 CSS 클래스는 살린다.

예:

* `qt-title`
* `qt-subbox`
* `qt-section-title`
* `qt-message-title`
* `qt-reflection`
* `qt-prayer`

다만 앞으로는 이것을 “원본 문서 구조”가 아니라
**JSON을 HTML로 렌더링하는 템플릿**으로 사용한다.

### 5) PDF 생성 흐름

현재 HTML → PDF 생성 방식은 일단 유지 가능하다.

다만 앞으로는 흐름을 다음과 같이 바꾼다.

* JSON → HTML 렌더링
* HTML → PDF 생성

즉 PDF 생성 방식은 살리되, 입력 기준을 JSON으로 바꾼다.

## 4.2 역할을 바꿔 살릴 작업

### 1) pdf_service.go

기존 역할:

* HTML 입력
* temp.md, temp.html, temp.pdf, temp.json 생성

향후 역할:

* JSON 입력
* HTML/PDF 렌더링 서비스
* 필요 시 preview HTML 생성
* 템플릿 적용

즉 HTML 파서 중심이 아니라,
**JSON 기반 렌더링 서비스**로 재정의한다.

### 2) Step3 편집 화면

기존 역할:

* HTML 직접 편집

향후 역할:

* JSON 기반 구조화 편집
* HTML preview 병행

즉 화면 자체는 살리되, 편집 대상은 HTML이 아니라 JSON이 된다.

### 3) Step4 출력 UI

기존 역할:

* HTML/PDF 중심 산출물 생성 및 열기

향후 역할:

* JSON 기준 산출물 생성
* HTML/PDF/DOCX/PPTX 공통 관리

즉 UI 흐름은 유지하되, 생성 기준을 JSON으로 전환한다.

### 2) 프롬프트 구조

### 3) Step 구조

## 5. 공통 원문 모델 정의

공통 원문 모델은 한 번 전사한 설교의 기준 자산이다.

예시:

```json
{
  "source_id": "sermon_001",
  "metadata": {
    "title": "십자가의 죽으심",
    "bible_text": "갈 2:20",
    "hymn": "",
    "preacher": "",
    "church_name": "",
    "sermon_date": "",
    "source_url": ""
  },
  "transcript": "설교 전사문 전체"
}
```

### 역할

* 전사문 저장
* 공통 메타 저장
* audience별 QT 생성의 원본 자산

---

## 6. QT Output JSON 스키마 정의

QT 산출물의 원본은 항상 아래 구조를 따른다.

```json
{
  "version": "1.0",
  "doc_type": "qt",
  "audience": "adult",
  "template_id": "qt_classic",
  "metadata": {
    "title": "[QT] 십자가의 죽으심",
    "bible_text": "갈 2:20",
    "hymn": "",
    "preacher": "",
    "church_name": "",
    "sermon_date": "",
    "source_url": ""
  },
  "sections": [
    {
      "type": "summary",
      "title": "🌿 말씀의 창: 본문 요약",
      "blocks": [
        {
          "type": "paragraph",
          "text": ""
        }
      ]
    },
    {
      "type": "message",
      "title": "✨ 오늘의 메시지",
      "blocks": [
        { "type": "message_title", "text": "" },
        { "type": "paragraph", "text": "" },
        { "type": "message_title", "text": "" },
        { "type": "paragraph", "text": "" },
        { "type": "message_title", "text": "" },
        { "type": "paragraph", "text": "" }
      ]
    },
    {
      "type": "reflection",
      "title": "🔍 깊은 묵상과 적용",
      "blocks": [
        {
          "type": "list",
          "items": ["", "", ""]
        }
      ]
    },
    {
      "type": "prayer",
      "title": "🙏 오늘의 기도",
      "blocks": [
        {
          "type": "paragraph",
          "text": ""
        }
      ]
    }
  ]
}
```

### 구조 원칙

* sections는 4개 고정
* 메시지는 3개 고정
* 묵상과 적용은 3개 항목 고정
* 기도문은 paragraph 1개 고정

## 7. 연령별 생성 구조

## 7.1 연령 구분

다음 audience 값을 사용한다.

* `adult` : 장년
* `young_adult` : 청년
* `teen` : 중고등부
* `child` : 어린이

## 7.2 핵심 원칙

연령별로 **JSON 구조는 바뀌지 않는다.**

달라지는 것은 다음뿐이다.

* 표현 수준
* 문장 길이
* 어휘 선택
* 적용 방식
* 기도문의 길이와 톤

즉 연령별 차이는 **prompt 규칙**에서 처리한다.

## 8. Step 구조 재설계

## 8.1 공통 단계

### 공통 Step. 설교 원문 준비

이 단계는 한 번만 수행한다.

수행 항목:

* 영상/오디오 입력
* 전사
* 공통 메타 입력
* 본문 성구 확정

결과:

* `sermon_source.json`

## 8.2 연령대별 반복 단계

### Step1. 연령대별 생성 조건 확인

입력:

* audience 선택
* template 선택
* 필요 시 옵션 확인

결과:

* request JSON 생성

### Step2. AI QT 생성

입력:

* 공통 원문
* 공통 메타
* base prompt
* audience prompt

결과:

* audience별 output JSON 생성

예:

* `qt_adult.json`
* `qt_young_adult.json`
* `qt_teen.json`
* `qt_child.json`

### Step3. QT 내용 검토/편집

입력:

* output JSON

역할:

* 구조화 편집
* HTML preview 제공
* audience별 결과 개별 수정

### Step4. 산출물 생성

입력:

* 최종 output JSON
* template_id

출력:

* HTML
* PDF
* DOCX
* PPTX

## 9. 프롬프트 운영 방향

## 9.1 기본 구조

프롬프트는 다음 3개 요소를 조합한다.

1. 공통 base prompt
2. audience별 규칙 prompt
3. request JSON

즉:

```text
final_prompt = base_prompt + audience_prompt + request_json
```

## 9.2 파일 구성 권장안

* `qt_prompt_base.txt`
* `qt_prompt_adult.txt`
* `qt_prompt_young_adult.txt`
* `qt_prompt_teen.txt`
* `qt_prompt_child.txt`

이렇게 분리하면 유지보수가 쉽다.

## 10. 향후 구현 우선순위

## 10.1 1차 우선순위

1. 공통 원문 모델 확정
2. QT output JSON 스키마 확정
3. base/audience 프롬프트 파일 분리
4. Step 구조 재정의

## 10.2 2차 우선순위

1. Step2에서 JSON 결과 받기
2. Step3에서 JSON 기반 편집 UI 설계
3. JSON → HTML 렌더링 연결

## 10.3 3차 우선순위

1. JSON → PDF 연결
2. Python 기반 DOCX/PPTX POC
3. template_id 기반 렌더링 확장

---

## 11. 프로젝트 진행 원칙

앞으로 본 프로젝트는 다음 원칙을 기준으로 진행한다.

1. HTML 파싱 중심 구조로 되돌아가지 않는다.
2. JSON을 중심 데이터로 사용한다.
3. QT 구조는 공통 스키마로 유지한다.
4. 연령별 차이는 prompt와 표현 규칙에서만 처리한다.
5. 전사는 한 번, 생성은 여러 번 수행하는 구조로 간다.
6. 모든 산출물은 JSON을 기준으로 생성한다.

---

## 12. 결론

이번 작업을 통해 확인된 가장 중요한 결론은 다음과 같다.

* QT 작성기는 HTML 편집기가 아니라 JSON 중심 생성기 구조가 적합하다.
* 설교 전사는 공통 1회 수행이 적절하다.
* 장년/청년/중고등부/어린이용 QT는 audience별로 반복 생성하는 구조가 맞다.
* HTML/PDF/DOCX/PPTX는 모두 JSON 기반으로 생성해야 한다.

따라서 향후 프로젝트는
**공통 원문 1회 확보 + audience별 JSON 생성/편집/산출물 생성**
구조를 기준으로 추진한다.

```

이 문서를 기준으로 다음 대화부터는  
**1) 공통 원문 모델 확정 → 2) output JSON 스키마 확정 → 3) Step 화면 정의**  
순서로 진행하면 좋겠습니다.
```
