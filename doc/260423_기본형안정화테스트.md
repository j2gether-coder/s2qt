# S2QT 기본형(A4 세로) 산출물 안정화 테스트 항목

## 1. 목적

상세형(A4 가로)은 일단 보류하고, 현재 운영 중인 **기본형(A4 세로)** 산출물의 안정성을 먼저 확보한다.

이번 테스트의 목적은 다음과 같다.

1. Step1 프롬프트 수정 이후 산출물 품질이 안정적으로 유지되는지 확인
2. Step2 미리보기와 Step3 최종 산출물(PDF/PNG)의 일관성을 검증
3. A4 한 장 기준에서 여백, 줄 수, 문장 길이, 정보량 한계를 확인
4. 템플릿/스타일/출력 구조를 실제 운영 가능한 수준으로 다듬는다

현재 구조는 Step2가 `temp.json`을 읽고 편집/저장하며 `temp.html`을 생성하고, Step3가 이 `temp.html` 기반으로 PDF/PNG를 생성하는 흐름이다. 따라서 산출물 안정화의 핵심은 **Step2 HTML과 Step3 출력의 일치성**을 점검하는 것이다.  

---

## 2. 테스트 범위

### 포함 범위

* 기본형(A4 세로) 기준 산출물
* Step2 미리보기 HTML
* Step3 PDF 출력
* Step3 PNG 출력
* 최근 수정된 관련 성구 기능
* 성구 정규화 결과
* 프롬프트 수정 이후 JSON 결과 품질

### 제외 범위

* 상세형(A4 가로)
* 성경/찬송가 DB 구축
* 상세형용 추가 입력 데이터
* 상세형 전용 UI/레이아웃

---

## 3. 현재 기준선

현재 기본형 산출물은 아래 요소를 포함한다.

* 제목
* 본문 성구
* 찬송
* 관련 성구
* 말씀의 창
* 오늘의 메시지
* 깊은 묵상과 적용
* 오늘의 기도
* footer 문구

최근 최종 PDF 결과에서는 `본문 성구: 데살로니가전서 4:13~18`, `관련 성구: 요한복음 14:3, 고린도전서 15:51-52`로 정상 반영되는 것이 확인되었다. 즉, 관련 성구 기능과 성구 표기 정규화는 현재 기준선에서 정상 동작하는 상태이다. 

---

## 4. 테스트 원칙

1. **A4 한 장 유지 가능성**을 우선 검증한다.
2. 미리보기와 최종 출력 결과가 최대한 일치해야 한다.
3. 프롬프트 수정으로 인한 텍스트 길이 변화가 산출물에 어떤 영향을 주는지 확인한다.
4. PDF와 PNG의 차이를 별도로 확인한다.
5. 수정 후에는 반드시 같은 케이스로 재검증한다.
6. 명백한 레이아웃 깨짐은 즉시 수정 대상으로 분류한다.
7. 애매한 부분은 “필수 수정”과 “선택 조정”으로 나누어 기록한다.

## 5. 핵심 테스트 항목

## 5.1 상단 정보 영역 테스트

상단 정보는 현재 기본형에서 다음 순서로 출력된다.

1. 제목
2. 본문 성구
3. 찬송
4. 관련 성구

이 영역은 문서의 첫 인상과 A4 압축도에 큰 영향을 준다. 특히 `qt-subbox`는 여러 줄 정보를 담고 있으므로 길이에 따라 전체 문서 높이에 영향을 준다.  

### 테스트 항목

* 제목이 짧을 때
* 제목이 길 때
* 본문 성구가 짧을 때
* 본문 성구가 길 때
* 관련 성구가 0개일 때
* 관련 성구가 1개일 때
* 관련 성구가 2~3개일 때
* 관련 성구 문자열이 길 때
* 찬송 값이 있을 때
* 찬송 값이 비어 있을 때

### 확인 포인트

* 줄바꿈이 자연스러운가
* 상단 정보가 본문 영역을 과도하게 밀어내지 않는가
* 본문 성구 / 관련 성구 형식이 통일되는가
* 정보가 많아도 A4 한 장 흐름을 크게 깨지 않는가

---

## 5.2 말씀의 창 테스트

`말씀의 창`은 본문 요약 역할을 하며 상단 이후 가장 먼저 읽히는 본문 영역이다. 요약이 길어질수록 전체 문서 높이에 직접 영향이 간다. 현재 Step2 HTML에서도 본문 요약은 paragraph 기준으로 렌더링된다.  

### 테스트 항목

* 3문장 수준
* 5~6문장 수준
* 문장이 짧은 경우
* 문장이 긴 경우
* 줄바꿈 없는 긴 문단
* 문장부호가 많은 경우

### 확인 포인트

* 문단 간격이 과하지 않은가
* 본문 폭에서 가독성이 유지되는가
* 하단 메시지 영역을 과도하게 밀지 않는가

---

## 5.3 오늘의 메시지 테스트

오늘의 메시지는 3개 항목 고정 구조이며, 실제 길이에 따라 A4 한 장 유지 여부가 크게 흔들릴 수 있는 영역이다. 현재 JSON schema도 message_title 3개와 paragraph 3개를 전제로 한다.  

### 테스트 항목

* 메시지 3개 모두 짧은 경우
* 메시지 1개만 긴 경우
* 메시지 3개 모두 긴 경우
* 제목은 짧고 본문이 긴 경우
* 제목 자체가 긴 경우
* 성구 절 표기가 포함된 긴 제목

### 확인 포인트

* 메시지 3개가 모두 자연스럽게 구분되는가
* 제목과 본문 간격이 적절한가
* 긴 메시지가 다음 섹션을 무리하게 밀지 않는가
* A4 한 장 내에서 가장 먼저 무너지는 지점이 어디인지 확인

---

## 5.4 깊은 묵상과 적용 테스트

이 영역은 현재 list 3개 항목으로 고정되어 있다. 항목별 길이가 길어지면 세로 공간이 빠르게 증가한다.  

### 테스트 항목

* 항목이 모두 짧은 경우
* 1개만 긴 경우
* 3개 모두 긴 경우
* 한 줄형 문장
* 두 줄 이상 넘어가는 문장

### 확인 포인트

* 리스트 들여쓰기 가독성
* 줄바꿈 시 bullet/문장 정렬 상태
* 긴 적용문이 너무 답답하게 보이지 않는지

---

## 5.5 오늘의 기도 테스트

기도문은 감성적으로 중요한 영역이지만, 길이에 따라 A4 한 장 기준에서 가장 민감한 영역 중 하나다. 현재 PDF 결과에서도 기도문 길이가 늘어나면 하단 여백과 footer를 빠르게 압박할 수 있다. 

### 테스트 항목

* 3문장 기도문
* 5~6문장 기도문
* 한 문장이 긴 경우
* 여러 문장이 짧은 경우
* 줄바꿈이 많은 경우

### 확인 포인트

* 하단 박스/배경/간격 유지 여부
* footer와 충돌하지 않는가
* A4 한 장 기준에서 마지막 영역이 잘리지 않는가

---

## 5.6 관련 성구 기능 테스트

관련 성구는 최근 추가된 기능이므로 별도 안정화 테스트가 필요하다. 현재 `support_scriptures`는 배열 구조이며, Step2 미리보기와 최종 PDF에서 정상 출력된 상태가 확인되었다.  

### 테스트 항목

* 관련 성구 없음
* 관련 성구 1개
* 관련 성구 2개
* 관련 성구 3개
* 짧은 성구명
* 긴 성구명
* 약어 입력 후 정식 성경명 변환
* 콤마 구분 입력 저장/복원

### 확인 포인트

* 배열 저장이 유지되는가
* Step2 입력값이 Step3 산출물까지 일관되게 반영되는가
* 본문 성구와 관련 성구 표기 방식이 통일되는가

---

## 5.7 성구 정규화 테스트

현재는 약어 입력을 허용하되, 출력에서는 정식 성경명으로 통일하는 방향으로 정리되었다. 이 기능은 Step2 HTML과 최종 산출물의 인상에 직접 영향을 준다. 

### 테스트 예시

* `살전 4:13~18` → `데살로니가전서 4:13~18`
* `고전 15:51-52` → `고린도전서 15:51-52`
* `요 14:3` → `요한복음 14:3`

### 확인 포인트

* 저장 후 다시 열어도 같은 형식인가
* PDF/PNG에서도 같은 형식인가
* 약어/정식명이 혼재되지 않는가

---

## 5.8 PDF / PNG 일관성 테스트

현재 Step3는 `temp.html`을 기반으로 PDF와 PNG를 생성하므로, 이론상 내용은 같아야 하지만 실제 출력에서는 줄바꿈, 여백, 위치, footer 처리 차이가 있을 수 있다.  

### 테스트 항목

* 같은 입력으로 HTML / PDF / PNG 모두 생성
* 짧은 문서 / 긴 문서 각각 생성
* footer 포함 상태 확인
* QR 포함 상태 확인
* 본문이 긴 경우 PDF와 PNG 차이 비교

### 확인 포인트

* 줄바꿈 위치 차이
* footer 잘림 여부
* QR 위치 차이
* 상단 여백 / 하단 여백 일관성
* PDF는 한 장인데 PNG는 잘리거나 반대로 넘치지 않는지

---

## 5.9 여백 및 안전 영역 테스트

현재 가장 중요한 안정화 포인트 중 하나는 **A4 한 장 기준에서 실제로 어디까지 허용되는가**를 찾는 것이다. 즉, “몇 글자 / 몇 줄 / 몇 문단까지 안전한가”를 테스트해야 한다.

### 테스트 항목

* 제목 길이 증가
* summary 길이 증가
* message 길이 증가
* prayer 길이 증가
* 관련 성구 길이 증가
* 여러 항목이 동시에 길어지는 경우

### 확인 포인트

* 상단/하단 여백이 무너지지 않는가
* footer가 본문과 겹치지 않는가
* QR이 본문과 충돌하지 않는가
* 한계치를 넘었을 때 어느 영역부터 먼저 깨지는가

## 6. 테스트 케이스 구분

### A. 정상 길이 케이스

일반적인 분량의 QT 결과
→ 현재 기준선 확인용

### B. 긴 텍스트 케이스

제목, summary, message, prayer가 모두 길게 생성된 케이스
→ A4 한 장 한계 확인용

### C. 관련 성구 다량 케이스

관련 성구가 2~3개이고 각 문자열도 긴 케이스
→ 상단 정보 안정성 확인용

### D. 최소 정보 케이스

찬송 없음, 관련 성구 없음, 전체 본문도 짧은 케이스
→ 빈 공간/균형 확인용

### E. 성구 정규화 케이스

약어 입력 중심 케이스
→ 출력 정규화 확인용

## 7. 판정 기준

## 필수 수정

* A4 한 장 기준이 무너지는 경우
* 본문/기도문/메시지가 잘리는 경우
* footer/QR과 본문이 겹치는 경우
* PDF와 PNG 결과 차이가 과도한 경우
* 관련 성구 / 성구 정규화가 깨지는 경우

## 선택 조정

* 줄간격/여백이 조금 답답한 경우
* 제목 크기/간격이 다소 과한 경우
* 관련 성구 한 줄 표현 방식이 미세하게 아쉬운 경우
* section 간 간격의 시각적 균형 조정

## 8. 우선순위

1. **A4 한 장 유지 여부**
2. **Step2 미리보기와 Step3 최종 산출물 일치성**
3. **PDF / PNG 일관성**
4. **상단 정보(본문 성구/찬송/관련 성구) 안정성**
5. **성구 정규화 품질**
6. **시각적 미세 조정**

## 9. 이번 단계의 결론

상세형(A4 가로)은 일단 보류하고, 현재는 **기본형(A4 세로) 산출물 안정화 테스트를 먼저 완료**한다.

즉, 이번 단계의 핵심은 다음과 같다.

* 산출물의 실제 한계치 확인
* 프롬프트 수정 영향 점검
* 여백/출력/문장 길이 안정성 검증
* 관련 성구 기능 포함 상태에서 기본형 품질 고정

이 작업이 충분히 끝난 뒤에야 상세형(A4 가로)을 다시 검토하는 것이 적절하다.
