# 서버형 작업 절차 정리

## 1. 개요

서버형은 계정 기반으로 동작하는 구독 서비스 구조를 전제로 한다.  
핵심 원칙은 다음과 같다.

- 로그인 후에만 서버 데이터에 접근할 수 있도록 한다.
- 서버에는 원본 음성, 설교문, 전사 결과, 구조정리 중간결과를 저장하지 않는다.
- 최종 QT와 최소 메타정보만 저장한다.
- 구독형 사용자는 서버에 저장된 QT를 PC/모바일 어디서나 열람할 수 있다.
- 신규 생성만 월 사용량으로 카운트하고, 재열람/재사용은 사용량에 포함하지 않는다.
- 인증, 세션, 구독 상태, 결제 이력, 감사 로그를 분리 관리한다.

즉 서버형은  
**콘텐츠 저장소 + 구독 운영 시스템 + 인증/보안 시스템**  
을 함께 가진 구조이다.

---

## 2. 기본 보안 원칙

### 2.1 로그인 방식
- 로그인 ID는 **이메일 주소 전체**를 사용한다.
- 비밀번호는 매번 입력받는다.
- 로그인 성공 후 세션 또는 토큰을 발급한다.
- 멀티디바이스 접근을 허용한다.

### 2.2 비밀번호 처리
- 비밀번호는 평문 저장하지 않는다.
- 비밀번호 해시만 저장한다.
- 해시 알고리즘은 **Argon2id 우선**으로 한다.
- 로그인 실패 횟수 제한 및 잠금 정책을 적용할 수 있다.

### 2.3 세션 처리
- 로그인 성공 시 세션 발급
- 세션 만료시간 관리
- 로그아웃 시 세션 폐기
- 필요 시 다중 세션 관리

### 2.4 저장 원칙
서버에는 다음만 저장한다.

- 설교 제목
- 설교자
- 본문 reference
- 언급 성구 reference
- 최종 QT Markdown
- 최소 검색 메타
- 구독 상태
- 결제 이력
- 감사 로그

저장하지 않는 것:
- 원본 음성 파일
- 원본 설교문
- 전사 결과
- 구조정리 중간결과
- 임시 PDF

## 3. 최초 가입 절차

### 3.1 가입 화면
입력 항목:
- 이메일 주소 전체
- 비밀번호
- 비밀번호 확인
- 표시 이름(선택)

### 3.2 가입 처리
1. 이메일 중복 확인
2. 사용자 계정 생성
3. 비밀번호 해시 생성 후 저장
4. 기본 사용자 상태 생성
5. 필요 시 이메일 인증 토큰 발급
6. 체험 구독 또는 기본 플랜 설정

### 3.3 가입 결과
- 서버형 계정 생성 완료
- 필요 시 이메일 인증 대기
- 이후 로그인 가능

---

## 4. 로그인 절차

### 4.1 로그인 화면
입력 항목:
- 이메일 주소 전체
- 비밀번호

### 4.2 로그인 처리
1. 이메일 기준 사용자 조회
2. 비밀번호 검증
3. 계정 상태 확인
4. 실패 횟수 및 잠금 상태 확인
5. 로그인 성공 시 세션 발급
6. 마지막 로그인 시각 갱신
7. 감사 로그 기록

### 4.3 로그인 결과
- 내 문서 열람 가능
- 구독 상태 확인 가능
- 신규 생성 가능 여부 확인 가능
- PC/모바일 어디서나 접속 가능

---

## 5. 구독 상태 확인 절차

로그인 후 서버는 사용자의 구독 상태를 확인한다.

확인 항목:
- 무료 / 체험 / 월간 / 연간 여부
- 시작일 / 종료일
- 현재 월 신규 생성 가능 횟수
- 유료 템플릿 구매 여부

이 단계는 이후 작업 가능 범위를 결정한다.

---

## 6. 메인 화면 진입

로그인 후 사용자는 메인 화면에서 아래 기능을 선택할 수 있다.

- 새 QT 생성
- 기존 QT 열람
- 검색
- 템플릿 보기
- 유료 템플릿 구매
- 구독 상태 확인
- 설정
- 로그아웃

---

## 7. 새 QT 생성 절차

### 7.1 입력 방식 선택
사용자는 입력 소스를 선택한다.

- YouTube
- 음성 파일
- 설교문 텍스트

현재 구조에서는 전사는 client에서 수행한다.

---

## 8. 입력 데이터 준비

### 8.1 YouTube
- client가 youtube id 확인
- 자막 또는 기존 텍스트가 있으면 우선 활용
- 필요한 메타정보 입력 또는 보정

### 8.2 음성 파일
- client에서 저품질 전사 수행
- 원본 음성은 서버에 저장하지 않음

### 8.3 설교문 텍스트
- 사용자가 직접 입력

---

## 9. 메타정보 입력 및 보정

최소 입력 또는 확인 항목:
- 설교 제목
- 설교자
- 본문 reference
- tradition

원칙:
- 서버에는 원본 저장 안 함
- 생성에 필요한 최소 입력만 사용

---

## 10. 신규 생성 여부 판단

서버형에서 중요한 단계이다.

서버는 요청을 받으면 먼저 아래를 확인한다.

1. 사용자의 월 신규 생성 가능 횟수가 남아 있는지
2. 같은 문서를 재열람하는 것인지
3. 이미 저장된 QT를 재사용하는 것인지
4. 정말 새로운 생성 요청인지

원칙:
- **신규 생성만 사용량 차감**
- **재열람 / 재사용 / 재출력은 미차감**

---

## 11. 구조정리 절차

신규 생성으로 판단되면 서버가 구조정리를 수행한다.

### 11.1 입력
- client에서 전달받은 텍스트
- 설교 제목
- 설교자
- 본문 reference
- tradition

### 11.2 출력
- 구조화된 설교 흐름
- 핵심 메시지
- QT 생성용 정리 결과

### 11.3 저장 정책
- 구조정리 결과는 저장하지 않는다
- 세션 처리 중간단계에서만 사용한다

---

## 12. QT 작성 절차

서버가 구조정리 결과를 바탕으로 QT를 작성한다.

### 12.1 입력
- 구조정리 결과
- 제목
- 설교자
- 본문 reference
- tradition
- 템플릿 옵션 또는 기본 규칙

### 12.2 출력
- 최종 QT Markdown
- 요약
- primary_ref_text
- 언급 성구 reference 목록
- 필요 시 찬송 추천(기독교만)

### 12.3 저장 정책
- 최종 QT만 저장 대상이다
- 중간 산출물은 저장하지 않는다

---

## 13. 사용량 차감 절차

QT 생성이 성공하면:
- 해당 월 신규 생성 횟수 1 증가
- 사용량 테이블 갱신
- 감사/운영 로그 기록 가능

실패 시:
- 사용량 차감 여부는 정책적으로 결정
- 일반적으로는 **완전 성공 시만 차감**하는 것이 적절하다

---

## 14. QT 저장 절차

서버에는 최종 결과만 저장한다.

### 14.1 저장 대상
- user_id
- 설교 제목
- 설교자
- tradition
- source_type
- generation_mode
- primary_ref_text
- qt_markdown
- qt_summary
- has_hymn_recommend
- 본문 reference
- 언급 성구 reference
- 생성 시각
- 수정 시각

### 14.2 저장하지 않는 것
- 원본 음성
- 원본 설교문
- 전사 결과
- 구조정리 결과

---

## 15. 기존 QT 열람 절차

사용자는 서버에 저장된 기존 QT를 열람할 수 있다.

특징:
- PC/모바일 어디서나 가능
- 구독형 핵심 차별점
- 별도 생성 없이 즉시 열람 가능

가능 기능:
- QT VIEW
- 제목 / 설교자 / 본문 확인
- 요약 확인
- 템플릿 적용
- 재사용

---

## 16. 검색 절차

검색 기준은 아래와 같이 단순화한다.

- 설교 제목
- 설교자
- 본문 reference

즉 서버형은 개인 클라우드 QT 라이브러리처럼 동작한다.

---

## 17. 기본 열람 절차

주력 경험은 PDF보다 VIEW / eBook 스타일이다.

특징:
- 모바일/태블릿 친화
- 멀티디바이스 열람
- 로그인만 하면 접근 가능

---

## 18. PDF 출력 절차

PDF는 보조 포맷으로 본다.

정책:
- 무료 템플릿 PDF 가능
- 유료 템플릿 PDF 가능
- 출력은 client 또는 별도 처리 구조에 맞춘다
- 서버는 권한/구매 여부 판단 중심으로 동작한다

---

## 19. 구독 시작 절차

사용자는 월간 또는 연간 구독을 시작할 수 있다.

구독 유형:
- 월간
- 연간
- 체험(예: 30일)

처리:
1. 결제 요청
2. PG 결제 승인
3. 결제 이력 저장
4. 구독 상태 갱신
5. 월 quota 설정

---

## 20. 구독 갱신 및 만료 절차

서버는 구독 상태를 주기적으로 확인한다.

처리:
- 종료일 경과 시 만료 처리
- 갱신 성공 시 연장 처리
- 갱신 실패 시 상태 변경

---

## 21. 유료 템플릿 구매 절차

사용자는 유료 템플릿을 별도 구매할 수 있다.

처리:
1. 템플릿 선택
2. 결제 진행
3. 구매 이력 저장
4. 템플릿 사용 권한 활성화

---

## 22. 로그아웃 절차

사용자가 로그아웃하면:
- 세션 폐기
- refresh token 또는 session token 무효화
- 클라이언트는 로그인 상태 해제

---

## 23. 비밀번호 변경 절차

입력:
- 현재 비밀번호
- 새 비밀번호

처리:
1. 현재 비밀번호 검증
2. 새 비밀번호 해시 저장
3. 필요 시 세션 재발급 또는 다른 세션 만료

---

## 24. 비밀번호 재설정 절차

입력:
- 이메일 주소

처리:
1. 재설정 토큰 발급
2. 이메일 발송
3. 토큰 검증
4. 새 비밀번호 설정

---

## 25. 감사 로그 절차

아래 이벤트는 감사 로그 대상이다.

- 로그인 성공
- 로그인 실패
- 비밀번호 변경
- 비밀번호 재설정
- 구독 시작 / 만료 / 취소
- 유료 템플릿 구매
- 보안 관련 이벤트

---

## 26. 전체 흐름 요약

### 최초 사용
1. 가입
2. 이메일 인증(선택 또는 권장)
3. 로그인
4. 체험/구독 상태 확인
5. 생성 또는 열람 시작

### 일반 사용
1. 로그인
2. 구독 상태 및 quota 확인
3. 새 QT 생성 또는 기존 QT 열람
4. 신규 생성 시 구조정리 → QT 작성 → 저장
5. 재열람/재사용은 무제한
6. 로그아웃 또는 세션 만료

---

## 27. 최종 원칙 요약

- 로그인 후에만 서버 데이터 접근 가능
- 이메일 주소 전체를 로그인 ID로 사용
- 비밀번호는 매번 입력
- 신규 생성만 quota 차감
- 재열람/재사용은 quota 미차감
- 서버에는 최종 QT와 최소 메타만 저장
- 원본/전사/구조정리 결과는 저장하지 않음
- 구독형 핵심 차별점은 멀티디바이스 VIEW
- 인증 / 세션 / 구독 / 결제 / 감사 로그를 분리 관리

---

## 28. 한 줄 정리

**서버형은 “로그인 → 구독 상태 확인 → 신규 생성 시 서버 구조정리/QT 작성 → 최종 QT 저장 → PC/모바일 어디서나 열람” 흐름으로 설계한다.**
