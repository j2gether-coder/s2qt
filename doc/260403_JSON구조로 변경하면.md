# QT 문서 생성기 JSON 중심 구조 설계안

## 1. 문서 개요

본 문서는 QT 문서 생성기의 산출물 생성 구조를 기존 **HTML 중심 방식**에서 **JSON 중심 방식**으로 전환하기 위한 설계안을 정의한다.

기존 구조에서는 LLM이 HTML 형태의 본문을 생성하고, 이를 기반으로 HTML/PDF/DOCX/PPTX를 생성하려고 하였다.  
그러나 이 방식은 HTML이 표현 중심 포맷이라는 한계로 인해 문서 구조 복원, 섹션 구분, 산출물 간 일관성 유지 측면에서 여러 문제를 야기하였다.

이에 따라 본 설계안에서는 **JSON을 원본(Source of Truth)** 으로 정의하고, HTML/PDF/DOCX/PPTX는 모두 해당 JSON을 기반으로 생성하는 구조를 제안한다.

---

## 2. 추진 배경

### 2.1 기존 HTML 중심 구조의 한계

기존 방식은 다음과 같은 흐름을 가진다.

- Step1: 기본정보 입력
- Step2: LLM이 HTML 본문 생성
- Step3: HTML 편집
- Step4: HTML을 기반으로 PDF/WORD/PPT 생성

이 방식의 주요 문제점은 다음과 같다.

#### 1) HTML은 표현 중심 포맷이다
HTML은 화면 렌더링을 위한 표현 포맷이며, 문서의 의미 구조를 안정적으로 유지하는 데 불리하다.

예를 들어 다음과 같은 문제가 발생하였다.

- `오늘의 메시지`가 일반 `<h2>`인지, 특정 class인지에 따라 구조 인식이 달라짐
- `오늘의 기도`가 별도 섹션인지, 특정 박스 내부 블록인지 파서가 해석해야 함
- footer와 같은 표현 요소를 문서 구조에서 제외해야 하는 추가 로직이 필요함

#### 2) 구조 복원 과정에서 의미 손실이 발생한다
기존 구조는 사실상 아래 흐름을 가진다.

- 문서 구조 → HTML → 다시 문서 구조 복원

이 과정에서 다음과 같은 문제가 발생할 수 있다.

- 섹션 분리가 불완전함
- DOCX/PPTX에서 섹션 구분이 약해짐
- 동일 본문이라도 산출물별 구조가 달라질 가능성이 생김

#### 3) 산출물 생성 기준이 불명확하다
HTML이 원본처럼 동작하므로, 다음과 같은 충돌이 생길 수 있다.

- Step1 최신 제목 vs HTML 안 제목
- 본문/찬송/제목의 우선순위 충돌
- DOCX/PPTX 생성 시 HTML 파싱 규칙 의존

## 3. 목표 구조

### 3.1 기본 원칙

본 설계의 핵심 원칙은 다음과 같다.

1. **JSON을 문서의 유일한 원본으로 정의한다.**
2. HTML은 미리보기 및 웹 렌더링 결과물로 생성한다.
3. PDF, DOCX, PPTX는 모두 JSON을 기준으로 생성한다.
4. QT 문서의 구조는 고정하고, 내용 표현만 연령층/템플릿에 따라 달라지도록 한다.

### 3.2 목표 아키텍처

```text
Step1 기본정보 입력
  - 제목
  - 본문 성구
  - 찬송
  - 연령층
  - 템플릿 선택(향후)

Step2 LLM 생성
  - QT JSON 생성

Step3 편집
  - JSON 기반 편집
  - HTML Preview 렌더링

Step4 산출물 생성
  - JSON -> HTML
  - JSON -> PDF
  - JSON -> DOCX
  - JSON -> PPTX
````

## 4. JSON 중심 구조의 장점

### 4.1 문서 구조가 명확해진다

QT 문서의 섹션 구조를 JSON 스키마로 강제할 수 있으므로, 섹션 해석 과정이 필요 없어진다.

예:

* 말씀의 창
* 오늘의 메시지
* 깊은 묵상과 적용
* 오늘의 기도

이와 같은 섹션은 JSON에서 명시적으로 정의된다.

### 4.2 산출물 간 일관성이 높아진다

HTML, PDF, DOCX, PPTX가 모두 동일한 JSON을 기준으로 생성되므로, 산출물별 구조 차이가 줄어든다.

### 4.3 템플릿 적용이 쉬워진다

내용(JSON)과 표현(템플릿)을 분리할 수 있으므로, 동일한 QT 내용을 다양한 스타일로 출력할 수 있다.

### 4.4 연령층별 분기가 쉬워진다

장년, 청년, 중고등부, 어린이용 문서를 모두 동일한 구조로 유지하되, 표현 수준만 다르게 생성할 수 있다.

## 5. QT 문서의 공통 구조 정의

QT 문서는 기본적으로 다음 구조를 가진다.

1. 제목
2. 본문 정보

   * 본문 성구
   * 찬송
3. 말씀의 창: 본문 요약
4. 오늘의 메시지
5. 깊은 묵상과 적용
6. 오늘의 기도

이 구조는 연령층과 무관하게 공통으로 유지한다.

## 6. QT JSON 스키마 초안

## 6.1 최상위 구조

```json
{
  "version": "1.0",
  "doc_type": "qt",
  "audience": "adult",
  "template_id": "qt_classic",
  "title": "",
  "bible_text": "",
  "hymn": "",
  "sections": []
}
```

### 필드 설명

| 필드명         | 설명                 |
| ----------- | ------------------ |
| version     | JSON 스키마 버전        |
| doc_type    | 문서 유형. 현재는 `qt` 고정 |
| audience    | 대상 연령층             |
| template_id | 렌더링 템플릿 식별자        |
| title       | QT 문서 제목           |
| bible_text  | 본문 성구              |
| hymn        | 찬송                 |
| sections    | QT 본문 섹션 목록        |

---

## 6.2 audience 정의

| 값           | 설명   |
| ----------- | ---- |
| adult       | 장년   |
| young_adult | 청년   |
| teen        | 중고등부 |
| child       | 어린이  |

필요 시 화면에는 한글로 보이되 내부 값은 영문 enum을 사용하는 것을 권장한다.

---

## 6.3 section 구조

```json
{
  "type": "summary",
  "title": "🌿 말씀의 창: 본문 요약",
  "blocks": []
}
```

### 필드 설명

| 필드명    | 설명          |
| ------ | ----------- |
| type   | 섹션 유형       |
| title  | 섹션 제목       |
| blocks | 섹션 내부 블록 목록 |

---

## 6.4 section type 정의

| 값          | 설명           |
| ---------- | ------------ |
| summary    | 말씀의 창: 본문 요약 |
| message    | 오늘의 메시지      |
| reflection | 깊은 묵상과 적용    |
| prayer     | 오늘의 기도       |

---

## 6.5 block 구조

블록은 섹션 내부의 실제 콘텐츠 단위이다.

### 1) paragraph

```json
{
  "type": "paragraph",
  "text": "문단 내용"
}
```

### 2) message_title

```json
{
  "type": "message_title",
  "text": "1. 나는 그리스도와 함께 십자가에 못 박힌 존재입니다"
}
```

### 3) list

```json
{
  "type": "list",
  "items": [
    "항목 1",
    "항목 2",
    "항목 3"
  ]
}
```

## 7. 연령층별 확장 방안

## 7.1 기본 방향

연령층별 문서는 **스키마를 다르게 가져가는 것이 아니라**, 동일 스키마를 유지한 상태에서 **내용 표현만 다르게 생성**한다.

즉 다음과 같이 운영한다.

* 스키마: 공통
* 렌더러: 공통
* 프롬프트 규칙: 연령층별 차등

## 7.2 연령층별 생성 원칙

### 장년(adult)

* 차분하고 깊이 있는 어조
* 신앙적 성찰과 적용 강조
* 문장 길이는 다소 길어도 무방

### 청년(young_adult)

* 현실 고민과 연결
* 진로, 관계, 선택, 정체성 중심
* 너무 딱딱하지 않게 구성

### 중고등부(teen)

* 학교생활, 친구관계, 습관, 감정과 연결
* 쉬운 어휘 사용
* 문장을 짧고 명확하게 작성

### 어린이(child)

* 아주 쉬운 단어 사용
* 짧은 문장 중심
* 추상 개념 최소화
* 적용 문항을 단순화

---

## 8. 템플릿 확장 구조

## 8.1 기본 개념

JSON은 콘텐츠 구조를 나타내고, 템플릿은 표현 방식을 나타낸다.

즉 다음과 같이 분리한다.

* JSON: 내용
* template_id: 스타일/레이아웃

## 8.2 예시 template_id

| template_id  | 설명          |
| ------------ | ----------- |
| qt_classic   | 기본형         |
| qt_modern    | 모던형         |
| qt_youth     | 청년/중고등부 친화형 |
| qt_kids      | 어린이 친화형     |
| qt_print     | 인쇄 중심형      |
| qt_ppt_basic | 발표용 기본형     |

## 8.3 기대 효과

동일한 JSON으로도 다음이 가능하다.

* 장년용 클래식 DOCX
* 청년용 모던 PDF
* 어린이용 친화형 PPT
* 교회별 커스터마이징 템플릿

---

## 9. 단계별 시스템 구조 재정의

## 9.1 Step1 자료입력

입력 항목 예시:

* 제목
* 본문 성구
* 찬송
* 대상 연령층
* 템플릿(향후)

## 9.2 Step2 생성

LLM은 HTML이 아니라 JSON을 생성한다.

즉 프롬프트는 다음 방향으로 바뀌어야 한다.

* 기존: 보기 좋은 HTML 생성
* 변경: QT JSON 스키마에 맞는 구조화 결과 생성

## 9.3 Step3 편집

Step3는 HTML 직접 편집보다 다음 중 하나로 전환하는 것이 바람직하다.

### 방식 A. JSON 구조 기반 편집

* 섹션별 편집창 제공
* 메시지 1/2/3 개별 편집
* 묵상 적용 리스트 개별 편집
* 기도문 편집

### 방식 B. JSON -> HTML Preview 렌더링

* 내부 저장은 JSON
* 화면에는 HTML 미리보기 제공

## 9.4 Step4 산출물 생성

생성 기준은 모두 JSON이다.

* JSON -> HTML
* JSON -> PDF
* JSON -> DOCX
* JSON -> PPTX

## 10. 구현 권장 방향

## 10.1 단기 방향

1. QT JSON 스키마 확정
2. LLM 프롬프트를 JSON 출력 방식으로 전환
3. HTML은 JSON으로부터 렌더링
4. PDF는 HTML 기반 출력 유지 가능
5. DOCX/PPTX는 JSON 기반 생성으로 전환

## 10.2 중기 방향

1. Step3 편집기를 JSON 중심 구조로 전환
2. Python Worker 또는 템플릿 기반 DOCX/PPTX 생성 모듈 도입
3. 템플릿 체계 분리 적용

## 10.3 장기 방향

1. 연령층별/교회별 템플릿 확장
2. PPTX 발표자료 자동 생성 고도화
3. 교회별 브랜딩 템플릿 적용
4. 문서 유형 확대(QT 외 설교요약, 소그룹 나눔지 등)

## 11. 기대 효과

### 11.1 구조 안정성 향상

문서 구조를 HTML 파싱에 의존하지 않고 JSON으로 명시적으로 관리할 수 있다.

### 11.2 산출물 일관성 확보

HTML/PDF/DOCX/PPTX가 동일한 구조 기준으로 생성된다.

### 11.3 유지보수성 향상

파싱 규칙 복잡도가 줄어들고, 템플릿과 콘텐츠의 책임 분리가 가능해진다.

### 11.4 확장성 확보

연령층별 문체 분기, 템플릿 확장, 출력 포맷 다양화가 쉬워진다.

---

## 12. 결론

QT 문서 생성기는 구조상 자유 문서 생성기가 아니라 **형식이 있는 문서 생성기**에 가깝다.
따라서 표현 중심 포맷인 HTML을 원본으로 두기보다, 구조를 명확히 담을 수 있는 JSON을 원본으로 두는 것이 적합하다.

향후 시스템은 다음 원칙으로 설계하는 것이 바람직하다.

* JSON은 원본
* HTML은 미리보기
* PDF는 인쇄 산출물
* DOCX는 편집/배포 문서
* PPTX는 발표 자료

즉 **JSON 중심 아키텍처**로 전환하는 것이 본 시스템의 구조 안정성, 확장성, 유지보수성을 모두 향상시키는 방향이다.

```

원하시면 다음으로 이 문서를 바탕으로 **더 실무적인 개발설계서 형식**으로 바꾸거나, **바로 사용할 수 있는 JSON 스키마 명세서**로 분리해서 작성해드릴게요.
```
