# S2QT 보안설계 초안
## PIN 모달 UI 구조 + 세션 Unlock 정책 + Settings DB 항목

---

## 1. 개요

S2QT는 작업 생산성이 중요한 데스크톱 앱이므로,  
앱 실행 시마다 로그인하도록 강제하지 않는다.

대신, 민감 기능 사용 시에만 **PIN 기반 잠금 해제**를 요구하고,  
한 번 인증되면 현재 앱 실행 세션 동안 Unlock 상태를 유지하는 방식으로 설계한다.

본 문서는 다음을 정의한다.

- PIN 입력 모달 UI 구조
- 세션 Unlock 정책
- Settings DB 저장 항목
- PIN / 민감정보 처리 원칙

---

## 2. 기본 보안 정책

### 2.1 앱 실행 정책
- 앱 실행 시 로그인 요구 없음
- QT 준비 / Step2 / Step3 / History 조회는 즉시 사용 가능
- 생산성 흐름을 우선함

### 2.2 PIN 인증이 필요한 경우
다음 경우에만 PIN 인증을 요구한다.

1. 연령대별 Step1에서 **자동 LLM API 사용 직전**
2. 환경설정에서 **API KEY 수정 직전**
3. 환경설정에서 **SMTP 비밀번호 수정 직전**
4. 필요 시 환경설정에서 **SMTP 테스트 / API 연결 테스트 전**

### 2.3 PIN 인증이 필요 없는 경우
- 앱 시작 직후
- QT 준비
- Step2
- Step3
- History 목록/조회
- 교회명 / 로고 / 홈페이지 URL 수정
- 라이선스 안내 / 사용법 조회

## 3. PIN 모달 UI 구조

## 3.1 목적
PIN 모달은 앱 전체 로그인창이 아니라,  
민감 기능 사용 전 **보안 확인용 입력 모달**이다.

---

## 3.2 표시 시점

### 예시 1. Step1 자동 LLM 사용
- 사용자가 Step1에서 실행 버튼 클릭
- 세션이 잠금 상태이면 PIN 모달 표시
- 성공 시 Step1 계속 진행

### 예시 2. 환경설정 > AI 설정
- 사용자가 API KEY 수정 버튼 클릭
- 세션이 잠금 상태이면 PIN 모달 표시
- 성공 시 수정 가능 상태로 전환

### 예시 3. 환경설정 > 이메일 설정
- 사용자가 SMTP 비밀번호 수정 버튼 클릭
- 세션이 잠금 상태이면 PIN 모달 표시
- 성공 시 수정 가능 상태로 전환

---

## 3.3 UI 구성 요소

### 모달 제목
- `보안 확인`

### 안내 문구
상황에 따라 문구를 다르게 표시한다.

예:
- `LLM API 사용을 위해 PIN을 입력해 주세요.`
- `민감정보 수정을 위해 PIN을 입력해 주세요.`

### PIN 표시 영역
- 입력된 숫자는 마스킹 처리
- 예: `● ● ● ●`
- 실제 숫자는 화면에 그대로 노출하지 않음

### 가상키보드
숫자 전용 가상키패드 사용

예시 배열:

```text
1 2 3
4 5 6
7 8 9
전체삭제 0 지우기
````

또는

```text
1 2 3
4 5 6
7 8 9
지우기 0 확인
```

### 버튼

* 확인
* 취소
* 지우기
* 전체삭제

---

## 3.4 입력 규칙

* PIN은 숫자만 허용
* 길이는 4~6자리
* 권장 기본값은 6자리
* 입력 완료 후 확인 버튼 활성화
* 입력 중 취소 가능

---

## 3.5 오류 메시지

PIN 입력 실패 시 모달 내부에 inline message로 표시한다.

예:

* `PIN이 올바르지 않습니다.`
* `입력 횟수가 초과되었습니다. 잠시 후 다시 시도해 주세요.`

완료 메시지는 toast로 별도 표시하지 않아도 된다.
성공 시 모달이 닫히고 다음 절차가 진행되면 충분하다.

## 4. 세션 Unlock 정책

## 4.1 기본 원칙

한 번 PIN 인증이 성공하면
**현재 앱 실행 세션 동안 Unlock 상태를 유지**한다.

즉:

* 장년 QT Step1에서 PIN 인증 성공
* 같은 앱 실행 세션 안에서는

  * 청년 QT Step1
  * 중고등부 QT Step1
  * 어린이 QT Step1
    에서 다시 PIN을 묻지 않음

---

## 4.2 세션 종료 시점

다음 시점에 Unlock 상태는 해제된다.

* 앱 종료 시
* 앱 재실행 시

초기 설계에서는 이것만으로 충분하다.

---

## 4.3 확장 가능 정책

추후 필요 시 아래 정책을 추가할 수 있다.

* 일정 시간 미사용 시 자동 잠금
* 민감정보 수정과 API 사용에 대해 서로 다른 잠금 범위
* 수동 잠금 버튼 제공

초기 구현에서는 적용하지 않는다.

---

## 4.4 세션 상태 값 예시

프론트/메모리 상태 예시:

```js
securityState = {
  unlocked: false,
  unlockedAt: '',
  unlockScope: 'session'
}
```

설명:

* `unlocked`: 현재 세션 잠금 해제 여부
* `unlockedAt`: 인증 성공 시각
* `unlockScope`: 현재는 `session` 고정

---

## 5. PIN 사용 정책

## 5.1 PIN의 역할

PIN은 앱 로그인 비밀번호가 아니라
**민감 기능 접근 통제용 보안 수단**이다.

즉:

* 앱 전체 사용 권한을 제어하지 않음
* API KEY / SMTP password 사용 및 수정 전에만 사용함

---

## 5.2 PIN 저장 방식

PIN은 평문 저장하지 않는다.

### 저장 원칙

* DB에는 `PIN 해시값`만 저장
* 입력값 비교 시 해시 비교 수행
* 원본 PIN은 다시 조회할 수 없음

### 정책

* PIN 보기 기능 없음
* PIN 분실 시 재설정 흐름 필요
* PIN 변경 시 기존 PIN 확인 후 새 PIN 등록

---

## 5.3 실패 처리 정책

초기 정책 권장:

* 연속 5회 실패 시 경고
* 필요 시 짧은 대기 시간 부여 가능

초기 버전에서는 단순 실패 메시지 처리로 시작 가능하다.

---

## 6. 민감정보 표시 정책

## 6.1 공통 원칙

저장된 민감정보는 다시 평문으로 보여주지 않는다.

### 대상

* LLM API KEY
* SMTP 비밀번호

### 화면 표시 방식

* API KEY: `등록됨` 또는 `********`
* SMTP 비밀번호: `등록됨` 또는 `********`

즉, 사용자는 “등록 여부”만 확인 가능하고,
기존 값을 평문으로 다시 보는 기능은 제공하지 않는다.

---

## 6.2 수정 정책

민감정보 수정 시 기존 값을 보여주지 않고
새 값을 다시 입력하도록 한다.

예:

* `새 API KEY 입력`
* `새 SMTP 비밀번호 입력`

수정 전에는 PIN 인증이 필요하다.

---

## 7. Settings DB 항목

## 7.1 공통 테이블

환경설정은 key-value 방식의 단일 테이블로 관리한다.

### app_settings

```sql
CREATE TABLE app_settings (
    setting_key TEXT PRIMARY KEY,
    setting_value TEXT,
    value_type TEXT NOT NULL DEFAULT 'text',
    is_secret INTEGER NOT NULL DEFAULT 0,
    updated_at TEXT NOT NULL
);
```

---

## 7.2 보안 관련 setting_key

### PIN / 보안 설정

* `security.pin_enabled`
* `security.pin_hash`
* `security.pin_length`
* `security.unlock_scope`

설명:

* `security.pin_enabled` : PIN 사용 여부 (`true/false`)
* `security.pin_hash` : PIN 해시값
* `security.pin_length` : 4 또는 6
* `security.unlock_scope` : 현재는 `session`

---

## 7.3 AI 설정

* `llm.provider`
* `llm.api_key`
* `llm.model`
* `llm.mode`
* `llm.enabled`

설명:

* `llm.api_key`는 암호화 저장 대상
* 자동 LLM 호출 시 PIN 인증 필요

---

## 7.4 SMTP 설정

* `smtp.enabled`
* `smtp.from_email`
* `smtp.host`
* `smtp.port`
* `smtp.username`
* `smtp.password`
* `smtp.security`

설명:

* `smtp.password`는 암호화 저장 대상
* 수정 및 테스트 시 PIN 인증 필요

---

## 7.5 교회/브랜드 설정

* `church.name`
* `church.logo_path`
* `church.homepage_url`
* `church.default_footer_text`

설명:

* 로고는 파일 경로만 저장
* 일반 설정으로 취급
* PIN 인증 불필요

---

## 7.6 안내/라이선스 설정

* `license.type`
* `license.status`
* `license.expire_date`
* `app.version`
* `app.guide_text`

설명:

* 조회 중심 항목
* PIN 인증 불필요

## 8. value_type / is_secret 정책

## 8.1 value_type 예시

* `text`
* `password`
* `number`
* `boolean`
* `url`
* `multiline`

## 8.2 is_secret 정책

### is_secret = 1

* `llm.api_key`
* `smtp.password`

### is_secret = 0

* provider, model, email, host, port
* church.name
* church.logo_path
* church.homepage_url
* app.guide_text
* license 상태 정보
* security.pin_enabled
* security.pin_length
* security.unlock_scope

참고:

* `security.pin_hash`는 민감값이므로 내부적으로 secret 취급 가능

---

## 9. 암호화 및 해시 정책

## 9.1 PIN

* PIN은 암호화가 아니라 **해시 저장**
* 원본 복원 불가

## 9.2 API KEY / SMTP 비밀번호

* 평문 저장 금지
* 암호화 후 DB 저장
* 읽을 때 복호화 가능
* 단, 화면에는 평문 표시하지 않음

## 9.3 역할 구분

* PIN = 접근 통제
* API KEY / SMTP password = 암호화 저장 대상
* 세션 Unlock = 현재 실행 중 메모리 상태

## 10. 동작 흐름

## 10.1 Step1 자동 LLM 호출

1. 사용자가 Step1 실행 버튼 클릭
2. `securityState.unlocked` 확인
3. 잠금 상태면 PIN 모달 표시
4. PIN 확인 성공
5. 세션 Unlock=true
6. LLM API 호출 진행
7. 같은 세션에서는 다른 audience Step1에서 재입력 없음

---

2. 세션 Unlock 확인
3. 잠금 상태면 PIN 모달 표시
4. 성공 시 수정 입력 가능
5. 저장 시 암호화 후 DB 저장
6. 화면에는 다시 마스킹 상태만 표시

## 10.3 환경설정 > SMTP 비밀번호 수정

4. 성공 시 수정 입력 가능
5. 저장 시 암호화 후 DB 저장

## 13. 최종 정리

### 보안 정책 핵심

* 앱 실행 시 로그인 없음
* 민감 기능 사용 시에만 PIN 확인
* 한 번 인증되면 현재 실행 세션 동안 유지
* 같은 세션에서는 연령별 Step1에서 다시 묻지 않음

### PIN 정책

* 숫자형 PIN
* 가상키보드 모달 입력
* 해시 저장
* 평문 조회 불가

### 민감정보 정책

* API KEY / SMTP 비밀번호는 암호화 저장
* 화면에는 다시 평문 표시하지 않음
* 수정 시 PIN 인증 후 새 값 입력만 허용

### DB 정책

* app_settings key-value 구조 유지
* 보안 항목 추가
* 세션 상태는 DB가 아니라 메모리에서 관리
