# S2QT 업데이트 전략 및 구현 절차

## 1. 문서 목적

본 문서는 S2QT의 향후 배포 및 업데이트 전략을 **update 기준**으로 정리한 기준 문서이다.

목적은 다음과 같다.

* 현재 배포 구조에서 어떤 파일이 업데이트 대상인지 명확히 정의한다.
* `s2qt.exe` 중심의 업데이트 전략을 정리한다.
* 해시값 비교 기반의 업데이트 판단 방식을 정리한다.
* 설치형(Inno Setup) 업데이트 절차를 정리한다.
* 차후 새로운 채팅에서 본 문서를 기준으로 구현 작업을 이어갈 수 있도록 한다.

---

## 2. 현재 전제

### 2.1 애플리케이션 구조

현재 S2QT는 다음 구조를 전제로 한다.

* 루트

  * `bin`
  * `var`

    * `conf`
    * `temp`
    * `data`
    * `model`
    * `log`

실행 파일은 기본적으로 `bin/s2qt.exe`를 기준으로 한다.

### 2.2 현재 파일 역할 구분

#### 앱 본체

* `s2qt.exe`

#### 설치본 포함, 자주 바뀌지 않는 런타임 구성요소

* `whisper-cli.exe`
* whisper 관련 DLL

#### 런타임 복구/다운로드 대상

* `ffmpeg.exe`
* `ffprobe.exe`
* `yt-dlp.exe`
* `ggml-tiny.bin`

#### 사용자/운영 데이터

* `var/conf/*`
* `var/db/*`
* `var/log/*`
* `var/temp/*`
* `var/data/*`

## 3. 업데이트 기준

### 3.1 업데이트 기준의 핵심

현재 구조에서는 실제 업데이트 대상의 핵심은 사실상 `s2qt.exe`이다.

그 이유는 다음과 같다.

* 앱 로직과 프론트 변경은 대부분 `s2qt.exe`에 반영된다.
* `whisper-cli.exe` 및 관련 DLL은 설치본 포함 대상으로 자주 변경하지 않는다.
* `ffmpeg`, `ffprobe`, `yt-dlp`, 모델은 런타임 시 복구/다운로드 정책으로 처리한다.
* `var/conf`, `var/db`, `var/log`는 사용자 데이터이므로 업데이트 시 보존해야 한다.

즉, 기본 업데이트 판단은 **`s2qt.exe` 기준**으로 수행한다.

### 3.2 버전보다 해시값 비교 우선

현재 단계에서는 버전 문자열보다 **해시값 비교가 더 적절하다.**

이유는 다음과 같다.

* 실제 교체 대상이 대부분 `s2qt.exe` 하나이다.
* 버전 문자열은 사람이 누락하거나 잘못 관리할 수 있다.
* 해시값은 실제 파일 내용이 바뀌었는지 직접 판단할 수 있다.
* 업데이트 필요 여부 판단이 단순하다.

따라서 기본 업데이트 판단 기준은 아래와 같다.

* 현재 설치된 `s2qt.exe`의 SHA-256 계산
* 배포 대상 `s2qt.exe`의 SHA-256과 비교
* 다르면 업데이트 필요
* 같으면 업데이트 불필요

### 3.3 버전 정보의 역할

버전은 완전히 없애지 않는다.

버전의 역할은 다음과 같다.

* 사용자에게 현재 앱 버전 표시
* 로그에 현재 빌드 식별 정보 기록
* 릴리스 관리 및 장애 대응 시 참고

즉:

* **업데이트 판단 = 해시값 비교**
* **표시/운영 관리 = 버전 정보**

## 4. 설치형 업데이트 전략

### 4.1 왜 설치형이 필요한가

실행 중인 `s2qt.exe`는 자기 자신을 바로 교체하기 어렵다.

Windows 환경에서는 실행 중인 EXE 파일이 잠기므로, 앱 내부에서 즉시 덮어쓰는 방식은 불안정하다.

따라서 현재 단계에서는 **설치형(Inno Setup) 업데이트 전략**이 가장 현실적이다.

### 4.2 현재 권장 전략

현재 권장 전략은 다음과 같다.

1. 앱이 업데이트 필요 여부를 판단한다.
2. 업데이트 필요 시 사용자에게 알린다.
3. 사용자가 확인하면 앱을 종료한다.
4. 설치 프로그램(Inno Setup)을 통해 `s2qt.exe`를 교체한다.
5. 사용자 데이터는 유지한다.
6. 완료 후 앱을 다시 실행한다.

### 4.3 현재 단계의 결론

현재 S2QT는 다음 전략이 적절하다.

* 평소 업데이트는 설치형 업데이트를 기본으로 한다.
* 실제 업데이트 판단은 해시값 비교로 한다.
* 실제 교체는 설치 프로그램이 수행한다.
* 사용자 데이터는 보존한다.

## 5. Git 기반 update 구조

### 5.1 Git 사용 가능 여부

전용 서버가 없더라도 Git을 update 기준 저장소로 사용할 수 있다.

즉, Git 저장소 또는 Git Release를 다음 용도로 사용할 수 있다.

* 최신 설치본 배포
* 최신 `s2qt.exe` 배포
* 해시값 파일 배포
* 업데이트 메타 정보(JSON) 배포

### 5.2 권장 Git 배포 구성

예시:

* `s2qt-setup-1.0.3.exe`
* `s2qt-1.0.3.exe`
* `sha256.txt`
* `update.json`

예시 `update.json`:

```json
{
  "version": "1.0.3",
  "sha256": "<SHA256_OF_S2QT_EXE>",
  "installer_url": "<URL>",
  "exe_url": "<URL>"
}
```

### 5.3 앱이 update 정보를 확인하는 방식

앱은 추후 아래 흐름으로 구현할 수 있다.

1. `update.json` 다운로드
2. 현재 `s2qt.exe` 해시 계산
3. `update.json`의 해시와 비교
4. 다르면 업데이트 필요 알림
5. 사용자가 원하면 설치본 다운로드/실행 안내

## 6. 유지/보존 정책

### 6.1 업데이트 시 유지해야 하는 것

다음은 업데이트 시 보존해야 한다.

* `var/conf/*`
* `var/db/*`
* `var/log/*`

### 6.2 업데이트 시 덮어쓰지 않는 것

다음은 설치형 업데이트 시 함부로 덮어쓰지 않는다.

* 사용자 설정 파일
* 보안 관련 설정 파일
* 사용자 이력/DB
* 로그 파일

### 6.3 런타임 복구 대상 유지

다음은 설치형 업데이트에서 핵심 교체 대상이 아니며, 앱 런타임 정책으로 유지한다.

* `ffmpeg.exe`
* `ffprobe.exe`
* `yt-dlp.exe`
* `ggml-tiny.bin`

## 7. 구현 절차

아래 절차는 차후 이 문서를 기준으로 구현을 진행하기 위한 순서이다.

### 단계 1. 기준 문서 확정

목적:

* update 전략을 고정한다.
* 해시값 비교 방식을 기준으로 삼는다.
* 설치형 업데이트 전략을 기본으로 확정한다.

완료 기준:

* 본 문서 기준 확정

### 단계 2. 앱 버전/릴리스 정보 정리

목적:

* 앱 내부에 버전 문자열 또는 빌드 식별 정보를 둘지 결정
* 표시용 버전과 해시 기준을 분리

검토 항목:

* `AppVersion` 상수 도입 여부
* 로그 시작 시 버전 기록 여부
* 설정 화면 또는 About 표시 여부

완료 기준:

* 버전 정보 보조 정책 확정

### 단계 3. 해시값 계산 기능 설계

목적:

* 현재 설치된 `s2qt.exe`의 SHA-256 계산 기능 정의

구현 항목:

* `s2qt.exe` 경로 확보
* SHA-256 계산 함수 작성
* 결과 문자열 반환 형식 정의

완료 기준:

* 현재 exe 해시 계산 가능

### 단계 4. update 메타 파일 구조 설계

목적:

* Git Release 또는 저장소에 둘 `update.json` 구조 확정

구현 항목:

* JSON 필드 정의
* `version`
* `sha256`
* `installer_url`
* `exe_url`
* 필요 시 `release_notes`

완료 기준:

* update.json 샘플 확정

### 단계 5. 업데이트 필요 여부 판단 로직 설계

목적:

* 로컬 exe 해시와 원격 해시 비교 절차 정의

구현 항목:

* 원격 `update.json` 조회
* 로컬 SHA-256 계산
* 비교 로직
* 동일 / 다름 판정

완료 기준:

* 업데이트 필요 여부 판단 가능

### 단계 6. 사용자 알림 정책 설계

목적:

* 업데이트가 필요할 때 언제, 어떻게 알릴지 정의

후보 시점:

* 앱 시작 시
* Step3 종료 시
* 설정 화면의 수동 확인 버튼

완료 기준:

* 업데이트 알림 UX 확정

### 단계 7. 설치형 업데이트 절차 설계

목적:

* Inno Setup 설치본을 이용한 실제 교체 절차 정의

구현 항목:

* 설치본 실행 조건
* 앱 종료 유도 방식
* 설치 후 재실행 여부

완료 기준:

* 설치형 업데이트 시나리오 확정

### 단계 8. Inno Setup overwrite / preserve 정책 확정

목적:

* 설치본이 어떤 파일을 교체하고 어떤 파일을 유지할지 정의

정책:

* 교체: `s2qt.exe` 등 앱 본체
* 유지: `var/conf`, `var/db`, `var/log`

완료 기준:

* 설치 시 파일별 처리 정책 확정

### 단계 9. update 기능 구현

목적:

* 실제 update 확인 기능 구현

구현 범위(1차):

* 수동 확인 방식
* 해시 비교
* 사용자 알림
* 설치본 실행 안내

완료 기준:

* 업데이트 필요 여부를 앱 내에서 확인 가능

### 단계 10. update 테스트

목적:

* 실제 업데이트 판단 및 설치 절차 검증

테스트 항목:

* 해시 동일 시 업데이트 없음
* 해시 상이 시 업데이트 감지
* 설치본 실행
* 기존 설정/DB/로그 유지
* 교체 후 앱 정상 실행

완료 기준:

* update 시나리오 통과

## 9. 현재 시점의 최종 결론

현재 S2QT의 update 전략은 아래처럼 정리한다.

1. 기본 update 기준은 `s2qt.exe`이다.
2. 업데이트 필요 여부 판단은 SHA-256 해시값 비교로 한다.
3. 실제 교체는 실행 중인 앱이 직접 하지 않고, 앱 종료 후 설치 프로그램(Inno Setup)이 수행한다.
4. `ffmpeg`, `ffprobe`, `yt-dlp`, 모델은 앱 런타임 정책으로 복구/다운로드한다.
5. `var/conf`, `var/db`, `var/log`는 사용자 데이터로 보고 업데이트 시 보존한다.
6. Git Release 또는 저장소를 update 메타와 설치본 배포 지점으로 활용할 수 있다.
