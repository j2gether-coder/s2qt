# S2QT 정리 문서

## 1. 서비스 방향성

S2QT는 설교 원문이나 음성 자체를 저장하는 플랫폼이 아니라,  
설교를 개인의 묵상과 삶의 적용이 가능한 QT 형태로 전환하는 서비스이다.

핵심 방향은 다음과 같다.

- 설교의 원본 보관보다 최종 QT 결과물 중심
- 인쇄물 중심보다 읽기 좋은 VIEW/eBook 중심
- 무료는 로컬/클라이언트 중심
- 구독형은 서버 저장 및 멀티디바이스 열람 중심
- 데이터 최소화를 유지하되 검색 가능 구조 확보
- 특정 종교 전용이 아닌 확장 가능한 구조 고려

---

## 2. 핵심 처리 흐름

S2QT의 기본 처리 단계는 다음과 같다.

1. 저품질 전사
2. 구조정리
3. QT 작성

저품질 전사라도 설교의 핵심 문맥이 살아 있으면  
QT 초안 생성에는 충분히 활용 가능하다고 본다.

즉, 목적은 완벽한 전사가 아니라  
QT를 만들 수 있을 정도의 의미 보존이다.

---

## 3. 사용자 유형별 처리 구조

### 3.1 무료 사용자 (LLM API 사용자)
무료 사용자는 자신의 LLM API를 사용하는 방식이다.

처리 구조:
- client: 전사, 구조정리, QT 작성
- server: 최소 저장, 검색, 정책 관리

결과물:
- 로컬 사용 중심
- 무료 템플릿 PDF 가능
- 무료 템플릿 PDF에는 워터마크 포함

### 3.2 유료 템플릿 사용자
유료 템플릿 사용자는 생성 엔진이 아니라 출력 권한이 유료인 구조이다.

처리 구조:
- client: 전사, 구조정리, QT 작성
- server: 저장, 권한 확인

결과물:
- 유료 템플릿 PDF 가능
- 워터마크 없음

### 3.3 구독형 사용자
구독형은 플랫폼이 제공하는 LLM을 이용하는 SaaS 구조이다.

처리 구조:
- client: 전사
- server: 구조정리, QT 작성, 저장, 검색, 권한 관리

결과물:
- 서버 저장
- PC/모바일 어디서나 열람 가능
- 멀티디바이스 VIEW 제공

## 4. 입력 정책

입력 소스는 아래와 같이 구분한다.

### 4.1 YouTube
YouTube는 자동 메타정보 활용이 가능하다.

활용 가능한 값:
- youtube id
- 제목
- 채널명 또는 설교자 추정
- 업로드 일자

원칙:
- 내부 식별은 youtube id 우선
- 자막이 있으면 자막 재사용 우선
- 전사가 필요할 때만 추가 처리

### 4.2 음성 파일
음성 파일은 고유 외부 식별자가 없다.

원칙:
- client에서 전사 수행
- 서버에는 원본 음성 저장하지 않음
- 검색을 위해 제목, 설교자, 본문 참조값 정도만 관리

### 4.3 설교문 텍스트
설교문은 텍스트 자체를 입력받는 형태이다.

원칙:
- 원문 파일 자체는 장기 보관하지 않음
- 필요한 핵심 메타와 최종 QT만 저장

---

## 5. 검색 정책

핵심 검색은 아래 3가지로 단순화한다.

- 본문
- 설교자
- 설교 제목

주제 태그는 사용자 입력 항목으로 두지 않는다.  
필요 시 내부 분석용 보조 정보로만 활용할 수 있다.

---

## 6. 저장 정책

### 6.1 저장하지 않는 것
다음 항목은 서버에 장기 저장하지 않는다.

- 원본 음성 파일
- 원본 설교문 파일
- 전사 결과 텍스트
- 구조정리 중간 결과
- 임시 PDF

즉, 원본과 중간 처리물은 저장 대상이 아니다.

### 6.2 저장하는 것
서버에는 최종 결과 중심으로 최소 정보만 저장한다.

저장 권장 항목:
- 설교 제목
- 설교자
- 본문 reference
- 언급 성구 reference 목록
- QT Markdown
- source_type
- generation_mode
- created_at

즉 서버는 설교 원문 저장소가 아니라  
QT 결과와 최소 메타정보를 저장하는 경량 데이터 플랫폼이 된다.

---

## 7. 본문 및 언급 성구 저장 방식

데이터 최소화를 위해 본문과 언급 성구는  
원문 텍스트를 저장하지 않고 reference만 저장한다.

예시:
- 본문 reference: `John 3:16`
- 언급 성구 reference 목록: `["Romans 8:28", "James 2:17"]`

원칙:
- 본문 text 저장 X
- 언급 성구 text 저장 X
- reference 저장 O

---

## 8. 다양한 종교 확장 가능성

초기에는 기독교 중심으로 시작할 수 있으나,  
데이터 구조는 종교 확장을 고려한 범용 구조로 설계하는 것이 바람직하다.

저장 구조는 특정 종교 텍스트 형식에 묶이지 않도록  
다음과 같은 범용 참조 구조를 고려한다.

- tradition
- work_id
- ref_start_1
- ref_start_2
- ref_end_1
- ref_end_2

즉 저장은 종교별 약어가 아니라  
정규화된 참조 구조를 우선하고,  
표시는 종교별 형식으로 렌더링한다.

---

## 9. QT 형식의 범용성

QT는 기독교 용어이지만 구조적으로 보면  
다양한 종교의 말씀/경전 기반 성찰과 삶의 적용 형식으로 확장 가능하다.

공통 구조 예시:
- 본문/말씀 참조
- 핵심 의미
- 성찰 질문
- 삶의 적용
- 마무리 문장

다만 명칭과 표현은 종교별로 조정이 필요하다.

---

## 10. 찬송/음악 요소 정책

찬송가/CCM 추천은 기독교에 한해서만 제공한다.

이유:
- 기독교에서는 찬송 추천이 자연스러움
- 다른 종교의 낭송/독경 등은 A4 1장 또는 간결한 VIEW 구조를 해칠 수 있음
- 결과물의 분량과 가독성을 유지하기 위함

원칙:
- 기독교: 찬송가/CCM 추천 가능
- 그 외 종교: 기본 구조에서 제외

---

## 11. 결과물 포맷 정책

### 11.1 PDF의 위치
PDF는 주력 열람 포맷이 아니라 출력용/템플릿 상품으로 본다.

정책:
- 무료 템플릿 PDF 가능
- 유료 템플릿 PDF 가능
- PDF는 출력 상품의 성격

### 11.2 VIEW / eBook의 위치
실제 서비스의 주력 경험은 PDF보다  
읽기 좋은 VIEW/eBook 스타일이어야 한다.

이유:
- 모바일/태블릿 친화적
- 줄간격, 글자크기, 여백 최적화 가능
- 인쇄보다 열람과 적용 중심 서비스에 적합

원칙:
- 기본 열람은 VIEW/eBook
- 다운로드/출력은 PDF

즉:
- 읽는 것은 VIEW
- 내려받는 것은 PDF

## 12. 무료 / 유료 / 구독 정책

### 12.1 무료
- 월 4~5회 사용 가능
- 1일 QT 중심
- 무료 템플릿 PDF 가능
- 무료 PDF에는 워터마크 포함
- 기본적으로 로컬/클라이언트 사용 중심

### 12.2 유료 템플릿
- 유료 템플릿 1장 PDF: 500원
- 워터마크 없음
- 생성 엔진이 아니라 출력 권한 상품

### 12.3 구독형
- 플랫폼 제공 LLM 사용
- 월 4~5회 신규 생성
- 재사용/열람은 무제한
- PC/모바일 어디서나 열람 가능
- 30일 체험 후 전환 구조 고려

구독형의 핵심 차별점:
- 언제 어디서나 열람 가능
- 멀티디바이스 지원
- 서버 저장 기반 VIEW 제공

## 13. 책자/전자책 방향

### 13.1 인쇄물보다 전자책 우선
S2QT는 인쇄 서비스보다  
전자 묵상집/전자책 중심이 더 적합하다.

이유:
- 소량 인쇄 운영 부담 큼
- 재고/배송/제본 문제 없음
- 읽기 중심 서비스와 잘 맞음

### 13.2 연간 묵상집
52주 기준:
- 표지
- 목차
- 머리글
- 주별 QT

를 포함하면 약 60페이지 안쪽으로 볼 수 있다.

이 경우 종이책보다  
연간 전자 묵상집(eBook) 상품이 더 적합하다.

## 15. 서버 사양 및 역할

초기 서버 가정:
- Intel Quad-Core CPU
- RAM 4GB
- SSD 120GB
- 네트워크 5Mbps
- 월 50,000원

이 서버는 AI 서버가 아니라  
가벼운 웹/API/저장/검색 서버로 봐야 한다.

전제:
- 전사 = client
- 1장 PDF/VIEW = client
- 책자형 = 별도 작업 PC 가능
- server = 저장, 검색, 정책, 구독형 LLM 처리

즉 서버는 QT 데이터 플랫폼 + 구독형 생성 서버 역할에 집중한다.

## 16. 손익 구조 요약

### 비용 전제
- 서버: 50,000원/월
- 도메인(.kr): 20,000원/년
- SSL: 무료 Let’s Encrypt
- 구독형: 9,900원/월
- 연간: 99,000원/년
- 구독형 사용량: 월 4~5회 신규 생성
- 전사/PDF는 client 처리
- 재사용 시 LLM 미사용

### LLM 변동비 추정
가정:
- 구조정리 + QT 작성 1회당 약 22.5원
- 월 5회 신규 생성 시 약 113원/인

### 고정비
- 서버 50,000원/월
- 도메인 월 환산 1,667원
- SSL 0원
- 합계 약 51,667원/월

### 결제 수수료 가정 포함
가정:
- 수수료 3.3% ~ 5.0%

### 손익분기점
월 9,900원 / 연 99,000원 구조에서  
현재 가정상 손익분기점은 대략 유료 구독자 6~7명 수준으로 추정된다.

### 해석
구독자 10명 수준에서 남는 금액은  
진짜 순이익이라기보다 최소 운영비를 제외한 1차 운영 잉여로 보는 것이 적절하다.

즉:
- 사업 모델 성립 여부는 확인 가능
- 그러나 개발/운영 인건비까지 포함한 진짜 순이익은 아님

## 17. 최종 정리

### 핵심 구조
- 무료: client 중심 생성, 로컬 사용, 무료 PDF
- 유료 템플릿: client 생성, 프리미엄 PDF
- 구독형: server 생성, 서버 저장, 멀티디바이스 VIEW

### 저장 원칙
- 원본 불필요
- 전사 불필요
- 구조정리 중간 결과 불필요
- 최종 QT와 최소 메타정보만 저장

### 결과물 원칙
- 주력은 VIEW/eBook
- PDF는 보조 출력 상품

### 서비스 방향
- 설교 저장 플랫폼이 아니라
- 말씀을 삶의 적용 가능한 QT/묵상 결과물로 전환하는 플랫폼
