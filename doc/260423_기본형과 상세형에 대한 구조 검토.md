# S2QT 상세형(A4 가로) 검토 정리

## 1. 배경

현재 S2QT는 Step1 → Step2 → Step3 흐름으로 동작한다.

* **Step1**: LLM 프롬프트를 통해 JSON 초안을 생성
* **Step2**: `temp.json`을 불러와 사용자 검토/편집 후 다시 저장하고, `temp.html` 미리보기를 생성
* **Step3**: `temp.html` 기반으로 PDF/PNG 산출물을 생성

즉, Step3는 독립적으로 새로운 데이터를 만드는 구조가 아니라, **Step2까지 확보된 데이터를 레이아웃에 태워 출력하는 구조**이다.  

---

## 2. 출력 형식 구분

출력 형식은 아래 두 가지로 구분한다.

* **기본형(A4 세로)**
* **상세형(A4 가로)**

### 기본형(A4 세로)

* 현재 표준 산출물 형식
* 제목, 본문 성구, 찬송, 관련 성구, 말씀의 창, 오늘의 메시지, 깊은 묵상과 적용, 오늘의 기도 중심
* 현재 Step2 미리보기 및 최종 PDF/PNG와 가장 가까운 구조

### 상세형(A4 가로)

* A4 가로 기반 2단 구조
* 기본형보다 정보량이 많은 활용형 문서
* 본문 성구, 관련 성구, 찬송가 1절 등 보조 정보를 더 적극적으로 활용하는 방향 검토

---

## 3. 중요한 전제

상세형(A4 가로)은 단순히 레이아웃만 바꾸는 작업이 아니다.
**추가 정보가 있어야 의미 있는 상세형이 된다.**

현재 확보된 기본 데이터는 다음과 같다.

* `title`
* `bibleText`
* `hymn`
* `support_scriptures`
* summary / message / reflection / prayer

이 구조는 `QTStep2Data` 기준으로 이미 존재한다. 즉, **본문 성구 / 관련 성구 / 찬송 제목 정도는 바로 활용 가능**하다. 

하지만 아래 데이터는 현재 기본 구조에 없다.

* **찬송가 1절**
* 상세형용 추가 설명 데이터
* 성경 본문 전문 데이터

따라서 상세형을 제대로 구성하려면, **Step2에서 사용자 입력으로 추가 정보를 확보하는 방향**이 가장 현실적이다.

---

## 4. 데이터베이스 구축은 1차 범위에서 제외

성경 데이터베이스나 찬송가 데이터베이스를 구축하면 자동화 수준은 높아질 수 있다.
그러나 현재 단계에서는 다음 이유로 범위가 지나치게 커진다.

* 별도 데이터 저장소 설계 필요
* 검색/조회 로직 필요
* 유지관리 범위 증가
* 상세형 옵션 추가 작업의 범위를 넘어서게 됨

따라서 1차 범위에서는 **DB 구축 없이, Step2 사용자 입력 기반으로 상세형 데이터를 확보**하는 방식으로 간다.

---

## 5. 상세형(A4 가로) 1차 방향

## 5.1 기본 원칙

상세형은 1차에서 아래 원칙으로 접근한다.

1. **기존 데이터는 그대로 활용**

   * 제목
   * 본문 성구
   * 관련 성구
   * 찬송
   * 말씀의 창
   * 오늘의 메시지
   * 깊은 묵상과 적용
   * 오늘의 기도

2. **부족한 데이터는 Step2에서 사용자 입력으로 보완**

   * 찬송가 1절
   * 필요 시 관련 성구 상세 입력

3. **Step3에서는 상세형용으로만 해당 추가 정보를 사용**

   * 기본형은 현재 구조 유지

---

## 6. Step2 추가 정보 방향

상세형에 필요한 추가 정보는 Step2에서 생성/입력하도록 한다.

### 6.1 관련 성구 입력

현재 `support_scriptures`는 배열 구조이며, 이미 Step2에서 편집 가능하다.
상세형용 추가 입력에서는 이 배열을 활용할 수 있다.

권장 방향:

* Step1 또는 기존 저장 결과의 `support_scriptures` 개수에 따라
* Step2에서 관련 입력 칸을 자동 생성
* 각 칸에 기존 값 자동 세팅
* 필요 시 추가 / 삭제 가능하도록 확장 검토

### 6.2 찬송가 1절

찬송가 1절은 현재 기본 데이터에 없으므로, Step2에서 별도 입력 받는다.

권장 내부 필드명:

* `hymn_verse_1`

권장 화면 라벨:

* `찬송가 1절`

권장 입력 방식:

* textarea 1개

---

## 7. 상세형 활성화 기준

상세형(A4 가로)은 **항상 사용 가능**하게 두는 것이 아니라,
**필요한 추가 정보가 준비된 경우에만 Step3에서 선택 가능**하도록 해야 한다.

이 기준은 반드시 필요하다.
현재 Step3는 요청 payload를 만들어 그대로 서비스에 전달하는 구조이고, 서비스도 요청값 기준으로 산출물을 생성하기 때문에, 상세형 데이터 준비 여부를 프런트와 백엔드에서 함께 검증해야 한다. `bindQTStep3.js`는 현재 `makeHtml / makePdf / makePng / dpi` 수준의 payload를 만들고, `QTStep3Request`도 그 정도만 정의하고 있다.  

### 7.1 기본형(A4 세로)

* 항상 활성화
* 현재 데이터만으로 생성 가능

### 7.2 상세형(A4 가로)

* Step2의 추가 정보가 모두 준비된 경우에만 활성화

---

## 8. 상세형 준비 완료 판정 기준

1차 기준은 아래처럼 단순하게 가는 것이 좋다.

### 필수

* `hymn_verse_1` 이 비어 있지 않을 것

### 조건부

* 관련 성구 상세 입력칸이 생성된 경우, 생성된 칸이 모두 입력 완료일 것
* 관련 성구가 0개라서 칸이 생성되지 않은 경우에는 자동 통과

즉, 아래처럼 해석한다.

### 준비 완료

* 찬송가 1절 입력 완료
* 생성된 관련 성구 추가 입력칸 모두 완료

### 준비 미완료

* 찬송가 1절 비어 있음
* 관련 성구 추가 입력칸 중 하나 이상 비어 있음

---

## 9. Step3 표시 원칙

상세형은 **숨기기보다 비활성화 + 안내문 표시**가 더 적절하다.

### 권장 방식

* 기본형(A4 세로): 활성
* 상세형(A4 가로): 조건 충족 전까지 비활성화
* 안내문 표시:

  * `Step2에서 찬송가 1절과 상세형 추가 정보를 입력하면 선택할 수 있습니다.`

### 이유

* 사용자가 기능 존재를 알 수 있음
* 왜 선택할 수 없는지 이해 가능
* Step2에서 무엇을 더 입력해야 하는지 바로 알 수 있음

---

## 10. 검증 위치

상세형 활성화 기준은 **프런트와 백엔드 양쪽 모두 검증**해야 한다.

### 10.1 프런트

대상:

* `bindQTStep3.js`

역할:

* 상세형 준비 여부 판정
* 상세형 선택 UI disabled 처리
* 안내 문구 표시

### 10.2 백엔드

대상:

* `qtstep3_service.go`

역할:

* 상세형 요청이 들어왔는데 필요한 추가 정보가 부족하면 오류 반환
* 예:

  * `상세형(A4 가로) 생성에 필요한 추가 정보가 부족합니다.`

---

## 11. 1차 구현 범위

상세형(A4 가로)의 1차 구현 범위는 아래로 한정한다.

### 포함

* Step2에서 상세형용 추가 정보 입력
* Step3에서 기본형 / 상세형 선택
* 상세형 준비 여부에 따른 비활성화 처리
* 상세형 전용 A4 가로 레이아웃 산출

### 제외

* 성경 DB 구축
* 찬송가 DB 구축
* 찬송가 1절 자동 조회
* 성경 본문 전문 자동 조회
* Step2 상세형 미리보기까지의 즉시 확장

---

## 12. 현재 구조 기준 수정 축

### 우선 수정 대상

* `QTStep2Data`

  * 상세형용 추가 필드 필요
* `qtStep2.js`

  * 추가 정보 입력 UI
* `bindQTStep2.js`

  * 입력값 수집 / 복원
* `qtstep2_service.go`

  * 추가 정보 저장 / 복원
* `qtStep3.js`
* `bindQTStep3.js`
* `qtstep3_service.go`

### 후속 검토 대상

* `pdf_service.go`
* `png_service.go`
* 상세형용 전용 HTML/CSS 구조

현재 구조상 Step2는 `temp.json` 저장과 `temp.html` 생성의 중심이고, Step3는 `temp.html` 기반 산출물 생성 구조이므로, 상세형 추가는 결국 **Step2 데이터 확보 + Step3 옵션 처리**의 조합으로 봐야 한다.  

---

## 13. 결론

이번 논의로 정리된 핵심은 다음과 같다.

1. **기본형(A4 세로)** 은 현재 표준 산출물로 유지한다.
2. **상세형(A4 가로)** 은 별도 옵션으로 추가한다.
3. 상세형은 단순 레이아웃 변경이 아니라, **추가 정보가 필요한 확장형 모드**로 본다.
4. DB 구축은 1차 범위에서 제외한다.
5. 부족한 데이터는 Step2에서 사용자 입력으로 보완한다.
6. 상세형은 Step2 추가 정보가 준비된 경우에만 Step3에서 활성화한다.
7. 상세형이 준비되지 않은 경우에는 숨기지 말고 **비활성화 + 안내문 표시**로 처리한다.
