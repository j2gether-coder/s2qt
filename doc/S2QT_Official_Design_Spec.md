# 1. 개요

## 1.1 시스템 개요

S2QT는 동영상, 오디오 또는 URL 기반 입력으로부터 전사 결과를 생성하고, 이를 구조화된 요약 문서로 변환하여 Markdown, HTML, PDF 형태로 출력하는 데스크톱 중심 애플리케이션이다.  
본 시스템은 로컬 전사 엔진과 LLM 기반 구조화 처리를 결합하여 사용자의 입력을 재가공 가능한 문서 자산으로 변환하는 것을 목표로 한다.

## 1.2 구축 목적

본 시스템의 구축 목적은 다음과 같다.

- 입력 미디어로부터 자동 전사 결과를 생성한다.
- 전사 결과를 구조화된 요약 문서로 변환한다.
- 결과물을 재사용 가능한 형태로 저장하고 출력한다.
- 민감한 중간 산출물의 저장을 최소화하고 정리 가능한 운영 구조를 제공한다.
- 반복 사용이 가능한 설정, 이력, 재처리 체계를 제공한다.

## 1.3 적용 범위

### 포함 범위

- YouTube URL 입력
- 로컬 미디어 파일 입력
- 오디오 추출 및 포맷 변환
- 로컬 전사 처리
- LLM 기반 구조화 처리
- Markdown / HTML / PDF 출력
- 작업 이력 관리
- 설정 관리
- 임시 파일 정리

### 제외 범위

- 다중 사용자 권한 관리
- 클라우드 협업 기능
- 외부 CMS 직접 연동
- 대규모 분산 처리
- 포렌식 관점의 완전 무흔적 보장

# 2. 시스템 목표 및 설계 원칙

## 2.1 시스템 목표

S2QT는 다음 목표를 기준으로 설계한다.

1. 입력부터 출력까지의 전체 파이프라인을 단순하고 일관되게 유지한다.
2. 출력 품질을 자유문이 아닌 구조화된 데이터로 통제한다.
3. 실제 데스크톱 환경에서 운영 가능한 수준의 보안성을 확보한다.
4. 예외 발생 시 재시도 및 복구가 가능한 구조를 제공한다.
5. 1차 버전에서는 확장성보다 안정성과 구현 가능성을 우선 확보한다.

## 2.2 설계 원칙

### 기능 원칙
- 단순한 단계형 파이프라인을 우선 적용한다.
- 각 단계의 입력과 출력은 명확히 분리한다.
- 외부 엔진은 표준화된 방식으로 호출한다.

### 데이터 원칙
- 원본 및 중간 산출물은 최소 저장을 원칙으로 한다.
- 최종 결과물과 재처리에 필요한 메타데이터는 보존한다.
- 출력은 구조화 JSON을 기반으로 생성한다.

### 보안 원칙
- 민감정보의 평문 저장을 최소화한다.
- 로그에는 키와 민감 원문을 남기지 않는다.
- 임시 산출물은 정리 가능하도록 관리한다.

### 운영 원칙
- 작업 상태는 단계별로 추적 가능해야 한다.
- 실패 시 마지막 성공 단계 이후부터 재시작 가능해야 한다.
- 1차 버전은 기능 확장보다 안정적 완료율 확보를 우선한다.

# 3. 기능 요구사항

## 3.1 입력 기능

### FR-01 URL 입력
- 사용자는 YouTube URL을 입력할 수 있어야 한다.
- 시스템은 입력된 URL의 유효성을 검증해야 한다.
- 시스템은 URL 기반 다운로드 작업을 생성해야 한다.

### FR-02 파일 입력
- 사용자는 로컬 오디오 또는 동영상 파일을 선택할 수 있어야 한다.
- 시스템은 지원 가능한 파일 형식을 식별해야 한다.
- 지원하지 않는 형식은 사용자에게 안내해야 한다.

## 3.2 미디어 처리 기능

### FR-03 미디어 다운로드
- URL 입력 시 yt-dlp를 이용해 미디어를 수집할 수 있어야 한다.
- 다운로드 실패 시 재시도 또는 실패 상태 저장이 가능해야 한다.

### FR-04 오디오 추출
- 시스템은 ffmpeg를 이용해 입력 미디어에서 오디오를 추출할 수 있어야 한다.
- 시스템은 전사 엔진 입력에 적합한 포맷으로 변환해야 한다.

## 3.3 전사 기능

### FR-05 로컬 전사
- 시스템은 whisper.cpp를 이용하여 로컬 전사를 수행해야 한다.
- 전사 결과는 원문, 구간별 결과, timestamp 정보를 포함할 수 있어야 한다.
- 전사 실패 시 원인을 구분하여 기록해야 한다.

## 3.4 구조화 기능

### FR-06 LLM 구조화
- 시스템은 전사 결과를 LLM에 전달하여 구조화된 문서를 생성해야 한다.
- 구조화 결과는 JSON 형식을 기본으로 해야 한다.
- 응답 결과는 검증 가능한 스키마를 따라야 한다.

### FR-07 재시도 및 fallback
- JSON 파싱 실패 또는 응답 오류 시 재시도를 수행할 수 있어야 한다.
- 설정에 따라 fallback 모델을 사용할 수 있어야 한다.

## 3.5 출력 기능

### FR-08 Markdown 출력
- 시스템은 구조화된 결과를 Markdown 형식으로 저장할 수 있어야 한다.

### FR-09 HTML 출력
- 시스템은 Markdown 또는 JSON 결과를 HTML로 렌더링할 수 있어야 한다.

### FR-10 PDF 출력
- 시스템은 HTML 기반으로 PDF를 생성할 수 있어야 한다.
- PDF는 기본적으로 A4 1페이지를 목표로 하되, 의미 보존이 어려울 경우 2페이지까지 허용한다.

## 3.6 저장 및 관리 기능

### FR-11 작업 이력 관리
- 시스템은 작업 단위별 이력을 저장해야 한다.
- 각 작업은 고유한 job_id로 식별되어야 한다.

### FR-12 상태 관리
- 시스템은 단계별 상태를 저장해야 한다.
- 작업 중단 후 재개 시 마지막 성공 단계부터 재처리할 수 있어야 한다.

### FR-13 설정 관리
- 시스템은 사용자 설정을 저장하고 재사용할 수 있어야 한다.

### FR-14 임시파일 정리
- 시스템은 작업 종료 후 TEMP 폴더 내 임시파일을 정리할 수 있어야 한다.
- 사용자가 보관 옵션을 선택한 경우 정리 대상에서 제외할 수 있어야 한다.

# 4. 비기능 요구사항

## 4.1 성능 요구사항

### NFR-01 처리 응답성
- 일반적인 단일 작업 기준으로 단계별 진행 상태를 사용자에게 표시해야 한다.
- 장시간 처리 시 진행 중 상태가 식별 가능해야 한다.

### NFR-02 출력 안정성
- JSON 파싱 및 PDF 생성 실패율을 최소화해야 한다.
- 렌더링 실패 시 대체 출력(예: Markdown, HTML)은 유지되어야 한다.

## 4.2 품질 요구사항

### NFR-03 구조 준수율
- LLM 출력은 정의된 JSON 구조를 높은 일관성으로 준수해야 한다.

### NFR-04 재현성
- 동일 입력과 동일 설정에서 유사한 수준의 결과를 생성할 수 있어야 한다.

## 4.3 보안 요구사항

### NFR-05 민감정보 보호
- API 키는 평문 파일 또는 DB에 직접 저장하지 않아야 한다.
- 로그에는 API 키와 민감 원문을 기록하지 않아야 한다.

### NFR-06 임시 데이터 최소화
- 민감한 중간 산출물은 임시 저장 후 정리 가능한 구조여야 한다.
- 완전한 포렌식 무흔적 보장은 범위에 포함하지 않는다.

## 4.4 운영 요구사항

### NFR-07 장애 복구성
- 단계별 상태와 오류 코드를 저장하여 장애 후 분석이 가능해야 한다.
- 재처리 시 마지막 성공 단계 이후부터 재개할 수 있어야 한다.

### NFR-08 유지보수성
- 프롬프트, 출력 템플릿, 설정 항목은 코드 변경 없이 교체 또는 수정 가능해야 한다.

### NFR-09 확장성
- 향후 다중 입력, 다국어, 템플릿 확장, 복수 LLM 지원이 가능하도록 모듈형 구조를 유지해야 한다.

# 5. 시스템 아키텍처

## 5.1 논리 아키텍처

시스템은 다음 7단계의 논리 흐름으로 구성한다.

1. 입력 수집
2. 미디어 전처리
3. 전사 처리
4. LLM 구조화
5. 결과 검증
6. 문서 렌더링
7. 저장 및 정리

## 5.2 구성 요소

### 5.2.1 UI 계층
- Vue + Vite 기반 화면
- 작업 입력, 상태 확인, 결과 미리보기, 설정 관리 제공

### 5.2.2 데스크톱 셸
- Wails 기반 데스크톱 애플리케이션 패키징
- UI와 Go 백엔드 간 연결 담당

### 5.2.3 애플리케이션 제어 계층
- Go 기반 메인 제어 로직
- 작업 생성, 외부 엔진 호출, 상태 저장, 결과 생성 수행

### 5.2.4 외부 처리 엔진
- yt-dlp: URL 기반 다운로드
- ffmpeg: 오디오 추출 및 포맷 변환
- whisper.cpp: 로컬 전사 엔진
- LLM Provider: 구조화 요약 생성

### 5.2.5 저장 계층
- SQLite: 설정, 이력, 상태, 메타데이터 저장
- File Storage: TEMP, OUTPUT, LOG, PROMPTS, CONFIG 관리

## 5.3 아키텍처 원칙

- 전사 및 구조화는 분리된 단계로 처리한다.
- 외부 도구는 독립 실패 가능성을 갖는 별도 단계로 관리한다.
- 출력 생성 전 JSON 검증을 필수 단계로 둔다.
- 렌더링 실패 시 상위 단계 산출물은 보존한다.

# 6. 기술 구성

## 6.1 기술 스택

- Frontend: Vue, Vite
- Desktop Shell: Wails
- Main Backend: Go
- Security Helper: Rust (선택 적용)
- Downloader: yt-dlp
- Media Processor: ffmpeg
- STT Engine: whisper.cpp
- Local Database: SQLite
- PDF Renderer: HTML 기반 PDF 렌더러

## 6.2 기술 적용 원칙

### Go
- 전체 파이프라인 orchestration 담당
- 파일, 상태, 외부 명령 실행에 적합

### Rust
- 키 저장, 민감정보 보조 처리 등 제한적 보안 모듈에 적용
- 전체 시스템 복잡도 증가를 방지하기 위해 범위를 제한한다.

### SQLite
- 단일 사용자 기반 로컬 애플리케이션에 적합
- 이력 및 설정 관리 중심으로 사용한다.

# 7. 디렉터리 및 파일 구조

## 7.1 표준 디렉터리 구조

```text
S2QT/
├─ BIN/
├─ VAR/
│  ├─ DB/
│  ├─ LOG/
│  ├─ STATE/
│  ├─ CACHE/
│  └─ OUTPUT/
├─ TEMP/
├─ CONFIG/
└─ PROMPTS/
```

## 7.2 디렉터리별 용도

- `BIN/`: 애플리케이션 실행 파일 및 외부 엔진
- `VAR/DB/`: SQLite DB 저장소
- `VAR/LOG/`: 운영 로그
- `VAR/STATE/`: 작업 상태 및 체크포인트
- `VAR/CACHE/`: 재사용 가능한 비민감 캐시
- `VAR/OUTPUT/`: 최종 결과물 저장소
- `TEMP/`: 작업 중 임시 산출물
- `CONFIG/`: 애플리케이션 설정
- `PROMPTS/`: LLM 프롬프트 템플릿

## 7.3 운영 원칙

- TEMP는 작업 단위별 하위 디렉터리를 구성한다.
- 작업 종료 후 TEMP는 정리 대상이 된다.
- LOG에는 민감정보를 남기지 않는다.
- OUTPUT은 최종 산출물만 저장한다.

# 8. 데이터 설계

## 8.1 주요 데이터 엔터티

### app_config
- 시스템 및 사용자 설정 저장

### job_history
- 작업 이력 저장
- 입력 유형, 시작/종료 시각, 상태, 결과 위치 등 관리

### job_state
- 단계별 처리 상태 저장
- 장애 복구 및 재개에 사용

### output_metadata
- 최종 산출물 정보 저장
- Markdown/HTML/PDF 파일 위치 및 생성 시각 관리

### prompt_version
- 프롬프트 버전 및 변경 이력 관리

### llm_provider_config
- LLM 공급자 설정 및 사용 정책 저장

## 8.2 작업 상태 코드 예시

- RECEIVED
- DOWNLOADED
- AUDIO_EXTRACTED
- TRANSCRIBED
- STRUCTURED
- VALIDATED
- RENDERED
- COMPLETED
- FAILED

## 8.3 JSON 출력 스키마

```json
{
  "title": "string",
  "scripture": "string",
  "summary": "string",
  "messages": ["string"],
  "reflection": ["string"],
  "prayer": "string"
}
```

## 8.4 JSON 검증 규칙

- `title`: 필수, 1~120자
- `scripture`: 선택, 최대 200자
- `summary`: 필수, 최대 800자
- `messages`: 필수, 1~3개
- `reflection`: 선택, 1~3개
- `prayer`: 선택, 최대 500자
- 빈 문자열 금지
- 배열 항목 중복 금지
- 필수 필드 누락 시 재요청
- 파싱 실패 시 재시도 또는 fallback 수행

# 9. 처리 흐름 설계

## 9.1 단계별 처리 흐름

### 1단계: 입력 접수
- URL 또는 파일 입력 수신
- job_id 생성
- 초기 상태 저장

### 2단계: 미디어 확보
- URL이면 다운로드 수행
- 파일이면 직접 참조
- 미디어 존재 여부 확인

### 3단계: 오디오 추출
- ffmpeg 기반 오디오 추출
- whisper.cpp 입력 포맷으로 변환

### 4단계: 전사 처리
- 로컬 전사 수행
- transcript 생성
- segment 및 timestamp 산출 가능

### 5단계: 구조화 처리
- transcript를 LLM에 전달
- JSON 구조 결과 생성

### 6단계: 검증
- JSON schema validation
- 길이 및 필수 필드 검증

# 10. 보안 설계

## 10.1 보안 목표

본 시스템의 보안 목표는 민감정보 노출 최소화와 운영 가능한 수준의 데이터 보호이다.  
일반 데스크톱 환경 특성상 완전한 무흔적 또는 포렌식 차단은 본 문서의 목표 범위에 포함하지 않는다.

## 10.2 키 관리 정책

- API 키는 평문 파일에 저장하지 않는다.
- API 키는 SQLite에 평문 저장하지 않는다.
- OS 보안 저장소 사용을 우선 검토한다.
  - Windows: DPAPI 또는 Credential Manager
  - macOS: Keychain
  - Linux: Secret Service 계열 검토
- OS 저장소 미지원 시, 별도 암호화 저장 정책을 수립한다.

## 10.3 로그 통제 정책

로그 허용 항목:
- job_id
- 단계명
- 시작/종료 시각
- 성공/실패 여부
- 오류 코드
- 비민감 경로

로그 금지 항목:
- API 키
- 전체 전사 원문
- 민감한 사용자 본문
- 필요 이상으로 상세한 입력 원문

## 10.4 임시파일 정책

- 전사 원문, 오디오 추출 결과, 중간 JSON은 TEMP에 저장한다.
- 작업 종료 후 자동 정리를 기본으로 한다.
- 사용자가 보관 설정을 켠 경우 예외적으로 유지할 수 있다.
- 삭제는 일반 삭제를 기본으로 하며, 보안 삭제는 운영체제와 파일시스템 제약을 고려해 별도 옵션으로 검토한다.

# 16. 출력 레이아웃 제어 정책 (A4 단일 페이지 전략)

## 16.1 목적

본 정책은 PDF 출력 시 A4 1페이지 내에서 최대한 의미를 유지하면서 콘텐츠를 표현하기 위한 레이아웃 제어 기준을 정의한다.

---

## 16.2 기본 원칙

출력 레이아웃은 다음 우선순위를 따른다.

1. 의미 보존
2. 가독성 유지
3. 페이지 수 최소화

A4 1페이지를 목표로 하되, 의미 훼손이 발생하는 경우 2페이지까지 허용한다.

---

## 16.3 레이아웃 제어 단계

출력 길이 제어는 다음 단계로 수행한다.

### 1단계: LLM 출력 제한

- 프롬프트 수준에서 길이 제한 적용
- 문장 수 및 항목 수 제한

### 2단계: JSON 검증

- 필드 길이 및 개수 제한 검증
- 초과 시 재요청 수행

### 3단계: CSS 압축

- 폰트 크기 및 line-height 조정
- 여백 최소화
- 페이지 overflow 제한 적용

### 4단계: 렌더링 후 높이 체크

- HTML 렌더링 후 실제 높이 측정
- A4 기준 초과 여부 판단

### 5단계: 다단 레이아웃 적용 (Fallback)

- 1단 레이아웃 초과 시 2단 레이아웃 적용
- column 기반 레이아웃으로 공간 재배치

### 6단계: 재생성

- 다단 적용 후에도 초과 시 LLM 재요청 수행
- 축약 버전 생성

---

## 16.4 다단 레이아웃 정책

### 적용 조건

- 기본은 단일 컬럼
- 초과 시에만 다단 적용

### 설정 기준

- 컬럼 수: 2단
- 컬럼 간격: 16px ~ 20px
- 컬럼 정렬: balance

### 금지 사항

- 3단 이상 레이아웃 금지 (가독성 저하)

---

## 16.5 레이아웃 안정성 규칙

다단 적용 시 다음 요소는 분리되지 않아야 한다.

- blockquote 영역
- 메시지 제목(h3, h4)
- 묵상 및 기도 블록

이를 위해 다음 CSS 규칙을 적용한다.

- break-inside: avoid
- break-after: avoid

---

## 16.6 렌더링 기준값

- A4 높이 기준: 약 1122px (96dpi 기준)
- 기준 초과 시 레이아웃 조정 수행

---

## 16.7 실패 처리 정책

다음 순서로 fallback을 수행한다.

1. 1단 레이아웃
2. 2단 레이아웃
3. LLM 축약 재요청
4. 최종적으로 2페이지 허용

---

## 16.8 설계 결론

A4 1페이지 제어는 단일 기술로 해결되지 않으며 다음 요소의 조합으로 달성한다.

- LLM 출력 제어
- CSS 레이아웃 조정
- 렌더링 결과 기반 동적 판단
- 다단 레이아웃 fallback
- 재생성 전략

본 정책은 출력 품질과 시스템 안정성의 균형을 유지하기 위한 기준으로 적용한다.
