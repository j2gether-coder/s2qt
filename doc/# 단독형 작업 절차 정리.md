# 단독형 작업 절차 정리

## 1. 개요

단독형은 로컬 PC 환경에서 동작하는 개인용 작업 구조를 전제로 한다.  
핵심 원칙은 다음과 같다.

- 로그인 후에만 로컬 데이터와 API Key를 복호화할 수 있도록 한다.
- 원본 음성, 설교문, 전사 결과, 구조정리 중간결과는 저장하지 않는다.
- 최종 QT 결과와 최소 메타정보만 로컬 DB(SQLite)에 저장한다.
- API Key는 별도 암호화 파일로 저장하고, 세션 중에는 메모리에서만 사용한다.
- 자동 잠금, 로그아웃, 앱 종료 시에는 메모리 내 민감 정보를 폐기한다.

---

## 2. 기본 보안 원칙

### 2.1 로그인 방식
- 로그인 ID는 이메일 주소의 **id 부분**을 사용한다.
- 최초 등록 시에는 **전체 이메일 주소**를 입력받고 내부적으로 함께 보관한다.
- 비밀번호는 매번 입력받는다.
- PIN은 사용하지 않는다.

### 2.2 비밀번호 처리
- 비밀번호는 평문 저장하지 않는다.
- 비밀번호 해시만 저장한다.
- 해시 알고리즘은 **Argon2id 우선**으로 한다.
- 필요 시 대안으로 PBKDF2를 고려할 수 있다.

### 2.3 로컬 DB 보호
- SQLite는 암호화된 상태로 운영한다.
- 로그인 후에만 복호화 가능한 구조로 설계한다.
- 최종 QT와 최소 메타정보만 저장한다.

### 2.4 API Key 보호
- API Key는 SQLite에 저장하지 않는다.
- 별도 암호화 파일로 저장한다.
- 로그인 후에만 복호화하여 메모리에서 사용한다.
- 세션 종료 시 메모리에서 폐기한다.

---

## 3. 최초 실행 절차

앱을 처음 실행하면 아직 로컬 계정과 보안 메타데이터가 없으므로 최초 등록 절차를 수행한다.

### 3.1 사용자 등록 화면
입력 항목:
- 전체 이메일 주소
- 로그인 ID로 사용할 이메일 id 부분 확인
- 비밀번호
- 비밀번호 확인

### 3.2 내부 처리
최초 등록 시 아래 처리를 수행한다.

1. 전체 이메일 주소 저장
2. 로그인 ID 생성 또는 확인
3. 비밀번호 해시 생성
4. KDF용 salt 생성
5. 랜덤 데이터 암호화 키(DEK) 생성
6. 비밀번호 기반 키로 DEK 암호화
7. 보안 메타데이터 저장
8. 로컬 계정 생성

### 3.3 결과
- 단독형 보안 환경이 초기화된다.
- 이후 로그인 절차로 이동한다.

---

## 4. 로그인 절차

### 4.1 로그인 화면
입력 항목:
- 로그인 ID
- 비밀번호

정책:
- 로그인 ID는 저장 및 표시 가능
- 비밀번호는 매번 입력받는다

### 4.2 로그인 처리
로그인 시 아래 절차를 수행한다.

1. 로그인 ID 조회
2. 비밀번호 검증
3. 비밀번호 기반 키 유도
4. 래핑된 DEK 복호화
5. 로컬 DB 복호화 준비
6. 세션 시작

### 4.3 로그인 결과
- 로컬 DB 접근 가능
- API Key 사용 가능
- 메인 화면 진입

---

## 5. API Key 설정 절차

로그인 후 사용자는 API Key를 등록할 수 있다.

### 5.1 API Key 입력
입력 항목:
- 사용자 API Key

### 5.2 처리
1. 로그인 후 확보된 복호화 가능 상태 확인
2. API Key를 별도 암호화 파일로 저장
3. 디스크에는 암호문만 유지
4. 세션 중에는 메모리에만 적재

### 5.3 원칙
- 복호화된 평문 파일은 만들지 않는다
- UI에는 마스킹된 값만 보여준다
- 로그/에러에 출력하지 않는다

### 5.4 미등록 상태
API Key가 없으면:
- QT 생성 기능은 제한된다
- 기존 QT 열람/검색은 가능하다

---

## 6. 메인 화면 절차

로그인 후 메인 화면에서 사용자는 아래 기능을 선택할 수 있다.

- 새 QT 생성
- 기존 QT 열람
- 검색
- 무료 템플릿 출력
- 유료 템플릿 출력
- 설정
- 로그아웃

---

## 7. 새 QT 생성 절차

### 7.1 입력 방식 선택
사용자는 입력 소스를 선택한다.

- YouTube
- 음성 파일
- 설교문 텍스트

### 7.2 입력 데이터 준비

#### YouTube
- youtube id 확인
- 자막 또는 기존 텍스트가 있으면 우선 활용
- 필요한 메타정보 확인 및 보정

#### 음성 파일
- client에서 저품질 전사 수행
- 원본 음성 파일은 저장하지 않음

#### 설교문 텍스트
- 텍스트 직접 입력

---

## 8. 메타정보 입력 및 보정

QT 생성을 위해 최소한 아래 정보를 확인하거나 입력한다.

- 설교 제목
- 설교자
- 본문 reference
- 전통 구분(tradition)

원칙:
- 원본 자체는 저장하지 않는다
- 최종 QT 생성에 필요한 정보만 사용한다

---

## 9. 구조정리 절차

client에서 사용자 API Key를 이용해 구조정리를 수행한다.

### 9.1 입력
- 전사 결과 또는 설교문 텍스트
- 설교 제목
- 설교자
- 본문 reference
- tradition

### 9.2 출력
- 설교 핵심 흐름
- 핵심 메시지
- QT 생성용 구조화 결과

### 9.3 저장 정책
- 구조정리 결과는 세션 중 사용만 한다
- 로컬 DB에는 저장하지 않는다

---

## 10. QT 작성 절차

client에서 사용자 API Key를 이용해 QT를 생성한다.

### 10.1 입력
- 구조정리 결과
- 제목
- 설교자
- 본문 reference
- tradition
- 템플릿 옵션

### 10.2 출력
- 최종 QT Markdown
- 요약
- 본문 표시값
- 언급 성구 reference 목록
- 필요 시 찬송 추천(기독교만)

### 10.3 저장 정책
- 최종 QT만 저장 대상이다
- 구조정리 중간결과는 저장하지 않는다

---

## 11. QT 저장 절차

최종 결과만 SQLite에 저장한다.

### 11.1 저장 대상
- 설교 제목
- 설교자
- tradition
- source_type
- generation_mode
- primary_ref_text
- qt_markdown
- qt_summary
- has_hymn_recommend
- 본문 reference
- 언급 성구 reference
- 생성일시
- 수정일시

### 11.2 저장하지 않는 것
- 원본 음성 파일
- 원본 설교문
- 전사 결과
- 구조정리 결과
- 임시 PDF

### 11.3 결과
- 생성된 QT는 개인 QT 보관함에 누적 저장된다

---

## 12. 열람 절차

저장된 QT는 목록 또는 상세 화면에서 열람할 수 있다.

### 가능 기능
- QT VIEW 열람
- 제목/설교자/본문 확인
- 요약 확인
- 기존 문서 재확인
- 템플릿 출력

---

## 13. 검색 절차

검색 기준은 아래와 같이 단순화한다.

- 설교 제목
- 설교자
- 본문 reference

즉 단독형은 로컬 QT 라이브러리처럼 동작한다.

---

## 14. 출력 절차

### 14.1 무료 템플릿 출력
- client에서 PDF 생성
- 무료 템플릿 적용
- 워터마크 포함

### 14.2 유료 템플릿 출력
- 유료 템플릿 구매 여부 확인
- client에서 PDF 생성
- 워터마크 없음

### 14.3 출력 정책
- PDF는 보조 출력 수단으로 본다
- 주력 열람 경험은 VIEW 중심으로 간다
- PDF는 필요 시 사용자가 직접 보관한다

---

## 15. 자동 잠금 절차

일정 시간 무입력 상태가 지속되면 자동 잠금을 수행한다.

### 권장 기준
- 10분 또는 15분 무입력 잠금

### 처리
- 메모리 세션 잠금
- 복호화된 사용 상태 중지
- 다시 비밀번호 입력 요구

---

## 16. 로그아웃 절차

사용자가 로그아웃하면 아래를 수행한다.

1. 메모리 내 DEK 폐기
2. 메모리 내 API Key 폐기
3. 민감 데이터 참조 제거
4. 잠금 상태 전환

결과:
- 다시 사용하려면 비밀번호 재입력이 필요하다

---

## 17. 앱 종료 절차

앱 종료 시 아래를 수행한다.

1. 세션 종료
2. 메모리 내 API Key 폐기
3. 복호화 상태 해제
4. 디스크에는 암호화된 DB와 암호화 파일만 유지

---

## 18. 비밀번호 변경 절차

사용자는 비밀번호를 변경할 수 있다.

### 입력
- 현재 비밀번호
- 새 비밀번호
- 새 비밀번호 확인

### 처리
1. 현재 비밀번호 검증
2. 새 비밀번호 기반 키 생성
3. DEK 재래핑
4. password hash 갱신

---

## 19. API Key 변경/삭제 절차

사용자는 API Key를 변경하거나 삭제할 수 있다.

### 변경
- 기존 API Key 폐기
- 새 API Key 입력
- 새 암호화 파일 생성

### 삭제
- 암호화 파일 삭제
- 세션 메모리 내 값 폐기

---

## 20. 앱 초기화 절차

비밀번호 분실 또는 완전 초기화가 필요한 경우 앱 초기화를 수행한다.

### 처리
- 로컬 DB 삭제
- API Key 암호화 파일 삭제
- 설정 초기화
- 보안 메타 초기화

### 주의
- 초기화 시 기존 데이터는 복구되지 않을 수 있다

---

## 21. 전체 흐름 요약

### 최초 사용
1. 앱 실행
2. 전체 이메일 + 로그인 ID + 비밀번호 등록
3. 보안 메타 생성
4. 로그인
5. API Key 등록
6. 작업 시작

### 일반 사용
1. 앱 실행
2. 로그인 ID 확인
3. 비밀번호 입력
4. 로컬 DB 및 보안 요소 복호화
5. 새 QT 생성 또는 기존 QT 열람
6. 필요 시 출력
7. 자동 잠금 / 로그아웃 / 종료 시 메모리 폐기

---

## 22. 최종 원칙 요약

- 로그인 후에만 로컬 데이터를 복호화한다
- 비밀번호는 매번 입력받는다
- API Key는 별도 암호화 파일로 관리한다
- 세션 중에는 메모리에서만 API Key를 사용한다
- 원본/전사/구조정리 결과는 저장하지 않는다
- 최종 QT와 최소 메타정보만 저장한다
- 단독형은 개인 QT 보관함 구조로 운영한다
