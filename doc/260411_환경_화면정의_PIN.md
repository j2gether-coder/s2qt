# S2QT 환경설정 화면 정의서(초안)

## 1. 개요

본 문서는 S2QT의 환경설정 화면에 대한 초안이다.  
환경설정은 작업 Step 흐름이 아니라 운영 관리 성격이므로 **탭형 구조**로 설계한다.

화면 목적은 다음과 같다.

- AI(LLM) 사용을 위한 기본값 관리
- 이메일/SMTP 발송 설정 관리
- 교회/브랜드 정보 관리
- 라이선스/사용 안내 제공
- 민감정보 수정 시 PIN 인증 적용

---

## 3. 탭별 항목표

# 3.1 AI 설정 탭

## 3.1.1 목적
- LLM 연동을 위한 필수 설정값 관리
- API KEY 및 모델 정보 관리
- 자동/수동 모드 관리

---

## 3.1.2 화면 항목표

| 항목 ID | 항목명 | 형식 | 필수 | 수정 가능 | PIN 필요 | 저장 key | 비고 |

| aiProvider | LLM 공급자 | select | Y | Y | N | llm.provider | openai, anthropic, manual 등 |
| aiEnabled | AI 사용 여부 | checkbox | N | Y | N | llm.enabled | true / false |

| aiModel | 모델명 | text/select | N | Y | N | llm.model | 예: gpt-5 |

| aiApiKeyEditBtn | API KEY 수정 | button | - | Y | Y | - | 클릭 시 PIN 확인 |
| aiApiKeyInput | 새 API KEY 입력 | password | N | Y | Y | llm.api_key | 수정 모드에서만 표시 |
| aiTestBtn | 연결 테스트 | button | - | Y | Y | - | 필요 시 PIN 확인 |

---

## 3.1.3 기본 동작

### 조회 시
- 공급자 / 모델 / 모드 / 사용 여부 표시
- API KEY는 아래처럼 마스킹 표시
  - `등록됨`
  - 또는 `********`

### 수정 시
- `API KEY 수정` 버튼 클릭
- 세션 Unlock 상태 확인
- 잠금 상태면 PIN 모달 호출
- 성공 시에만 `새 API KEY 입력` 활성화

### 저장 시
- 입력된 새 API KEY를 암호화 후 DB 저장
- 저장 완료 후 다시 마스킹 표시로 전환
- 평문 재표시 금지

## 4. 탭별 저장 정책

## 4.1 저장 방식
초기 버전에서는 **탭별 저장 버튼**을 권장한다.

예:
- AI 설정 저장
- 이메일 설정 저장
- 교회/브랜드 저장
- 안내 저장

이유:
- 수정 범위가 명확함
- 오류 위치 파악이 쉬움
- 민감정보 저장 흐름을 분리하기 쉬움

---

## 4.2 저장 성공 메시지
- toast 사용
- 예: `AI 설정이 저장되었습니다.`

## 4.3 저장 오류 메시지
- 탭 내부 inline message 사용
- 예: `SMTP 설정 저장 중 오류가 발생했습니다.`

## 5. PIN 모달 화면 초안

# 5.1 목적
PIN 모달은 앱 실행 로그인창이 아니라  
민감 기능 접근 전 잠금 해제용 보안 모달이다.

---

# 5.2 호출 사유(reason)

예시:
- `llm_use`
- `edit_api_key`
- `edit_smtp_password`
- `smtp_test`

reason에 따라 안내 문구만 바뀐다.

---

# 5.3 화면 구성

## 상단
- 제목: `보안 확인`

## 본문 안내 문구
상황에 따라 다음 중 하나 표시

- `LLM API 사용을 위해 PIN을 입력해 주세요.`
- `API KEY 수정을 위해 PIN을 입력해 주세요.`
- `SMTP 비밀번호 수정을 위해 PIN을 입력해 주세요.`
- `테스트 발송을 위해 PIN을 입력해 주세요.`

## PIN 표시부
- 마스킹 표시
- 예: `● ● ● ● ● ●`
- 입력 자릿수 시각화

## 가상키보드
예시 배열:

```text
1 2 3
4 5 6
7 8 9
전체삭제 0 지우기
````

또는

```text id="b0bzg9"
1 2 3
4 5 6
7 8 9
지우기 0 확인
```

## 하단 버튼

* 확인
* 취소

---

# 5.4 동작 규칙

### 입력 규칙

* 숫자만 허용
* 4~6자리
* 권장 6자리
* 최대 길이 도달 시 자동 대기 또는 확인 버튼 활성화

### 성공 시

* 모달 닫기
* session unlocked = true
* 원래 요청 작업 계속 진행

### 실패 시

* 모달 내부 inline error 표시
* 예: `PIN이 올바르지 않습니다.`

---

pinModalState = {

## 6. 세션 Unlock 정책

## 6.1 기본 정책

* 한 번 PIN 인증 성공 시 현재 앱 실행 세션 동안 Unlock 유지
* 같은 세션에서는 연령별 Step1에서 다시 PIN을 묻지 않음

예:

* 장년 QT Step1 인증 성공
* 이후 청년/중고등부/어린이 Step1은 재인증 없음

---

## 6.2 세션 해제 시점

* 앱 종료 시
* 앱 재실행 시

초기 버전에서는 이 기준만 적용한다.

---

## 6.3 세션 상태 예시

```js id="c2gphv"
securityState = {
  unlocked: false,
  unlockedAt: '',
  unlockScope: 'session'
}
```

## 8. 권장 구현 순서

## 8.1 1차

* app_settings 테이블 반영
* security 관련 key 저장/조회
* PIN 등록/검증 로직 구현

## 8.2 2차

* 공용 PIN 모달 UI 구현
* 가상키보드 입력 구현
* 세션 unlock 상태 구현

## 8.3 3차

* Step1 자동 LLM 사용 전 PIN 확인 연결
* AI 설정의 API KEY 수정 전 PIN 확인 연결
* SMTP 비밀번호 수정 전 PIN 확인 연결

## 8.4 4차

* 테스트 연결 / 테스트 메일 발송 연결
* 실패 횟수 제한
* 자동 잠금 옵션 검토

---

## 9. 최종 정리

### 환경설정 화면 구조

* 탭형 UI
* AI 설정 / 이메일 설정 / 교회/브랜드 / 안내

### PIN 모달

* 가상키보드 기반 숫자 PIN 입력
* 민감 기능 접근 전 공용 모달 호출
* 화면 내 inline error 사용

### 세션 정책

* 한 번 인증하면 앱 종료 전까지 유지
* 같은 세션에서는 연령별 Step1에서 재인증 없음

### DB 정책

* app_settings key-value 구조
* PIN은 해시 저장
* API KEY / SMTP 비밀번호는 암호화 저장
* 저장값은 다시 평문 표시하지 않음
