# S2QT 본문 텍스트(`bible_passage_text`) 추가 최소 수정 계획

## 1. 목표

현재의 `bible_text`는 `살전 4:13~18` 같은 **참조값**이다.
이번 수정에서는 여기에 더해 **실제 본문 문장**을 별도 필드로 관리한다.

### 분리 기준

* `bible_text`

  * 본문 성구 참조
  * 예: `데살로니가전서 4:13~18`

* `bible_passage_text`

  * 실제 본문 문장 텍스트
  * Step1에서 LLM이 초안 생성
  * Step2에서 사용자가 검토/수정
  * Step3 산출물에서 표시

---

## 2. 핵심 설계 원칙

### 2.1 저장과 출력은 분리

* 저장은 **전체 본문 텍스트**를 보존
* 출력은 길이에 따라 **축약 표시** 허용

### 2.2 축약 기준

* **5절 이하**: 전체 본문 텍스트 표시
* **5절 초과**: `첫절 ... 마지막절` 형식으로 표시

### 2.3 Step2 확인 절차 필수

LLM이 본문 텍스트 초안을 생성하더라도,
최종 산출물에 들어가는 데이터이므로 Step2에서 확인 절차를 둔다.

---

## 3. Step1 최소 수정 계획

현재 Step1 프롬프트에는 입력 메타데이터로 `본문 성구: {{bible_text}}`가 들어가고, JSON 스키마에도 `metadata.bible_text`가 있다. 여기에 `bible_passage_text`를 추가하면 된다.  

### 3.1 필수 수정 파일

* `prompt_qt_json.go`
* `prompt_qt_json.md`

### 3.2 필수 수정 내용

#### A. JSON 스키마에 필드 추가

`metadata`에 아래 필드 추가

```json id="1gt9qj"
"bible_passage_text": ""
```

#### B. 프롬프트 규칙 추가

LLM에게 아래 기준을 명확히 지시

* 입력된 `bible_text`에 해당하는 **실제 본문 문장 초안**을 생성할 것
* 결과는 `metadata.bible_passage_text`에 넣을 것
* **절 번호가 드러나도록** 작성할 것
* 가능하면 **한 절당 한 줄** 구조로 작성할 것
* Step2에서 사용자가 검토/수정할 수 있도록 **설명 없이 본문 텍스트만** 넣을 것

### 3.3 중요한 보강 포인트

축약을 하려면 Step3 쪽에서 “첫 절 / 마지막 절”을 식별할 수 있어야 하므로,
`bible_passage_text`는 단순 문단보다 아래 형태가 더 좋다.

예:

```text id="7m8qmw"
13절 형제들아 자는 자들에 관하여는 ...
14절 우리가 예수께서 죽으셨다가 다시 살아나심을 믿을진대 ...
15절 우리가 주의 말씀으로 너희에게 이것을 말하노니 ...
16절 주께서 호령과 천사장의 소리와 하나님의 나팔로 친히 하늘로부터 강림하시리니 ...
17절 그 후에 우리 살아 남은 자들도 ...
18절 그러므로 이러한 말로 서로 위로하라
```

이 형식이면 Step2/Step3에서 줄 단위로 절 수를 세고,
`첫절 ... 마지막절` 축약이 쉬워집니다.

### 3.4 선택 보강

* `llm_service.go`에서 `bible_passage_text`가 없으면 빈 문자열로 보정
* manual result 저장 경로에서도 누락 시 기본값 처리

## 4. Step2 최소 수정 계획

현재 `QTStep2Data`에는 `BibleText`만 있고, 실제 본문 문장 필드는 없다. 따라서 Step2가 이번 수정의 중심이다. 

### 4.1 필수 수정 파일

* `types.go` 계열 (`QTStep2Data`)
* `qtStep2.js`
* `bindQTStep2.js`
* `qtstep2_service.go`

### 4.2 필수 수정 내용

#### A. 데이터 구조 추가

`QTStep2Data`에 새 필드 추가

```go id="cw4bbh"
BiblePassageText string `json:"bible_passage_text"`
```

#### B. Step2 UI 추가

기존 “관련 성구” 근처 또는 “말씀의 창” 위쪽에 아래 2개를 둠

* `본문 성구` input
* `본문 텍스트` textarea

### 4.3 Step2 안내 메시지

본문 텍스트 아래에 안내문을 넣는 것이 좋다.

#### 안내문 예시 1

`AI가 생성한 본문 텍스트 초안입니다. 성경 본문과 대조하여 확인 후 저장해 주세요.`

#### 안내문 예시 2

`본문이 5절을 초과하면 Step3 산출물에서는 첫 절과 마지막 절 중심으로 축약 표시될 수 있습니다.`

이 메시지는 필수에 가깝다.
사용자는 Step2에서 전체 본문을 보더라도, Step3 출력에서는 일부 축약될 수 있기 때문이다.

### 4.4 바인딩 수정

`bindQTStep2.js`에서 아래를 load/save/history payload에 추가

* `bible_passage_text`

즉:

* load 시 textarea에 세팅
* save 시 payload에 포함
* history 저장 JSON에도 포함

### 4.5 저장 서비스 수정

`qtstep2_service.go`의 `Load()` / `Save()`에서 metadata에 아래 추가

* `bible_passage_text`

즉 metadata 예시는 아래처럼 됨.

```json id="y8b9jx"
"metadata": {
  "title": "...",
  "bible_text": "데살로니가전서 4:13~18",
  "bible_passage_text": "13절 ...\n14절 ...\n15절 ...",
  "hymn": "...",
  "support_scriptures": [],
  "preacher": "...",
  "church_name": "...",
  "sermon_date": "...",
  "source_url": "..."
}
```

### 4.6 이미 반영된 부분

사용자께서 요청하신 “미리보기를 하지 않아도 저장 시 HTML 저장”은 서비스 구조상 이미 Save 경로에서 같이 처리되는 방향이므로, 이번에는 `bible_passage_text`가 그 HTML에도 반영되도록만 보면 됩니다.

## 5. Step3 최소 수정 계획

현재 Step3는 `temp.html`을 전제로 PDF/PNG를 생성한다. 즉, **직접 Step3 서비스 로직을 크게 바꾸기보다 Step2 HTML 생성부를 수정하는 것이 최소 수정**이다. `qtstep3_service.go`는 `temp.html` 존재를 확인하고 그걸 기반으로 PDF/PNG를 만든다. 

### 5.1 필수 수정 파일

* `qtstep2_service.go`
  (실제 HTML 생성부)
* 필요 시 `style_qt_html.css`
* 필요 시 `style_qt_pdf.css`

### 5.2 필수 수정 내용

#### A. 제목/상단 정보 레이아웃 정리

현재 제목 아래 `qt-subbox`에 본문 성구/찬송/관련 성구를 넣고 있는데, 이 구조를 조금 정리해서:

* 1줄: 제목
* 2줄: 본문 성구 / 찬송
* 3줄: 관련 성구
* 그 아래: **본문 텍스트**

로 가는 것이 좋다.

#### B. 본문 텍스트 표시 helper 추가

`buildQTStep2HTML()` 안에서 `bible_passage_text`를 출력할 때 helper를 둔다.

추천 함수 개념:

* `formatBiblePassageForOutput(bibleText, biblePassageText string) string`

동작:

* 절 수 5 이하 → 전체 표시
* 절 수 5 초과 → 첫 줄 + `...` + 마지막 줄

예:

```text id="zyhjk5"
13절 형제들아 자는 자들에 관하여는 ...
...
18절 그러므로 이러한 말로 서로 위로하라
```

### 5.3 절 수 판정 기준

가장 최소 수정으로는 `bible_passage_text`를 **줄바꿈 기준으로 절 수를 판단**하는 것이 좋다.

즉:

* 줄 수 5 이하 → 전체
* 줄 수 6 이상 → 첫 줄 / 마지막 줄만 사용

이 방식이면 별도 성경 파서 없이도 구현 가능합니다.

### 5.4 Step3 서비스 직접 수정 여부

`qtstep3_service.go`는 크게 안 건드려도 됩니다.
이유는 Step3가 직접 본문 텍스트를 조합하는 구조가 아니라, `temp.html`만 읽어 PDF/PNG를 만들기 때문입니다. 따라서 핵심 수정은 `qtstep2_service.go` 쪽 HTML 생성부입니다.
