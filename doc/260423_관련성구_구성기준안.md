# S2QT 관련 성구 기능 추가 기준서 (최소 수정 원칙)

## 1. 작업 목적

S2QT 프로젝트에 **관련 성구** 기능을 추가한다.

이번 기능의 의미는 다음과 같다.

* 화면 용어는 **관련 성구**로 통일
* 내부 필드명은 **`support_scriptures`**
* 의미는 **본문 성구 외, 말씀 이해를 돕는 관련 성구**
* 다중값 구조를 기본으로 하며 JSON에서는 **배열**로 관리
* Step1 → Step2 → temp.json / temp.html → Step3(PDF/PNG)까지 일관되게 이어져야 함
* 기존 S2QT 구조를 크게 흔들지 않고 **최소 수정 원칙**으로 반영

현재 구조상 Step1은 프롬프트 생성과 결과 저장, Step2는 `temp.json` 로드/저장과 `temp.html` 생성, Step3는 `temp.html` 기반 산출물 생성 흐름으로 연결되어 있다.    

---

## 2. 사전 원칙

1. 수정 전 관련 파일은 반드시 백업을 먼저 받는다.
2. 사용자 화면 용어는 **관련 성구**로 통일한다.
3. 내부 필드명은 **`support_scriptures`** 를 사용한다.
4. 값이 없으면 **빈 배열 `[]`** 또는 출력 생략으로 처리한다.
5. Step2 화면에서는 **줄바꿈 기반 textarea** 입력으로 처리한다.
6. 최종 출력 순서는 **본문 성구 → 찬송 → 관련 성구** 기준으로 검토한다.
7. 구조 변경보다 **기존 metadata / temp.html 흐름 재사용**을 우선한다.

---

## 3. 현재 구조 요약

## 3.1 Step1

Step1 프롬프트는 `LLMService.BuildPrompt()`와 `GenerateQTJSON()`에서 모두 `BuildQTPromptJSON(meta)`를 사용하고 있다. 즉 프롬프트 내용 수정은 실제로 **프롬프트 원문(하드코딩 + var/conf fallback)** 쪽에서 처리해야 한다. `bindQTStep1.js`는 프롬프트 생성과 수동 결과 저장(`SaveManualLLMResult`)을 호출하는 연결부이다.   

또한 현재 JSON 프롬프트 템플릿에는 제목, 본문 성구, 찬송, 설교자, 교회명, 설교일, 원본 URL, audience 규칙은 있으나 **관련 성구 항목은 아직 없다.**  

## 3.2 Step2

`qtstep2_service.go`는 `temp.json`을 `QTSectionDoc{ metadata, sections }` 구조로 읽고 쓴다. 현재 metadata에는 `title`, `bible_text`, `hymn`, `preacher`, `church_name`, `sermon_date`, `source_url`만 사용하고 있다. 아직 `support_scriptures`는 없다. 

같은 파일의 `buildQTStep2HTML()`는 상단 `qt-subbox`에 현재 **본문 성구**와 **찬송**만 출력한다. 관련 성구는 아직 출력되지 않는다. 

프런트에서는 `bindQTStep2.js`가 Step2 payload 생성, 로드, history 저장까지 담당하고 있으며, 현재 payload에는 summary/message/reflection/prayer만 있고 `support_scriptures`는 없다. history도 `qtResultJson: JSON.stringify(step2Payload)` 구조로 저장한다. 

## 3.3 Step3

`qtstep3_service.go`는 `temp.html`이 있어야 실행되고, PDF는 `temp.html`을 읽어 `pdf_service.go`로 넘기고, PNG도 `temp.html`을 읽어 생성한다. 따라서 Step2 HTML에 관련 성구가 들어가면 Step3 출력은 대부분 자동으로 따라간다.  

---

## 4. 데이터 구조 기준

## 4.1 저장 기준

`support_scriptures`는 JSON에서 항상 배열로 저장한다.

예시:

```json
{
  "metadata": {
    "title": "[QT] 복 있는 사람의 길",
    "bible_text": "시편 1:1-6",
    "hymn": "찬송가 28장",
    "support_scriptures": [
      "이사야 40:31",
      "로마서 8:28"
    ]
  }
}
```

이번 구조에서는 **sections가 아니라 metadata에 두는 것이 최소 수정**이다. 현재 Step2 저장/복원 중심이 metadata이기 때문이다. 

## 4.2 Step2 입력 기준

Step2 UI에서는 textarea 한 칸으로 처리한다.

* 라벨: **관련 성구**
* 안내 문구: `본문 성구 외, 말씀 이해를 돕는 관련 성구를 입력합니다.`
* 입력 방식: 한 줄에 하나씩

예시 입력:

이사야 40:31
로마서 8:28

저장 시에는 다음 배열로 변환한다.

```json
[
  "이사야 40:31",
  "로마서 8:28"
]
```

## 5. 반영 원칙

## 5.1 Step1 반영 원칙

Step1은 UI보다 프롬프트/결과 저장이 핵심이다.

### 필수

* JSON 프롬프트에 `support_scriptures` 추가
* 의미 설명 추가: 본문 성구 외, 말씀 이해를 돕는 관련 성구
* 반드시 문자열 배열로 반환하도록 명시
* 값이 없으면 `[]` 반환하도록 명시
* `SaveManualLLMResult` 저장부에서 `support_scriptures` 누락/문자열/단일값일 경우 배열 정규화

### 선택

* HTML 프롬프트 템플릿(`prompt_qt_html.md`)에도 `qt-subbox` 예시에 관련 성구 라인 반영
  현재 예시에는 본문과 찬송만 있다. 다만 실제 최종 HTML 생성은 Step2 서비스가 담당하므로 1차 필수는 아니다.  

## 5.2 Step2 반영 원칙

Step2가 이번 작업의 핵심 반영 지점이다.

### 필수

* `QTStep2Data`에 `support_scriptures []string` 추가
* `qtstep2_service.go`

  * `Load()`에서 metadata의 `support_scriptures` 복원
  * `Save()`에서 metadata의 `support_scriptures` 저장
  * `BuildHTML()` / `buildQTStep2HTML()`에서 상단 subbox에 관련 성구 출력
* `qtStep2.js`

  * 관련 성구 textarea 추가
* `bindQTStep2.js`

  * load 시 배열 → 줄바꿈 문자열 변환
  * save 시 줄바꿈 문자열 → 배열 변환
  * history payload에는 기존 `JSON.stringify(step2Payload)` 흐름을 그대로 활용

현재 bind 구조상 Step2 payload와 history 저장이 한 경로로 묶여 있으므로, `buildStep2Payload()`에만 `support_scriptures`를 추가하면 저장과 history 반영이 함께 따라간다.  

## 5.3 Step3 반영 원칙

Step3는 1차에서 직접 수정 우선순위가 낮다.

### 이유

* Step3는 `temp.html`을 읽어 PDF/PNG를 생성한다.
* 따라서 Step2 HTML의 `qt-subbox`에 관련 성구가 반영되면 Step3 산출물에 그대로 이어질 가능성이 높다. 

### 1차 판단

* `qtstep3_service.go`: 직접 수정 불필요
* `png_service.go`: 직접 수정 불필요
* `bindQTStep3.js`, `qtStep3.js`: 직접 수정 불필요

## 7. 구조상 주의사항

현재 `pdf_service.go`는 PDF 생성 중 `SaveHtmlAndMakeJSON()`를 호출하여 HTML에서 다시 `temp.json`을 만든다. 이때 사용하는 JSON 구조는 `QTJSONDoc{ title, subbox, sections }` 중심이며, Step2가 사용하는 `metadata + sections` 구조와 다르다.   

따라서 이번 1차 구현에서 관련 성구를 Step2 metadata에만 넣고 끝내면:

* HTML/PDF/PNG 출력 자체는 잘 될 가능성이 높지만
* 이후 PDF 생성 후 다시 `temp.json`을 기준으로 재복원하는 특수 흐름에서는 구조 차이로 손실 가능성이 있다

이 부분은 **명백한 구조 주의사항**이며, 이번 작업에서는 아래처럼 처리하는 것이 좋다.

### 이번 1차 처리

* 출력 반영 우선
* Step2 metadata + Step2 HTML 중심으로 구현
* PDF/PNG 출력 확인

### 추후 2차 처리

* `pdf_service.go`의 `SaveHtmlAndMakeJSON()` 구조 보강
* 필요 시 `support_scriptures`를 subbox 파싱 또는 별도 필드로 보존

## 8. 최종 반영 기준

## 8.1 화면 기준

* 라벨: **관련 성구**
* 안내 문구: `본문 성구 외, 말씀 이해를 돕는 관련 성구를 입력합니다.`

## 8.2 내부 기준

* 필드명: `support_scriptures`
* 타입: `[]string`
* 빈값: `[]`

## 8.3 출력 기준

* 상단 subbox 표시 순서:

  1. 본문 성구
  2. 찬송
  3. 관련 성구

## 8.4 출력 형태

* 여러 값은 줄바꿈 또는 `<br />` 기반으로 표시
* 값이 없으면 해당 줄 생략

현재 HTML/CSS 구조상 `qt-subbox`는 여러 줄 텍스트를 수용할 수 있으므로, 관련 성구 한 줄 추가는 CSS 대수정 없이도 수용 가능하다.
