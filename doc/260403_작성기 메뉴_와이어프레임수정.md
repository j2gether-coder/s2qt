좋습니다. 그 기준으로 다시 정리하겠습니다.

핵심 원칙은 아래로 확정합니다.

* **좌측 메뉴는 이동만 담당**
* **상태는 각 화면 내부에서만 표시**
* **모든 audience를 다 만들 필요 없음**
* **사용자는 필요한 audience만 선택해서 작업**
* **전사는 공통 1회, audience 작업은 선택 반복**

---

# QT 작성기 메뉴/와이어프레임 수정안

## 1. 전체 구조

화면은 크게 두 영역으로 구성합니다.

### 좌측

단순 메뉴 영역

### 우측

선택한 메뉴의 작업 화면

---

## 2. 좌측 메뉴 구조

좌측 메뉴는 상태 없이 아래처럼 단순하게 갑니다.

```text
전사
장년
청년
중고등부
어린이
```

### 설계 원칙

* 메뉴에는 완료/미완료 상태를 표시하지 않음
* 메뉴는 “이동”만 담당
* 사용자는 필요한 audience만 들어가서 작업
* 장년만 만들고 끝내도 문제없는 구조

---

## 3. 전체 레이아웃

```text
+--------------+-------------------------------------------+
| 좌측 메뉴     | 우측 작업 화면                              |
+--------------+-------------------------------------------+
| 전사          | 선택된 메뉴의 상세 작업 화면                 |
| 장년          |                                           |
| 청년          |                                           |
| 중고등부      |                                           |
| 어린이        |                                           |
+--------------+-------------------------------------------+
```

---

## 4. 전사 화면 와이어프레임

전사 메뉴를 누르면 공통 원문 준비 화면이 나옵니다.

```text
+-----------------------------------------------------------+
| 전사                                                      |
+-----------------------------------------------------------+
| 입력 원천:  ( ) 동영상   ( ) 오디오   ( ) 텍스트          |
|                                                           |
| [입력 영역]                                               |
| - 동영상: URL 입력                                        |
| - 오디오: 파일 선택                                       |
| - 텍스트: 직접 입력 / 파일 불러오기                       |
|                                                           |
| 공통 메타 입력                                             |
| 제목        [____________________________]                |
| 본문 성구   [____________________________]                |
| 찬송        [____________________________]                |
| 설교자      [____________________________]                |
| 교회명      [____________________________]                |
| 설교일      [____________________________]                |
|                                                           |
| 원문 확인                                                 |
| ┌────────────────────┬────────────────────┐              |
| | raw_text           | cleaned_text       |              |
| |                    |                    |              |
| └────────────────────┴────────────────────┘              |
|                                                           |
| [전사 실행] [원문 정리] [저장]                            |
|                                                           |
| 상태: SOURCE_SELECTED / TRANSCRIBED / CLEANED / SAVED    |
+-----------------------------------------------------------+
```

### 전사 화면 특징

* 공통 원문 준비는 여기서만 수행
* 입력 원천은 동영상/오디오/텍스트 모두 지원
* 저장 후 audience 화면에서 공통 원문을 참조

---

## 5. audience 화면 공통 구조

장년/청년/중고등부/어린이는 동일한 화면 구조를 사용하고, 내용만 audience별로 달라집니다.

```text
+-----------------------------------------------------------+
| 장년                                                      |
+-----------------------------------------------------------+
| 현재 상태: 아직 생성되지 않음                             |
|                                                           |
| [Step1] [Step2] [Step3] [Step4]                           |
+-----------------------------------------------------------+
| 현재 선택된 Step 작업 영역                                 |
+-----------------------------------------------------------+
```

### audience 화면 원칙

* 상태는 이 화면 안에서만 표시
* 아직 생성 안 된 audience면 안내 문구 표시
* 필요한 audience만 작업 가능

---

## 6. audience 화면 상단 상태 영역

예: 장년 메뉴 진입 시

### 아직 미생성 상태

```text
장년 QT 작업
공통 원문이 연결되어 있습니다.
아직 생성된 장년용 QT가 없습니다.
[Step1 생성 조건 확인부터 시작]
```

### 생성 후 상태

```text
장년 QT 작업
공통 원문 연결 완료
생성 조건 저장 완료 / AI 생성 완료 / 편집 저장 완료 / 산출물 일부 생성
```

### 설계 원칙

* 상태는 간단한 문장 형태
* 메뉴에 노출하지 않고 해당 화면에서만 보여줌
* 사용자가 지금 이 audience에서 무엇을 해야 하는지 안내

---

## 7. Step1 생성 조건 확인 화면

```text
+-----------------------------------------------------------+
| Step1. 생성 조건 확인                                     |
+-----------------------------------------------------------+
| audience         [장년]                                   |
| template_id      [qt_classic ▼]                           |
| 문체 옵션        [차분함 ▼]                               |
| 찬송 자동 추천   [사용 ▼]                                 |
|                                                           |
| 공통 원문 요약                                            |
| - 제목: 십자가의 죽으심                                   |
| - 본문 성구: 갈 2:20                                      |
| - source_id: sermon_001                                   |
| - 입력 원천: 텍스트                                       |
|                                                           |
| [생성 요청 저장] [기본값 초기화]                          |
+-----------------------------------------------------------+
```

### 목적

* 해당 audience 생성 조건만 확인
* 공통 원문은 읽기 전용으로 요약 표시

---

## 8. Step2 AI QT 생성 화면

```text
+-----------------------------------------------------------+
| Step2. AI QT 생성                                         |
+-----------------------------------------------------------+
| [AI QT 생성] [다시 생성] [이전 결과 불러오기]             |
|                                                           |
| 생성 상태                                                 |
| - prompt 조합 완료                                        |
| - LLM 요청 완료                                           |
| - JSON 검증 완료                                          |
|                                                           |
| 생성 결과 요약                                            |
| - audience: adult                                         |
| - template_id: qt_classic                                 |
| - sections: 4                                             |
| - message: 3                                              |
|                                                           |
| [JSON 보기]                                               |
+-----------------------------------------------------------+
```

### 목적

* 공통 원문 + audience 규칙으로 QT JSON 생성
* 결과 생성 여부를 이 화면에서만 확인

## 9. Step3 내용 검토/편집 화면

이 단계는 좌우 2단 구성이 가장 좋습니다.

```text
+-----------------------------------------------------------+
| Step3. 내용 검토/편집                                     |
+-----------------------------------------------------------+
| ┌────────────────────────┬──────────────────────────────┐ |
| │ 구조화 편집            │ HTML Preview                 │ |
| └────────────────────────┴──────────────────────────────┘ |
+-----------------------------------------------------------+
```

### 좌측 구조화 편집

```text
[기본정보]
제목      [____________________]
본문 성구 [____________________]
찬송      [____________________]

[말씀의 창]
요약문단
[________________________________________]

[오늘의 메시지]
메시지1 제목 [________________]
메시지1 본문 [________________]
메시지2 제목 [________________]
메시지2 본문 [________________]
메시지3 제목 [________________]
메시지3 본문 [________________]

[깊은 묵상과 적용]
적용1 [________________]
적용2 [________________]
적용3 [________________]

[오늘의 기도]
기도문 [________________________________]

[저장] [원본으로 되돌리기]
```

### 우측 HTML Preview

* 현재 JSON 기준 미리보기
* template 적용 결과 확인
* 1차는 새로고침 버튼 방식도 가능

### 핵심 원칙

* HTML 직접 편집은 허용하지 않음
* JSON 구조를 기준으로만 편집
* 구조는 고정, 내용만 수정

## 11. 사용자 흐름 관점에서 본 구조

### 사용 예시 1: 장년용만 만드는 사용자

1. 전사에서 공통 원문 준비
2. 장년 메뉴 이동
3. Step1~Step4 진행
4. 종료

### 사용 예시 2: 장년 + 청년 만드는 사용자

1. 전사에서 공통 원문 준비
2. 장년 Step1~4 진행
3. 청년 메뉴 이동
4. 청년 Step1~4 진행

### 사용 예시 3: 중고등부만 만드는 사용자

1. 전사에서 텍스트 입력
2. 중고등부 메뉴 이동
3. Step1~Step4 진행

즉 UI는 “전체 audience를 다 해야 한다”는 느낌을 주면 안 됩니다.

---

## 12. 최종 화면 설계 원칙

### 좌측 메뉴

* 이동만 담당
* 상태 표시 없음
* 단순하고 가볍게 유지

### 우측 전사 화면

* 공통 원문 준비 전용
* video/audio/text 지원

### 우측 audience 화면

* 해당 audience의 상태와 step만 표시
* 아직 미생성인 경우 그 audience 안에서만 안내
* Step1~4 반복 수행

---

## 13. 최종 요약 구조

```text
+--------------+-------------------------------------------+
| 전사          | 공통 원문 준비                              |
| 장년          | 장년용 Step1~4                              |
| 청년          | 청년용 Step1~4                              |
| 중고등부      | 중고등부용 Step1~4                          |
| 어린이        | 어린이용 Step1~4                            |
+--------------+-------------------------------------------+
```

이 구조가 지금까지 정리한 원칙과 사용성 측면을 가장 잘 반영합니다.

다음 단계는 이 구조를 바탕으로 **실제 컴포넌트 목록**을 정리하는 것입니다.
즉 어떤 입력박스, 버튼, 상태값, 저장데이터가 필요한지까지 내려가면 바로 구현 설계 수준이 됩니다.
