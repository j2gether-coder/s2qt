# S2QT 산출물 정책

## 1. 목적

S2QT의 산출물은 사용 목적이 서로 다르므로, HTML / PDF / PNG의 역할을 명확히 분리한다.  
이 정책의 핵심은 다음과 같다.

- HTML은 **검토/편집/미리보기용**
- PDF는 **정식 문서용**
- PNG는 **공유용 이미지용**
- 특히 PNG는 PDF를 열어 보기 어려운 사용자를 위한 산출물이므로, **안정적이고 일관된 결과**를 제공해야 한다.
- PDF와 PNG는 서로 복불복처럼 달라지면 안 되며, **같은 기준과 같은 규칙 아래 생성**되어야 한다.

---

## 2. 산출물별 역할

### 2.1 HTML
HTML은 사용자가 내용을 확인하고 수정하는 **작업용 산출물**이다.

역할:
- Step2 검토 및 편집 화면의 미리보기
- 저장 후 확인용 중간 산출물
- PDF/PNG 생성 전 기준 데이터 확인용

원칙:
- HTML은 최종 배포 산출물이 아니다.
- HTML은 사용성, 편집 편의성, 미리보기에 초점을 둔다.
- 최종 결과 기준은 HTML이 아니라 PDF이다.

---

### 2.2 PDF
PDF는 S2QT의 **정식 문서 산출물**이다.

역할:
- 최종 보관 문서
- 인쇄 기준 문서
- 공식 배포 기준 문서
- 레이아웃, footer, QR, safe area의 기준 문서

원칙:
- PDF를 가장 우선되는 기준 산출물로 본다.
- 페이지 크기(A4), 여백, footer, QR 위치, divider, 템플릿 배경 규칙은 PDF를 기준으로 확정한다.
- PDF가 정상적이고 안정적으로 생성되는 것이 가장 중요하다.

---

### 2.3 PNG
PNG는 S2QT의 **공유용 이미지 산출물**이다.

역할:
- 모바일/메신저/SNS 공유용
- PDF를 열어 보기 어려운 사용자를 위한 대체 산출물
- 빠르게 확인 가능한 시각형 산출물

원칙:
- PNG는 PDF의 보조 산출물이지만, 품질은 안정적이어야 한다.
- PNG는 PDF와 다른 문서처럼 보이면 안 된다.
- PNG는 PDF를 기반으로 동일한 내용과 동일한 레이아웃 원칙을 유지해야 한다.
- PNG는 “공유용이라 대충 맞으면 된다”가 아니라, **PDF 기준을 시각적으로 충실히 전달하는 이미지**여야 한다.

---

## 3. 기준 산출물 원칙

S2QT에서 레이아웃의 기준 산출물은 **PDF**이다.

즉 다음 항목은 PDF 기준으로 먼저 확정한다.

- 제목 위치
- 본문 성구 영역
- 성경본문(축약 포함) 영역
- 각 섹션 제목 스타일
- reflection / prayer box 스타일
- footer 문구 위치
- church/logo/brand 위치
- QR 크기 및 위치
- safe area
- template 적용 시 본문 placement

PNG는 위 기준을 최대한 그대로 따라야 한다.

---

## 4. 생성 우선순위

산출물 생성 흐름의 우선순위는 다음과 같다.

1. JSON 데이터 확정
2. HTML 생성
3. PDF 생성
4. PNG 생성

설명:
- JSON은 콘텐츠의 기준 데이터이다.
- HTML은 편집/검토용 중간 산출물이다.
- PDF는 최종 문서 기준이다.
- PNG는 PDF 기준을 공유용 이미지로 변환한 결과물이다.

---

## 5. 레이아웃 정책

### 5.1 공통 원칙
- 제목 / subbox / 성경본문 / 말씀의 창 / 오늘의 메시지 / 깊은 묵상과 적용 / 오늘의 기도는 동일한 콘텐츠 구조를 유지한다.
- PDF와 PNG는 같은 문서처럼 보여야 한다.
- 섹션 순서와 의미가 달라지면 안 된다.

### 5.2 Footer 정책
Footer는 문서의 하단 고정 영역으로 본다.

포함 가능 요소:
- divider
- footer text
- church name / brand / logo
- QR code

원칙:
- footer는 본문과 분리된 고정 영역이다.
- 본문은 footer 영역을 침범하면 안 된다.
- PDF와 PNG 모두 footer를 독립 영역으로 취급한다.

### 5.3 QR 정책
QR은 footer 자산이다.

원칙:
- QR 크기 기준은 PDF에서 먼저 정한다.
- PNG는 PDF의 QR 기준을 동일하게 따른다.
- QR는 본문을 침범하지 않아야 한다.
- QR 위치는 left-bottom / right-bottom 중 정책에 따라 고정한다.
- QR 배경, margin, quiet zone은 스캔 안정성을 해치지 않는 범위에서 조정한다.

### 5.4 성경본문 정책
성경본문은 사용성 및 A4 안정성을 위해 축약 가능하다.

원칙:
- 5절 이상인 경우 축약을 허용한다.
- 축약 시 "첫절 ... 마지막절" 구조를 사용한다.
- 축약 여부는 본문 영역 높이에 영향을 줄 수 있으나, footer safe area를 임의로 줄이는 근거로 바로 사용하지 않는다.
- 본문 축약은 콘텐츠 영역 정책이고, footer는 하단 자산 정책이다.

---

## 6. 템플릿 정책

템플릿은 배경/디자인 확장 요소이며, 콘텐츠보다 우선하지 않는다.

원칙:
- 본문 가독성이 최우선이다.
- 템플릿은 본문을 침범하거나 QR/footer를 가리지 않아야 한다.
- PDF와 PNG 모두 템플릿 적용 시 동일한 placement 규칙을 사용한다.
- PNG는 템플릿 적용 시에도 공유용 이미지로서 readability를 유지해야 한다.

---

## 7. 구현 정책

### 7.1 HTML
- Step2 저장 및 미리보기 지원
- 사용자가 확인/편집 가능한 중간 결과
- 최종 문서 기준은 아님

### 7.2 PDF
- 독립적인 PDF 전용 스타일 사용
- PDF 전용 runtime style 사용
- 문서 기준 산출물로 유지

### 7.3 PNG
- PNG 전용 스타일 사용
- PNG 전용 runtime style 사용
- PDF와 동일한 콘텐츠/레이아웃 원칙 유지
- PNG는 공유용 이미지이므로 특히 생성 결과가 안정적이어야 한다.

---

## 8. 품질 기준

### 8.1 HTML 품질 기준
- 저장 후 즉시 미리보기 가능
- Step2 편집 결과가 반영됨
- temp.html이 일관되게 생성됨

### 8.2 PDF 품질 기준
- A4 한 장 기준
- footer와 QR 위치 안정
- 본문 박스, divider, 섹션 스타일 정상 반영
- 정식 문서로 배포 가능

### 8.3 PNG 품질 기준
- PDF와 시각적으로 큰 차이가 없어야 함
- footer/QR가 잘리지 않아야 함
- 본문이 footer를 침범하지 않아야 함
- 모바일/메신저 공유 시 읽기 쉬워야 함
- 생성 결과가 환경에 따라 복불복이면 안 됨

---

## 9. 운영 원칙

- HTML은 작업용
- PDF는 기준 문서
- PNG는 공유용 이미지
- PDF와 PNG는 서로 다른 결과물이 아니라, **같은 문서를 다른 방식으로 제공한 것**이어야 한다.
- 향후 수정/리팩토링 시에도 이 역할 분담을 유지한다.

---

## 10. 최종 정책 요약

- JSON = 콘텐츠 기준
- HTML = 검토/편집/미리보기
- PDF = 정식 문서 기준
- PNG = PDF를 보기 어려운 사용자를 위한 안정적 공유 이미지
- footer / QR / safe area / template placement는 PDF 기준으로 먼저 확정
- PNG는 PDF 기준을 충실히 따르되, 공유성/가독성 중심으로 튜닝
- PDF와 PNG는 복불복처럼 달라지면 안 되며, **일관성과 안정성**을 유지해야 한다.

## 11. 개발 원칙

### 11.1 공통 원칙
- 콘텐츠 기준 데이터는 JSON이다.
- HTML, PDF, PNG는 모두 동일한 JSON 콘텐츠를 기준으로 생성한다.
- 산출물 간 내용 차이는 허용하지 않는다.
- 산출물 간 차이는 표현 방식의 차이만 허용한다.

### 11.2 HTML 개발 원칙
- HTML은 편집/검토/미리보기용 산출물이다.
- Step2 저장 시 HTML은 항상 최신 상태로 갱신되어야 한다.
- 미리보기를 하지 않더라도 저장만으로 temp.html은 생성되어야 한다.
- HTML은 사용자가 내용을 확인하기 위한 산출물이므로, 구조와 텍스트 반영이 우선이다.

### 11.3 PDF 개발 원칙
- PDF는 정식 문서 기준 산출물이다.
- 레이아웃 정책은 PDF를 기준으로 먼저 확정한다.
- footer / QR / safe area / template placement는 PDF에서 먼저 안정화한다.
- PDF 서비스는 PDF 전용 style / layout / runtime 규칙을 독립적으로 관리한다.
- PDF 관련 수정은 PNG에 직접 의존하지 않는다.

### 11.4 PNG 개발 원칙
- PNG는 공유용 이미지 산출물이다.
- PNG는 PDF를 대체하는 별도 문서가 아니라, PDF를 보기 어려운 사용자를 위한 공유용 결과물이다.
- PNG는 PDF와 동일한 콘텐츠와 동일한 레이아웃 원칙을 유지해야 한다.
- PNG 서비스는 PNG 전용 style / layout / runtime 규칙을 독립적으로 관리한다.
- PNG 튜닝은 가능하되, PDF 기준을 훼손하지 않는 범위에서만 허용한다.

### 11.5 footer / QR 개발 원칙
- footer는 본문과 분리된 하단 고정 영역이다.
- QR는 footer 자산이다.
- QR 크기 기준은 PDF에서 먼저 정하고, PNG는 그 기준을 따른다.
- footer text / brand / church / QR는 각각 독립적인 위치 조정값을 가질 수 있다.
- 본문이 footer를 침범하지 않도록 safe area를 유지한다.

### 11.6 템플릿 개발 원칙
- 템플릿은 배경 확장 요소이며, 콘텐츠보다 우선하지 않는다.
- 템플릿은 본문, footer, QR의 가독성과 안정성을 해치면 안 된다.
- 템플릿 적용 시에도 PDF와 PNG는 동일한 placement 원칙을 유지한다.
- PNG 템플릿은 공유용 이미지 기준으로 readability를 확보해야 한다.

### 11.7 리팩토링 원칙
- PDF와 PNG는 독립적으로 리팩토링한다.
- 공통 로직은 최소 범위만 공유한다.
- 공유 가능 대상:
  - JSON 기반 콘텐츠 구조
  - footer asset 준비
  - QR 생성 로직
  - footer HTML 조립
  - page fit script
- 분리 대상:
  - PDF 전용 style / layout / runtime
  - PNG 전용 style / layout / runtime
- 수정 시 "공통화"보다 "안정성"을 우선한다.

### 11.8 변경 원칙
- 레이아웃에 영향을 주는 변경은 작은 단위로 나누어 적용한다.
- 한 번에 여러 요소를 동시에 바꾸지 않는다.
- 변경 후에는 temp.html / temp.pdf / temp.png를 각각 확인한다.
- 디버그용 CSS, outline, background 강조색은 테스트 종료 후 반드시 제거한다.

- [ ] 템플릿 적용 전/후에 산출물 내용이 바뀌지 않는가

---

### 12.8 리팩토링 후 체크
- [ ] temp_png_source.html에 CSS 중복이 없는가
- [ ] PNG에 PDF 전용 runtime 변수가 섞여 있지 않은가
- [ ] PDF에 PNG 전용 runtime 변수가 섞여 있지 않은가
- [ ] 디버그용 배경색 / outline / 임시 로그가 제거되었는가
- [ ] 기존 기능(저장 / 미리보기 / 다음 단계 이동)이 깨지지 않았는가
- [ ] 설치본에서도 동일하게 동작하는가

---

### 12.9 최종 승인 기준
- [ ] HTML = 편집/미리보기용으로 충분히 안정적
- [ ] PDF = 정식 문서로 배포 가능
- [ ] PNG = 공유용 이미지로 충분히 안정적
- [ ] PDF와 PNG가 복불복처럼 다른 결과를 만들지 않음
- [ ] footer / QR / 템플릿이 문서 전체 흐름을 해치지 않음

## 13. 운영 체크 원칙

- 수정은 항상 작은 단위로 진행한다.
- PDF와 PNG는 각각 독립 테스트한다.
- 하나의 수정에서 확인할 목적을 명확히 한다.
- "스타일 수정", "하단 수정", "QR 수정", "템플릿 수정"을 섞지 않는다.
- 테스트 성공 후에만 다음 단계 수정으로 넘어간다.
- 설치본 테스트 전에는 개발 환경에서 먼저 temp.html / temp.pdf / temp.png를 확인한다.
- 설치본 테스트에서는 생성 성공 여부뿐 아니라 시각 결과까지 반드시 확인한다.

## 14. 파일별 책임 정리

### 14.1 전체 원칙
각 파일은 자신의 책임 범위를 명확히 가져야 하며,  
산출물 간 역할이 섞이지 않도록 한다.

- HTML = 검토/편집/미리보기
- PDF = 정식 문서
- PNG = 공유용 이미지
- Template = 배경/스킨 확장
- QR / Footer = 하단 자산

파일별 책임은 "무엇을 생성하는가"와 "무엇을 결정하는가"를 기준으로 구분한다.

---

### 14.2 Step2 / HTML 관련 파일

#### `qtstep2_service.go`
역할:
- Step2 저장 시 JSON 기반으로 HTML 생성
- temp.html 생성
- 사용자가 편집한 제목, 본문 성구, 찬송, 관련 성구, 성경본문, 각 섹션 내용을 HTML에 반영
- 성경본문 축약 규칙 반영

책임:
- HTML 콘텐츠 구조 생성
- HTML 본문 내용 최신화
- 편집 결과 반영

비책임:
- PDF 생성
- PNG 생성
- QR 생성
- footer 자산 생성

원칙:
- Step2 저장만으로 temp.html은 항상 최신 상태여야 한다.
- 미리보기는 선택 기능이며, 저장은 필수 기능이다.

---

### 14.3 PDF 관련 파일

#### `pdf_service.go`
역할:
- temp.html 또는 HTML fragment를 바탕으로 PDF 생성
- PDF 전용 style / layout / runtime 규칙 관리
- PDF source html 생성
- Edge headless 기반 PDF 출력
- PDF 기준 footer / QR / safe area 적용

책임:
- 정식 문서 레이아웃 확정
- PDF 전용 스타일 관리
- PDF 전용 runtime style 관리
- PDF 생성 결과 안정성 확보

비책임:
- PNG 전용 layout/runtime
- PNG 배경 투명 처리
- PNG 전용 하단 zone 조정

원칙:
- PDF는 기준 문서이므로 레이아웃 우선권을 가진다.
- footer / QR / safe area 기준은 PDF에서 먼저 안정화한다.
- PDF style과 PNG style은 분리 관리한다.

---

### 14.4 PNG 관련 파일

#### `png_service.go`
역할:
- temp.html 또는 HTML fragment를 바탕으로 PNG 생성
- PNG 전용 style / layout / runtime 규칙 관리
- source html 생성 후 브라우저 렌더링 기반 PNG 생성
- 공유용 이미지 산출물 안정화

책임:
- PNG 전용 하단 zone 구성
- PNG 전용 footer / QR 위치 조정
- PNG 생성 안정성 확보
- PNG source html 디버깅 가능 구조 유지

비책임:
- PDF 전용 safe area 규칙 결정
- PDF 전용 footer runtime 관리

원칙:
- PNG는 PDF를 대체하는 문서가 아니라 공유용 이미지다.
- PNG는 PDF 기준을 따르되, 공유성과 가독성을 위해 별도 튜닝할 수 있다.
- PNG는 복불복처럼 결과가 달라지면 안 된다.

---

### 14.5 Footer 관련 파일

#### `footer_service.go`
역할:
- footer 영역 정책 계산
- footer 표시 여부, divider, footer text, church/brand/logo, QR 표시 정책 계산
- safe area 기준값 제공

책임:
- footer 구성 정책
- 하단 자산의 기본 배치 기준
- QR 유무/브랜드 유무에 따른 safe area 정책

비책임:
- 실제 PDF/PNG 렌더링
- QR 이미지 생성

원칙:
- footer는 본문과 분리된 하단 고정 영역이다.
- footer 정책은 문서 전체 레이아웃과 충돌하지 않아야 한다.

---

### 14.6 QR 관련 파일

#### `qr_service.go`
역할:
- QR 이미지 생성
- QR 스타일(dot 등), margin, quiet zone, 배경색/전경색, size_px 관리
- footer에서 사용할 QR 자산 준비

책임:
- QR 이미지 자체 품질
- QR 투명 배경 여부
- QR 내부 여백(margin / quiet zone) 조정
- 스캔 가능성 확보

비책임:
- PDF/PNG 내 최종 위치 배치
- footer zone 계산

원칙:
- QR는 footer 자산이다.
- QR 크기 기준은 문서 정책을 따르되, 이미지 자체 품질은 qr_service.go가 책임진다.
- QR는 스캔 안정성을 해치지 않는 범위에서만 스타일 변경한다.

---

### 14.7 Template 관련 파일

#### `template_service.go`
역할:
- 템플릿 선택/적용
- PDF/PNG 산출물에 배경 이미지 합성
- 템플릿 manifest / preview / placement 관리

책임:
- 템플릿 배경 자산 적용
- 템플릿과 본문 산출물의 합성
- placement 규칙 유지

비책임:
- 본문 콘텐츠 생성
- QR 이미지 생성

원칙:

- 템플릿은 본문/QR/footer보다 우선하지 않는다.

## 17. 리팩토링 원칙

### 17.1 목적
리팩토링의 목적은 단순히 코드를 줄이는 것이 아니라,  
각 산출물의 역할과 책임을 명확히 하여 **안정성과 유지보수성**을 높이는 것이다.

S2QT에서 리팩토링은 다음 목표를 가진다.

- HTML / PDF / PNG의 역할 분리
- 서비스별 책임 명확화
- 산출물 안정성 확보
- 디버깅 포인트 단순화
- 기능 수정 시 영향 범위 축소

---

### 17.2 최우선 원칙
리팩토링의 최우선 기준은 **공통화가 아니라 안정성**이다.

즉:
- 코드가 조금 중복되더라도
- PDF와 PNG가 각각 안정적으로 동작한다면
- 무리한 공통화보다 독립 구조를 우선한다

---

### 17.3 역할 분리 원칙
- HTML은 편집/검토/미리보기
- PDF는 정식 문서
- PNG는 공유용 이미지
- Template는 배경 확장
- Footer/QR는 하단 자산

리팩토링 시, 한 파일이 여러 역할을 동시에 갖지 않도록 한다.

---

### 17.4 공통화 최소화 원칙
공통 코드는 최소 범위로만 유지한다.

공통화 가능한 대상:
- JSON 기반 콘텐츠 구조
- footer asset 준비
- QR 생성 로직
- footer HTML 조립
- 공통 page fit script
- 공통 유틸 함수

공통화 지양 대상:
- PDF 전용 style / runtime
- PNG 전용 style / runtime
- PDF 전용 하단 레이아웃
- PNG 전용 하단 레이아웃
- template 적용 후의 세부 배치 규칙

---

### 17.5 독립성 원칙
PDF와 PNG는 각각 독립적으로 튜닝 가능해야 한다.

- PDF 수정이 PNG를 깨뜨리면 안 된다.
- PNG 수정이 PDF를 깨뜨리면 안 된다.
- 한쪽 수정 후 다른 쪽은 검증만 하면 되는 구조가 바람직하다.

---

### 17.6 단계적 변경 원칙
리팩토링은 반드시 작은 단위로 나누어 진행한다.

금지:
- style + footer + QR + template를 한 번에 변경
- HTML/PDF/PNG를 동시에 구조 변경
- 공통 함수 제거와 산출물 변경을 한 번에 수행

권장:
- 한 번에 하나의 책임만 변경
- 한 번에 하나의 서비스만 변경
- 변경 후 즉시 temp.html / temp.pdf / temp.png 확인

---

### 17.7 결과 우선 원칙
리팩토링 후 가장 먼저 확인해야 할 것은 코드가 아니라 산출물이다.

우선 확인 대상:
1. temp.html
2. temp.pdf
3. temp.png

리팩토링이 성공했는지는
- 코드가 예뻐졌는가
보다
- 산출물이 안정적으로 동일한가
로 판단한다.

---

### 17.8 후퇴 가능성 원칙
리팩토링은 언제든지 이전 단계로 되돌릴 수 있어야 한다.

원칙:
- 변경 전 백업
- 함수 단위 주석 처리 후 테스트
- 테스트 통과 후 삭제
- 한 번에 너무 많은 함수 제거 금지

---

### 17.9 디버그 분리 원칙
디버그용 코드는 반드시 본 코드와 구분한다.

예:
- outline
- background 강조색
- 테스트용 temp source 유지
- debug log
- 중간 산출물 확인용 주석 처리

원칙:
- 테스트 중에는 허용
- 테스트 완료 후 제거
- 운영 코드에 남기지 않음

---

### 17.10 문서화 원칙
리팩토링한 내용은 반드시 문서 기준과 연결한다.

즉:
- 왜 수정했는지
- 어떤 파일을 건드렸는지
- 무엇이 기준인지
- 어떤 테스트를 통과했는지

를 남겨야 한다.

코드만 바꾸고 기준 문서를 갱신하지 않으면, 다음 수정 시 다시 혼란이 생긴다.

## 18. 단계별 리팩토링 절차

### 18.1 1단계 — 기준 확정
목적:
- 무엇을 기준 산출물로 삼을지 확정

현재 기준:
- HTML = 편집/미리보기
- PDF = 정식 문서
- PNG = 공유용 이미지

확인 사항:
- PDF를 기준 문서로 본다
- PNG는 PDF를 공유용으로 전달하는 보조 산출물이다
- HTML은 최종 배포 기준이 아니다

완료 기준:
- 역할 분담이 문서로 정리됨
- 팀/작업자 간 같은 기준을 공유함

---

### 18.2 2단계 — 현상 고정
목적:
- 현재 정상 동작 상태를 기준점으로 고정

수행:
- temp.html 저장
- temp.pdf 저장
- temp.png 저장
- 필요 시 temp source html 저장

확인 사항:
- 현재 산출물이 정상인가
- footer / QR / 본문 / template 상태가 기준으로 삼을 만한가

완료 기준:
- "정상 상태" 샘플 확보
- 이후 비교 기준 확보

---

### 18.3 3단계 — 문제 범위 분리
목적:
- 문제를 서비스 단위로 나눔

분리 기준:
- HTML 문제인가
- PDF 문제인가
- PNG 문제인가
- QR 문제인가
- footer 문제인가
- template 문제인가

원칙:
- 한 문제를 한 서비스 범위로 최대한 축소
- 동시에 여러 서비스 수정 금지

완료 기준:
- 수정 대상 파일이 명확해짐

---

### 18.4 4단계 — PNG 먼저 분리
목적:
- 가장 흔들리기 쉬운 PNG 구조를 먼저 안정화

권장 작업:
- PNG 전용 runtime style 분리
- PNG 전용 footer zone 정리
- PNG source html 중복 제거
- 디버그 코드 제거
- QR 관련 값이 실제로 반영되는지 확인

완료 기준:
- temp_png_source.html이 한 체계로 정리됨
- PNG 산출물이 안정화됨

---

### 18.5 5단계 — PDF 분리
목적:
- PDF를 정식 문서 기준으로 독립 정리

권장 작업:
- PDF 전용 runtime style 함수 분리
- PDF 전용 layout style 정리
- template의 PDF 경로도 PDF 전용 runtime 사용
- 공용 footer runtime 의존 제거

완료 기준:
- PDF가 PNG 구조에 의존하지 않음
- PDF 전용 스타일 구조가 명확함
- PDF 산출물이 동일하게 유지됨

---

### 18.6 6단계 — 공용 함수 정리
목적:
- 더 이상 쓰지 않는 공용 함수 제거

절차:
1. 새 함수 추가
2. 호출부 교체
3. 기존 공용 함수 주석 처리
4. 테스트
5. 동일 결과 확인
6. 최종 삭제

완료 기준:
- 죽은 코드 제거
- 더 이상 참조되지 않는 함수 없음

---

### 18.7 7단계 — Template 영향 검증
목적:
- template 적용 시 PDF/PNG가 유지되는지 확인

확인 사항:
- template 미적용 PDF 정상
- template 적용 PDF 정상
- template 미적용 PNG 정상
- template 적용 PNG 정상
- footer / QR / 본문이 template에 가려지지 않음

완료 기준:
- template 유무에 따라 구조가 깨지지 않음

---

### 18.8 8단계 — QR / Footer 세부 조정
목적:
- 하단 자산의 세부 위치와 안정성 조정

확인 항목:
- footer text 위치
- brand/church/logo 위치
- QR 크기
- QR 배경 / margin / quiet zone
- safe area
- 본문과 하단 충돌 여부

원칙:
- QR 이미지 자체 조정은 qr_service.go
- 하단 위치 조정은 pdf/png service
- 정책 결정은 footer_service.go

완료 기준:
- 하단이 본문을 침범하지 않음
- QR가 잘리지 않음
- PDF와 PNG가 일관됨

---

### 18.9 9단계 — 디버그 제거
목적:
- 테스트 흔적 제거

제거 대상:
- outline
- 임시 background color
- 임시 source 유지 코드
- 테스트용 log
- 주석 처리된 임시 코드

완료 기준:
- 운영 코드에 디버그 흔적 없음
- 산출물 결과는 동일함

---

### 18.10 10단계 — 설치본 검증
목적:
- 개발 환경과 설치 환경 결과 일치 확인

확인 사항:
- Step3 실행 정상
- PDF 생성 정상
- PNG 생성 정상
- 링크 열기 / 파일 저장 정상
- temp 파일 경로 정상
- 설치 환경에서도 PDF와 PNG 결과가 동일 기준 유지

완료 기준:
- 설치본에서도 복불복 없이 동일 결과
- 운영 가능한 상태 확인

---

## 19. 리팩토링 실행 순서 권장안

1. 기준 문서 정리
2. 정상 산출물 샘플 확보
3. PNG 서비스 정리
4. PNG 테스트
5. PDF 서비스 정리
6. PDF 테스트
7. Template 연계 테스트
8. QR / Footer 세부 조정
9. 디버그 제거
10. 설치본 테스트

원칙:
- 항상 PNG와 PDF를 각각 확인한다
- 산출물 확인 없이 다음 단계로 넘어가지 않는다

## 20. 리팩토링 완료 판정 기준

다음 조건을 모두 만족하면 리팩토링이 완료된 것으로 본다.

### 구조 기준
- PDF와 PNG가 독립된 runtime/style 구조를 가진다
- 공통 함수는 최소 범위만 남는다
- 각 파일의 책임이 명확하다
- 사용되지 않는 함수가 제거되었다

### 결과 기준
- temp.html 정상
- temp.pdf 정상
- temp.png 정상
- PDF와 PNG가 복불복처럼 다른 결과를 만들지 않는다
- footer / QR / template가 안정적으로 유지된다

### 운영 기준
- 개발 환경 테스트 통과
- 설치본 테스트 통과
- 디버그 코드 제거 완료
- 문서 갱신 완료

### 유지보수 기준
- 이후 수정 시 어느 파일을 먼저 봐야 할지 명확하다
- PDF와 PNG 중 한쪽만 수정해도 영향 범위를 예측할 수 있다
- 산출물 정책과 코드 구조가 일치한다
