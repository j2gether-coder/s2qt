## 환경설정 / 보안 / AI 방식 / History / 메뉴 / 구현 원칙

## 1. 문서 목적

본 문서는 현재까지 논의한 S2QT 로컬형 기준의 설계 방향을 하나의 기준 문서로 정리한 것이다.

이 문서의 목적은 다음과 같다.

- 환경설정 화면 구조 재정의
- PIN / 암호화 / 이메일 기준 보안 절차 정리
- AI 사용 방식(수동 / 원격 / 로컬) 정리
- History 저장 범위와 메뉴 구조 정리
- 이후 구현 시 흔들리지 않도록 기준점 확보

본 문서는 **로컬형 기준**이며, 서버형/구독형 구조는 현재 범위에서 제외한다.

## 2. 현재 전제

### 2.1 앱 성격
- S2QT는 로컬 중심 문서 생성 앱이다.
- 핵심은 QT 준비 → Step1 → Step2 → Step3 흐름이다.
- SMTP, 원격 LLM API, 향후 템플릿/서버형은 부가 기능 성격이다.

### 2.2 저장 원칙
- 로컬 DB(SQLite) 사용
- 보안 메타는 `security.json` 사용
- 전사문은 DB에 저장하지 않음
- Step2/Step3 결과는 DB에 저장하지 않음
- Step1 JSON과 기본정보만 History로 저장

## 4. 보안 구조

## 4.1 기본 원칙
- 앱 시작 시 로그인 없음
- 민감 기능 사용 시 PIN 요구
- PIN은 세션 중 한 번 성공하면 유지
- 앱 종료 시 세션 해제

## 4.2 security.json 역할
`var/conf/security.json`에는 아래 보안 메타를 저장한다.

- version
- pin_enabled
- pin_length
- pin_hash
- salt
- kdf
- iterations
- email

### 예시
```json
{
  "version": 1,
  "pin_enabled": false,
  "pin_length": 6,
  "pin_hash": "",
  "salt": "",
  "kdf": "pbkdf2-sha256",
  "iterations": 120000,
  "email": ""
}
```

## 4.3 암복호화 키 생성 기준
암복호화 키는 다음 조합으로 생성한다.

- 최초 이메일 설정값
- PIN
- salt

즉 개념상:

`PBKDF2(email + ":" + pin, salt, iterations)`

### 의미
- PIN 단독보다 강한 구조
- 별도 key_id 없이 사용자 절차에 자연스럽게 연결 가능
- 이메일 변경 시 저장된 민감정보 재입력이 필요할 수 있음

## 4.4 알고리즘
### 채택
- PBKDF2-SHA256
- AES-256-GCM

### 제외
- AES-CBC
- ECC
- DB 이미지 은닉
- 과도한 디바이스 바인딩

## 5. 보안 관련 사용자 절차

## 5.1 기본 절차
사용자 기준 권장 절차는 아래와 같다.

1. 기본 이메일 설정
2. PIN 등록
3. AI/API 또는 SMTP 같은 부가 기능 설정
4. 필요 시 교회/브랜드 설정

## 5.2 최초 사용 시
### 기본 이메일 먼저 입력
- 이메일은 보안 기준값으로 사용
- 안내 문구로 이메일 변경 시 재입력 가능성 안내

### 최초 민감 기능 사용 시
- PIN이 없으면 PIN 등록 유도
- PIN 등록 완료 후 현재 작업 계속 진행

## 5.3 이후 사용 시
- 민감 기능 사용 시 PIN 확인
- 같은 앱 실행 세션에서는 재확인 없음

## 5.4 이메일 변경 시
강제 차단보다는 안내 문구로 사용자에게 설명한다.

예시 문구:
- 최초 등록한 이메일은 보안 기준으로 사용됩니다.
- 이메일 변경 시 저장된 민감정보를 다시 입력해야 할 수 있습니다.

## 6. PIN UI 구조

## 6.1 PIN 검증 모달
역할:
- 기존 PIN 확인
- 세션 unlock

구성:
- 보안 확인 제목
- 안내 문구
- 숫자 입력 표시
- 랜덤 숫자 키패드
- 지우기 / 전체삭제 / 취소 / 확인

## 6.2 PIN 등록 모달
역할:
- 최초 PIN 등록
- PIN 확인 입력

구성:
- PIN 등록 제목
- 안내 문구
- 새 PIN 입력
- PIN 확인 입력
- 등록 / 취소

## 6.3 랜덤 숫자 키패드
- 숫자 0~9는 모달이 열릴 때마다 랜덤 배치
- 지우기 / 전체삭제 / 확인 / 취소는 고정 위치 권장
- 입력값은 마스킹 표시

## 6.4 세션 메모리
- 검증 성공 시 `window.__lastVerifiedPin` 유지
- 세션 잠금/종료 시 제거
- `securityState.js`에서 unlock 상태 관리

## 7. AI 사용 방식 재정의

AI 기능은 하나가 아니라 아래 3가지 방식으로 구분한다.

### 7.1 수동
- 프롬프트 생성
- 사용자가 외부 AI에 직접 붙여넣기
- 결과 JSON 붙여넣기
- API KEY 불필요

### 7.2 원격 AI
- 외부 API 호출
- API KEY 필요
- 공급자/모델은 내부 기본값 또는 고급 설정

### 7.3 로컬 AI
- 로컬 LLM 사용
- API KEY 불필요
- 모델/엔진 설정 필요
- 향후 확장 검토

## 7.4 의미
따라서 AI API KEY는 “AI 전체의 필수값”이 아니라,
**원격 AI 사용 시 필요한 선택 설정**으로 본다.

- 코드 수정 없이 문서 교체 가능
- 카드형보다 읽기 흐름이 좋음

## 9. SMTP의 위치

SMTP는 핵심 기능이 아니라 **Step3 부가 기능**으로 본다.

### 역할
- 생성된 산출물(PDF 등)을 이메일로 보내는 부가 기능

### 정책
- SMTP 설정이 없어도 앱 핵심 기능은 동작
- SMTP 설정이 있으면 Step3에서 이메일 전송 버튼 활성화

### 의미
- 기본 이메일 = 보안 기준값
- SMTP = 부가 전송 기능

둘은 역할이 다르다.

## 13. 테스트 초기화 기준

테스트 초기화의 핵심 기준은 아래 두 파일이다.

- `var/conf/security.json`
- `var/db/s2qt.db`

### 의미
- `security.json`: 보안 메타 초기화
- `s2qt.db`: 설정/내역 초기화

즉 암호화 관련 테스트에서는 이 둘을 함께 삭제하는 것이 기준점이다.

### temp 파일
`var/temp/*`는 덮어쓰기 구조이므로 필수 초기화 대상으로 보지 않는다.

## 14. 구현 원칙 요약

### 유지할 것
- SQLite 기반 로컬 저장
- AES-GCM
- PBKDF2
- security.json
- 랜덤 숫자 PIN 키패드
- 세션 unlock

### 배제할 것
- ECC
- AES-CBC
- 이미지 은닉
- 과도한 서버형 보안 구조
- 과도한 강제 절차

### 제품 원칙
- 강제보다 유도
- 기본값 제안 + 선택 허용
- 사용자 절차 중심
- 과한 보안보다 현실적인 안정성

## 16. 최종 정리

현재 S2QT 로컬형 설계의 핵심은 다음과 같다.

- 기본 이메일 + PIN을 보안 기준으로 사용
- 민감값은 AES-GCM으로 암호화하여 DB에 저장
- 보안 메타는 security.json으로 분리
- 환경설정은 기본 정보 / 부가 기능 / 안내로 재구성
- AI는 수동 / 원격 / 로컬 방식으로 재정의
- SMTP는 Step3 이메일 전송용 부가 기능
- History는 “작업 내역”으로 제공
- 사용자 절차를 기준으로 UI를 유도

이 문서를 기준으로 이후 세부 구현을 이어간다.
