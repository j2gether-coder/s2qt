# 구독형 서버 결제 연동 정리

## 1. 개요

구독형 서버의 결제 연동은 단순히 결제창을 붙이는 것이 아니라,  
**서버가 최종 결제 상태와 구독 상태를 판정하는 구조**로 설계해야 한다.

핵심 원칙은 다음과 같다.

- 클라이언트는 결제 요청과 결과 표시만 담당한다.
- 서버는 결제 승인 검증, 구독 상태 갱신, 결제 이력 저장을 담당한다.
- 정기 구독은 빌링키 기반 자동결제 구조를 고려한다.
- 웹훅을 통해 결제 결과를 보완적으로 수신한다.
- 구독 상태 변경은 항상 서버 기준으로 처리한다.

즉 결제 연동은  
**결제창 + 서버 승인 검증 + 웹훅 + 정기결제 관리**  
구조로 보는 것이 맞다.

---

## 2. 결제 연동의 기본 방향

### 2.1 권장 구조
초기에는 **국내 PG 직접 연동**이 가장 현실적이다.

권장 방향:
- 토스페이먼츠 직접 연동
- 또는 확장성을 고려하면 포트원 연동

### 2.2 초기 권장안
초기 구현은 다음이 가장 단순하다.

- 결제창 호출: 클라이언트
- 승인/검증: 서버
- 구독 상태 갱신: 서버
- 정기 갱신: 서버 배치 또는 워커

즉 **서버 중심 결제 상태 관리**가 핵심이다.

---

## 3. 결제 대상 상품

현재 구독형 기준 상품은 아래와 같다.

- 월간 구독: 9,900원
- 연간 구독: 99,000원

추가 상품은 별도 관리한다.

- 유료 템플릿 1장: 500원
- 소형 책자형: 별도 가격
- 연간 전자책형: 별도 가격

원칙:
- **구독 상품**과 **템플릿/전자책 상품**은 구분해서 관리한다.

---

## 4. 결제 연동 원칙

### 4.1 클라이언트 역할
클라이언트는 다음만 담당한다.

- 요금제 선택
- 결제창 호출
- 결제 완료/실패 화면 표시

클라이언트는 결제 성공 여부를 최종 판정하지 않는다.

### 4.2 서버 역할
서버는 다음을 담당한다.

- 주문 생성
- 결제 승인 검증
- 웹훅 수신 및 검증
- 구독 상태 갱신
- 결제 이력 저장
- 자동결제(정기결제) 실행
- 실패/만료 처리

즉 결제와 구독의 진실 소스는 서버이다.

---

## 5. 기본 결제 흐름

### 5.1 최초 결제
최초 결제는 다음 순서로 진행한다.

1. 사용자가 월간 또는 연간 구독 선택
2. 서버가 내부 주문(order) 생성
3. 클라이언트가 PG 결제창 호출
4. 사용자가 결제 진행
5. 결제 결과를 서버가 승인 API 또는 검증 API로 확인
6. 서버가 결제 성공 시 구독 상태를 active로 변경
7. 서버가 billing_transactions에 결제 이력 저장
8. 서버가 subscriptions에 구독 시작/종료일 반영

### 5.2 실패 시 처리
결제 실패 시:
- 구독 상태는 변경하지 않음
- 실패 이력을 기록
- 사용자에게 실패 안내

---

## 6. 정기결제(자동갱신) 흐름

구독형은 월간/연간 구독이므로 자동갱신 구조를 고려한다.

### 6.1 기본 구조
1. 최초 결제 시 결제수단 등록
2. PG에서 빌링키 또는 자동결제용 식별자 확보
3. 서버에 빌링 참조값 저장
4. 다음 결제일에 서버가 자동 청구 수행
5. 성공 시 구독 종료일 연장
6. 실패 시 재시도 또는 유예 상태 처리

### 6.2 자동갱신 처리 주체
- 자동갱신은 반드시 서버가 수행한다.
- 클라이언트는 자동갱신을 직접 처리하지 않는다.

### 6.3 자동갱신 실패
자동갱신 실패 시 상태 예시는 아래와 같다.

- `active` → `past_due`
- 재시도 실패 지속 시 → `expired`

즉 실패 시 즉시 삭제하지 않고,  
유예 상태를 둘 수 있다.

---

## 7. 웹훅 처리

결제창 결과만으로는 상태 판정이 불완전할 수 있으므로  
웹훅을 반드시 고려하는 것이 좋다.

### 7.1 웹훅 역할
- 결제 성공/실패 통지 보완
- 취소/환불 통지 수신
- 자동결제 결과 수신
- 리다이렉트 실패 보완

### 7.2 웹훅 처리 원칙
- 웹훅 수신 후 서명/secret 검증
- 내부 주문과 매핑
- 중복 처리 방지
- 최종 상태 갱신

### 7.3 주의점
- 웹훅만 믿지 않고 가능하면 승인 API 또는 조회 API로 재확인
- 동일 이벤트 중복 수신 가능성을 고려해 idempotency 보장 필요

## 8. 서버 DB 반영 항목

결제 연동을 위해 서버 DB에는 다음 정보가 필요하다.

---

## 9. subscriptions 테이블 관리 항목

구독 관리에 필요한 주요 항목:

- user_id
- subscription_type (monthly / yearly / trial)
- status (active / trial / past_due / expired / canceled)
- start_at
- end_at
- monthly_quota
- renewal_at
- next_billing_at
- last_paid_at

원칙:
- 구독 상태는 결제 성공 후에만 갱신한다.

---

## 10. billing_transactions 테이블 관리 항목

결제 이력에는 아래 정보가 필요하다.

- user_id
- transaction_type (subscription / template_purchase)
- provider
- provider_tx_id
- order_id
- item_code
- amount_krw
- payment_fee_krw
- status (paid / failed / canceled / refunded)
- paid_at
- created_at

원칙:
- 결제 이력은 감사 및 손익 계산의 근거 데이터가 된다.

---

## 11. billing profile 또는 결제수단 참조 정보

정기결제를 위해 별도 참조 정보 저장이 필요할 수 있다.

예:
- user_id
- provider
- billing_key_ref
- card_brand_masked
- is_active
- created_at
- revoked_at

원칙:
- 민감한 카드정보는 직접 저장하지 않는다.
- PG가 제공하는 참조값과 마스킹 정보만 저장한다.

---

## 12. 결제 상태와 구독 상태의 관계

결제 상태와 구독 상태는 같은 것이 아니다.

### 예시
- 결제는 `paid`
- 구독은 `active`

또는
- 결제는 `failed`
- 구독은 `past_due`

즉 두 상태는 분리해서 관리해야 한다.

---

## 13. 권장 상태값 예시

### 13.1 구독 상태
- `trial`
- `active`
- `past_due`
- `expired`
- `canceled`

### 13.2 결제 상태
- `paid`
- `failed`
- `canceled`
- `refunded`

## 14. 구독 시작 절차

사용자가 구독을 시작할 때의 처리 순서는 다음과 같다.

1. 요금제 선택
2. 서버가 주문 생성
3. 결제창 호출
4. 결제 완료
5. 서버가 결제 승인 검증
6. billing_transactions 저장
7. subscriptions 상태를 `active` 또는 `trial`에서 `active`로 변경
8. 생성 quota 설정
9. 감사 로그 기록

---

## 15. 구독 갱신 절차

자동갱신 시 처리 순서는 다음과 같다.

1. 스케줄러 또는 worker가 갱신 대상 조회
2. 저장된 빌링 참조값으로 청구 요청
3. 결제 결과 확인
4. 성공 시:
   - billing_transactions 저장
   - subscriptions 종료일 갱신
   - status 유지 또는 active 갱신
5. 실패 시:
   - billing_transactions에 실패 기록
   - subscriptions 상태를 `past_due` 또는 `expired` 로 변경
6. 감사 로그 기록

---

## 16. 구독 취소 절차

사용자가 구독 취소를 요청하면 다음을 수행한다.

1. 현재 구독 조회
2. 자동갱신 중단 설정
3. 상태를 `canceled` 또는 종료 예정 상태로 반영
4. 현재 결제기간 종료일까지는 사용 가능 여부 정책 적용
5. 감사 로그 기록

원칙:
- 즉시 종료인지
- 기간 종료 후 종료인지
정책을 명확히 정해야 한다.

권장:
- **기간 종료 후 종료**

---

## 17. 환불/취소 처리

환불 또는 취소 발생 시:

1. PG 상태 확인
2. billing_transactions 상태 변경
3. 구독 상태 재판정
4. 필요 시 end_at 조정
5. 감사 로그 기록

원칙:
- 환불 이력은 별도로 보존
- 사용자가 이미 사용한 기간에 대한 정책은 서비스 정책에 따름

---

## 18. 구독형 서버와 결제 보안

### 18.1 원칙
- 결제 성공 여부를 클라이언트에 맡기지 않는다.
- 서버만 최종 구독 상태를 확정한다.
- 웹훅 검증을 수행한다.
- 빌링 참조값은 민감 정보로 취급한다.
- 결제 로그와 구독 로그를 분리 보관한다.

### 18.2 추가 고려 사항
- idempotency key 사용
- 웹훅 재전송 대응
- 중복 승인 방지
- 결제 실패 재시도 정책
- 장애 시 수동 재처리 절차

---

## 19. 개발환경 측면의 결제 연동 구성

구독형 서버 개발환경 기준:

- Linux
- Go API 서버
- PostgreSQL
- Nginx
- Redis(선택)
- Worker(선택)

결제 연동에서 역할은 아래와 같다.

### API 서버
- 주문 생성
- 결제 승인 검증
- 구독 상태 갱신
- 결제 조회 API 제공

### PostgreSQL
- 구독 정보 저장
- 결제 이력 저장
- 빌링 참조값 저장
- 감사 로그 저장

### Worker (선택)
- 자동결제 배치
- 결제 재시도
- 만료 처리
- 상태 동기화

### Redis (선택)
- 중복 요청 방지
- 작업 큐 상태
- rate limiting
- 임시 상태 관리

## 20. 추천 초기 구현 방향

초기에는 아래 구조를 권장한다.

- 국내 PG 직접 연동
- 결제창 기반 최초 결제
- 서버 승인 검증
- 웹훅 수신
- 월간/연간 구독 반영
- 정기결제는 빌링키 기반으로 확장 가능하게 설계

초기 MVP에서는:
- 수동 갱신 확인
- 또는 단순 월간/연간 결제
로 시작할 수도 있으나,  
장기적으로는 자동결제 구조를 고려해야 한다.

---

## 21. 최종 원칙 요약

- 결제 상태와 구독 상태는 분리 관리한다.
- 서버만 최종 결제 성공 여부를 판정한다.
- 구독 시작/갱신/취소/만료는 모두 서버가 관리한다.
- 웹훅을 통해 결제 상태를 보완 수신한다.
- 정기결제는 빌링키 또는 자동결제 참조값 기반으로 처리한다.
- 결제 이력은 손익 계산과 감사의 기준 데이터가 된다.

---

## 22. 한 줄 정리

**구독형 서버의 결제 연동은 “클라이언트 결제창 + 서버 승인 검증 + 웹훅 + 정기결제 관리” 구조로 설계하며, 구독 상태는 항상 서버가 최종 판정한다.**
